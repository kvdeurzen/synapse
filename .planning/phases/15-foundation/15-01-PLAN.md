**Step 1: Add import for resolveConfig**

import { resolveConfig } from './lib/resolve-config.js';

**Step 2: Add project_id validation function** (before the stdin handler)
```javascript
const PROJECT_ID_REGEX = /^[a-z0-9][a-z0-9_-]*$/;
function validateProjectId(id) {
  if (!id || typeof id !== 'string') {
    throw new Error('project_id is required in [project] section of project.toml');
  }
  if (!PROJECT_ID_REGEX.test(id)) {
    throw new Error(
      `project_id must be a lowercase slug (letters, numbers, hyphens, underscores, must start with alphanumeric). Got: "${id}"`
    );
  }
}
```

**Step 3: Add project.toml reading BEFORE the existing tier context block** (inside the try block, after `baseInstructions` is built)

Use `resolveConfig('project.toml')` to find the file.

**If project.toml is NOT found (resolveConfig returns null):**
- Hard fail per CONTEXT.md decision. Write to stderr: `[synapse-startup] ERROR: .synapse/config/project.toml not found. Run /synapse:init to set up this project.`
- Output an additionalContext message telling the agent: `ERROR: Synapse project not initialized. Run /synapse:init to create project.toml before using Synapse MCP tools.`
- `process.exit(0)` — do NOT exit non-zero (that would break Claude Code); the error message in context is the user-facing signal.

**If project.toml IS found:**
- Parse with `parseToml(fs.readFileSync(projectTomlPath, 'utf8'))`
- Extract `project` section: `project_id`, `name`, `skills`, `created_at`
- Call `validateProjectId(project_id)` — this throws on malformed IDs, caught by the outer try/catch
- Validate skills against existing SKILL.md files: for each skill in the array, check `path.join(projectRoot, '.claude', 'skills', skill, 'SKILL.md')` with `fs.existsSync`. Log warning to stderr for missing SKILL.md files but do NOT fail.
  - Derive `projectRoot` from the resolved project.toml path: `path.dirname(path.dirname(path.dirname(projectTomlPath)))` (goes from `.synapse/config/project.toml` up to project root)
- Build the structured context block exactly matching the CONTEXT.md format:
  ```
  ─── SYNAPSE PROJECT CONTEXT ───
  project_id: {id}
  name: {name}
  skills: {skill1}, {skill2}
  ────────────────────────────────
  IMPORTANT: Always include project_id: "{id}" in every Synapse MCP tool call.
  ```

**Step 4: Prepend project context to the existing additionalContext string**
The project context block should come BEFORE `baseInstructions` and `tierContext` in the final `additionalContext` string. This ensures project_id is the first thing the agent sees.

**Step 5: Update the existing tier config resolution to use resolveConfig**
Replace the `possibleRoots` loop (lines 42-59 of current file) with:
```javascript
const trustPath = resolveConfig('trust.toml');

```
This replaces the 3-root fallback with the shared resolver — DRY and consistent.

**Important: Preserve the outer try/catch** that currently wraps everything. The outer catch should still `process.exit(0)` silently on unexpected errors — but project.toml-missing is NOT an unexpected error, it's handled explicitly inside the try block.
