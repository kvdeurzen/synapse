  <files>packages/framework/commands/synapse/refine.md</files>
  <action>
Create `packages/framework/commands/synapse/refine.md` — the primary RPEV user interaction command. This is the most complex command in Phase 16, combining brainstorming, decision tracking, and state persistence.

**Frontmatter:**
```yaml
name: synapse:refine
description: Start or resume a refinement session — brainstorm ideas, track decisions, and shape requirements at any hierarchy level (epic, feature, or work package).
allowed-tools:
  - Read
  - Write
  - Bash
  - mcp__synapse__store_document
  - mcp__synapse__store_decision
  - mcp__synapse__check_precedent
  - mcp__synapse__get_smart_context
  - mcp__synapse__semantic_search
  - mcp__synapse__get_task_tree
  - mcp__synapse__create_task
```

**Objective section:** One sentence: "Run a structured brainstorming session that tracks decisions, surfaces what's missing, and persists state for cross-session continuity."

**Process steps:**

1. **Determine scope:** If the user provides arguments (e.g., `/synapse:refine "JWT auth"`), use that as the refinement target. If no arguments, check for in-progress refinement sessions: call `mcp__synapse__semantic_search` with query `"refinement state"`, category `"plan"`, to find existing refinement documents. If found, offer to resume. If nothing found and no arguments, ask: "What would you like to refine? You can describe a new epic, drill into an existing feature, or explore a work package."

2. **Detect hierarchy level:** Based on the target:
   - If it matches an existing epic in the task tree (via `get_task_tree` search) -> Epic level
   - If it matches a feature under an epic -> Feature level
   - If it matches a work package -> Work Package level
   - If it's a new concept with no task tree match -> Project-level or new Epic

   State the detected level to the user: "Refining at [LEVEL] level: [TITLE]"

3. **Load existing refinement state:** Call `mcp__synapse__semantic_search` with the item title + "refinement" to find prior session documents. If found, load and display the current state:
   ```
   ## Resuming Refinement: [Item Title] ([Level])

   ### DECIDED (N decisions stored)
   - [Decision text] (Tier N)

   ### OPEN (N decisions pending)
   - [Topic] — [why it matters]

   ### EMERGING (N topics surfaced)
   - [Topic] — [brief note]
   ```

4. **Load context and precedents:**
   - Call `mcp__synapse__get_smart_context` with the item title to gather related decisions and documents
   - Call `mcp__synapse__check_precedent` with the item's key themes to surface relevant prior decisions
   - Present relevant context: "Here's what Synapse knows about this area: [summary]"

5. **Run brainstorming session:** Act as a collaborative thinking partner (extending the brainstorm skill pattern). Use Socratic questioning to help the user explore:
   - Clarifying questions: "What problem does this solve?"
   - Consequence exploration: "What happens if we don't do this?"
   - Alternative surfacing: "What other approaches could work?"
   - Connection finding: "How does this relate to [existing decision]?"

   Throughout the conversation, maintain a mental model of three categories:
   - **DECIDED**: Statements the user has committed to. When a decision is made, call `mcp__synapse__check_precedent` to verify consistency, then `mcp__synapse__store_decision` with appropriate tier and rationale.
   - **OPEN**: Questions that must be resolved before refinement is complete at this level. Track these explicitly.
   - **EMERGING**: Topics that surfaced during discussion but haven't been fully explored yet.

6. **Periodically surface decision state:** After every few exchanges (or when the user seems ready), present the current decision tracker:
   ```
   ### Decision Tracker

   DECIDED (stored in Synapse):
   - [x] "Target: solo developers" (Tier 0, decision_id: ULID)

   OPEN (must resolve at this level):
   - [ ] "Auth strategy — OAuth vs magic links?" (Tier 1)

   EMERGING (surfaced, not yet explored):
   - "Multi-tenant support?" — came up in context discussion
   ```

7. **Check readiness (level-aware):** When the user indicates they're wrapping up, or when OPEN decisions are resolved, assess readiness against level-appropriate criteria:
   - **Project/Epic level:** All required-tier decisions captured? Requirements measurable? User explicitly signals "this foundation is solid" (per locked decision: explicit gate at project and epic levels).
   - **Feature level:** Tier 2 decisions captured? Requirements testable?
   - **Work Package level:** Spec unambiguous enough to implement?

   If not ready: "Before we can move to planning, we still need to resolve: [OPEN items]. Want to explore those now, or park them for next session?"

   If ready: Present readiness summary.

8. **Persist refinement state:** Call `mcp__synapse__store_document` with:
   - `project_id`: from session context
   - `title`: "Refinement: [Item Title] ([Level])"
   - `category`: "plan"
   - `status`: "active"
   - `tags`: "|refinement|[level]|[item_slug]|"
   - `content`: Structured document with DECIDED/OPEN/EMERGING sections, session summary, key insights
   - `doc_id`: If resuming an existing session, pass the existing doc_id to version rather than duplicate
   - `actor`: "synapse-orchestrator"

9. **Close session:**
   - If readiness confirmed:
     ```
     ## Current Behavior (Phase 16)
     Refinement state saved. When the RPEV orchestrator is available (Phase 18),
     it will auto-trigger planning for this item. For now, the decisions are stored
     in Synapse and the item is ready for manual coordination.

     ## What's stored:
     - N decisions in Synapse DB (queryable by any future agent)
     - Refinement state document (retrievable next session via /synapse:refine)
     ```
   - If pausing (OPEN items remain):
     "Session saved. Run `/synapse:refine` again to resume — your DECIDED, OPEN, and EMERGING items will be loaded automatically."

**Anti-patterns to explicitly state in the command:**
- Do NOT make decisions for the user during Refine — surface options, let user decide
- Do NOT skip check_precedent before storing new decisions — consistency is non-negotiable
- Do NOT auto-transition to Plan at Project/Epic level — user must explicitly signal readiness
- Always persist state before session ends — context window may fill during rich brainstorming
- Use existing doc_id when resuming to version (not create duplicates)
  </action>

  <files>packages/framework/commands/synapse/status.md</files>
  <action>
Rewrite `packages/framework/commands/synapse/status.md` to serve as the RPEV dashboard. Preserve the existing frontmatter pattern but expand the process to show the full "system drives, user unblocks" view.

**Updated frontmatter:**
```yaml
name: synapse:status
description: Show the RPEV project dashboard — epics by priority, feature progress, blocked items needing user input, recent decisions, and suggested actions.
allowed-tools:
  - Read
  - mcp__synapse__get_task_tree
  - mcp__synapse__get_smart_context
  - mcp__synapse__project_overview
  - mcp__synapse__semantic_search
```

**Process steps:**

1. **Get project overview:** Call `mcp__synapse__project_overview` for high-level stats (document counts, code index counts).

2. **Get task tree:** Call `mcp__synapse__get_task_tree` with depth sufficient to show epics and their features. Extract:
   - Epics sorted by priority
   - Per-epic: completion percentage, done/total features, current RPEV stage
   - Blocked items: tasks with status "blocked" and their block reasons
   - Items needing user input: blocked items where the block reason involves a decision or user action

3. **Get recent context:** Call `mcp__synapse__get_smart_context` in overview mode for recent decisions and activity.

4. **Check for active refinement sessions:** Call `mcp__synapse__semantic_search` with query "refinement state", category "plan", status "active" to find in-progress brainstorming sessions.

5. **Present RPEV dashboard:**
```
## Synapse Dashboard

### Epics (by priority)

**Epic A** [EXECUTING] (65% complete)
  - Feature 1 [DONE]
  - Feature 2 [IN PROGRESS] — 2/4 work packages done
  - Feature 3 [PENDING]

**Epic B** [REFINING] (0% complete)
  - Refinement in progress — 3 decisions made, 2 open
  - Last refined: [date]

**Epic C** [BACKLOG]
  - Not yet refined

### Needs Your Input

[If blocked items exist:]
N items need your attention:
  1. Epic B / Feature 4 — decision needed: "Auth strategy"
  2. Epic A / Feature 2 / WP 3 — blocked: "API spec unclear"

Use `/synapse:focus "item name"` or `/synapse:focus 2.4` to navigate to a specific item.

[If no blocked items:]
All clear — no items currently need your input.

### Agent Pool
[Phase 21 stub] Agent pool not yet active. When available, this section will show active agents and their current tasks.

### Recent Decisions
- [Decision title] (Tier N, [date])
- [Decision title] (Tier N, [date])

### Suggested Actions
[Based on current state, suggest 1-3 next steps:]
- Resume refinement of Epic B (`/synapse:refine "Epic B title"`)
- Review blocked item in Epic A (`/synapse:focus "blocked item"`)
- Start a new epic (`/synapse:refine`)
```

6. **Handle empty state:** If no epics exist and no refinement sessions are in progress:
```
## Synapse Dashboard

No work streams yet. Get started:
- `/synapse:refine` — describe your first epic and start shaping it
- `/synapse:map` — index your codebase for semantic search (if not done yet)
```

**Key design notes:**
- Status should feel like checking a team dashboard, not driving each step
- Blocked items get prominent placement — this is the "user unblocks" pattern
- Agent pool section is explicitly stubbed with a note about Phase 21
- Proactive notifications section omitted (deferred to Phase 23, per locked decision)
