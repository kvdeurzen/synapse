Create the `/synapse:init` and `/synapse:map` slash commands — the first two steps of the Synapse user journey.

Output: Two markdown command files in `packages/framework/commands/synapse/`.

  <action>
Create `packages/framework/commands/synapse/init.md` following the established slash command pattern (frontmatter + Objective + Process + Attribution).

**Frontmatter:**
```yaml
name: synapse:init
description: Initialize a Synapse project — creates project.toml, registers with Synapse DB, and configures RPEV preferences.
allowed-tools:
  - Read
  - Write
  - Bash
  - mcp__synapse__init_project
```

**Process steps (numbered, with bold step names):**

1. **Check for existing project:** Read `.synapse/config/project.toml`. If it exists, display the existing project_id, name, and created_at. Warn: "This project is already initialized (created: DATE). Re-running init will not affect existing data." Offer to continue (update config) or abort.

2. **Detect project name:** Use Bash to try: (a) read `package.json` `.name` field, (b) fallback to `basename $PWD`. Slugify the result: lowercase, replace spaces/special chars with hyphens, strip leading non-alphanumeric, validate against `/^[a-z0-9][a-z0-9_-]*$/`. Show the user the detected project_id and allow them to correct it before proceeding.

3. **Create .synapse/config/ directory:** Use Bash `mkdir -p .synapse/config`.

4. **Write project.toml:** Use the Write tool to create `.synapse/config/project.toml` with the template:
```toml
[project]
project_id = "{{detected_or_chosen_id}}"
name = "{{project_name}}"
skills = []
created_at = "{{current ISO timestamp}}"
```

5. **Interactive RPEV configuration:** Walk the user through trust/involvement preferences. Present defaults and let user adjust:
   - Per-layer involvement gradient: project (user-driven), epic (co-pilot), feature (advisory), workpackage (autopilot)
   - Explicit gate levels: project and epic by default (user must signal "foundation is solid" before Plan auto-triggers)
   - Proactive notifications: false by default (Phase 23 feature, seed the option now)

6. **Write trust.toml RPEV section:** Read existing `.synapse/config/trust.toml` if present (preserve existing sections). If no trust.toml exists, copy the default from `packages/framework/config/trust.toml` as the base. Append the `[rpev]` section with user's choices:
```toml
[rpev]
project_refine = "user-driven"
epic_refine = "co-pilot"
feature_refine = "advisory"
workpackage_refine = "autopilot"
explicit_gate_levels = ["project", "epic"]
proactive_notifications = false
```

7. **Register with Synapse DB:** Call `mcp__synapse__init_project` with `project_id` and `actor: "synapse-orchestrator"`. Report the result (tables created/skipped, starters seeded). NOTE: This does NOT require Ollama — it creates LanceDB tables only.

8. **Offer CLAUDE.md amendment:** Check if `CLAUDE.md` exists. If it has a `## Synapse` section, skip. Otherwise, offer to append:
```markdown
## Synapse
This project uses Synapse for AI agent coordination.
Run `/synapse:status` to check project state, `/synapse:refine` to start work.
```
Only append if the user agrees. Never modify CLAUDE.md silently.

9. **Detect skills (auto-suggestion):** Check `.claude/skills/` for subdirectories with SKILL.md files. If found, suggest adding them to project.toml `skills` array. Update project.toml if user agrees.

10. **Summary:** Display what was created/configured: project_id, config files, DB tables, CLAUDE.md status.

**Anti-patterns to explicitly state in the command:**
- Do NOT check for Ollama during init — only `/synapse:map` needs it
- Do NOT modify CLAUDE.md without explicit user consent
- Do NOT hardcode project_id — always use the detected/confirmed value
- Always include `actor: "synapse-orchestrator"` in MCP calls

<task type="auto">
  <name>Task 2: Create /synapse:map command</name>
  <files>packages/framework/commands/synapse/map.md</files>
  <action>
Create `packages/framework/commands/synapse/map.md` following the established slash command pattern.

**Frontmatter:**
```yaml
name: synapse:map
description: Index the project codebase for semantic search — checks Ollama health, runs index_codebase, and reports results.
allowed-tools:
  - Bash
  - mcp__synapse__index_codebase
  - mcp__synapse__project_overview
```

**Process steps:**

1. **Verify project is initialized:** Check that the SYNAPSE PROJECT CONTEXT block is present in session context (project_id is available). If not, tell the user to run `/synapse:init` first.

2. **Check Ollama health:** Use Bash to run `curl -sf http://localhost:11434/api/tags` (or equivalent). If Ollama is not reachable:
   - Display a clear error: "Ollama is not running. Synapse needs Ollama for code embeddings."
   - Provide actionable instructions: "Start Ollama with: `ollama serve` (in a separate terminal)"
   - Also check for the `nomic-embed-text` model: `curl -sf http://localhost:11434/api/tags | grep nomic-embed-text`. If missing: "Pull the embedding model with: `ollama pull nomic-embed-text`"
   - Do NOT proceed until Ollama is confirmed running with the model available

3. **Report indexing start:** Tell the user: "Indexing codebase at [project_root]... This may take a few minutes depending on project size. Synapse will parse source files, extract AST structures, and generate embeddings."

4. **Run index_codebase:** Call `mcp__synapse__index_codebase` with:
   - `project_id`: from session context
   - `project_root`: absolute path to the project root (use `$CLAUDE_PROJECT_DIR` or derive from `.synapse/config/` location)
   - `actor`: "synapse-orchestrator"

   If the tool returns an error (especially OllamaUnreachableError), display a user-friendly message and suggest remediation.

5. **Report results:** Call `mcp__synapse__project_overview` to show the updated project state including document and code index counts. Present a summary:
   - Files indexed
   - Code symbols extracted
   - "Your codebase is now searchable. Agents can find relevant code and documentation using semantic search."

6. **Suggest next step:** "Run `/synapse:refine` to start defining your first epic, or `/synapse:status` to see the project overview."

**Key design notes to include in the command:**
- This is the ONLY command that requires Ollama — clearly state this
- Progress feedback is important — the user should not see a silent wait
- The Ollama check is defensive, not blocking session start (unlike init which creates files)
