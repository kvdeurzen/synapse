# Phase 26.7: Pause, Continue, Upgrade, and Remove Synapse - Research

**Researched:** 2026-03-13
**Domain:** Slash command implementation, lifecycle management, state persistence, installation mechanics
**Confidence:** HIGH

## Summary

Phase 26.7 adds four lifecycle management slash commands to Synapse. All four are primarily `.md` slash command files (like the existing `init.md`, `map.md`, `refine.md`, `status.md`, `focus.md`) plus JS modifications to `synapse-startup.js` for pause-detection, and additions to `install.sh` for upgrade and remove shell logic. The commands themselves are agentically interpreted — they are markdown instruction files, not scripts.

The core technical challenge is that all four commands interact with existing infrastructure in non-trivial ways: pause reads and serializes live RPEV pipeline state; continue reconstructs that state and re-enters the pipeline; upgrade re-runs `install.sh` logic with version diffing; remove must surgically excise Synapse from multiple config files while handling self-deletion. None of these require new libraries — all build on established patterns already in the codebase.

The most complex sub-problem is the pause/continue snapshot format: it must be rich enough to fully reconstruct mid-pipeline context across session boundaries, stored where `synapse-startup.js` can detect it at session start, and resilient to partial agent failures. The upgrade command is largely mechanical — `install.sh` already handles most of it. Remove has the self-deletion paradox (the command file deletes itself as its final step) which requires careful sequencing.

**Primary recommendation:** Implement all four as `.md` slash command files following the exact frontmatter/process structure of existing commands. Use `.synapse/state/pause-snapshot.json` for pause state (matches the established `statusline.json` pattern). Wrap `install.sh --version` for upgrade. Build remove as a confirmation-gated file deletion sequence.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Pause behavior**
- Support BOTH mid-session interrupt and session-boundary pausing
- Mid-session interrupt: present user with choice — graceful drain (let in-flight agents finish) or immediate stop with revert of partial work
- Save a full state snapshot: pipeline position, active decisions, in-progress task IDs, pool state
- `/synapse:pause` is invokable any time, not just during active RPEV pipeline — even outside RPEV, saves current session state (open refinements, what user was working on)

**Continue behavior**
- `/synapse:continue` auto-detects what was paused from the snapshot and picks up automatically — no user menu
- Startup hook auto-detects pause snapshot on session start and prompts user: "You paused [phase/stage]. /synapse:continue to resume."

**Upgrade mechanism**
- `/synapse:upgrade` as convenience slash command wrapping `install.sh` under the hood — two entry points (command + script), one mechanism
- Overwrite all framework files (agents, hooks, commands, skills) AND server source — no schema migration, framework files are Synapse-owned
- Show changelog diff between current and target version before applying — user confirms
- Auto-detect both current installed version (from `project.toml` field) and latest available version from GitHub — user just runs `/synapse:upgrade`
- Version tracked in `project.toml` as a `synapse_version` field (set by install.sh and upgrade)
- No backup of previous installation — framework files are reproducible from any version tag, git history is the backup
- Auto-commit all changed framework files after upgrade with conventional commit: `chore(synapse): upgrade to {version}`
- Hot-reload updated framework files in-session if possible; fall back to restart recommendation if critical files changed (hooks, settings.json)

**Remove behavior**
- `/synapse:remove` deletes everything Synapse-owned: `.claude/agents/`, `.claude/hooks/synapse-*`, `.claude/commands/synapse/`, `.claude/server/`, Synapse entries from `.mcp.json` and `settings.json`, and `.synapse/` config dir

- Require confirmation before executing — show file list + data summary, explicit 'yes' required
- The remove command deletes itself as its last step

- Four new slash commands: `/synapse:pause`, `/synapse:continue`, `/synapse:upgrade`, `/synapse:remove`

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|

**When to use:** Pause snapshot storage.

### Pattern 3: Startup Hook Detection

**What:** `synapse-startup.js` reads additional state files and injects context.
**When to use:** Pause detection at session start.

The hook already reads `project.toml`, `trust.toml`, `agents.toml`, `statusline.json`, and `gateway-protocol.md`. The pause detection adds one more `fs.existsSync` + `JSON.parse` check. It must be **non-blocking** — if the snapshot file is corrupt or missing, the hook continues silently.

```javascript
// Pattern to add after existing contextParts assembly, before final output
let pauseContext = "";
try {
  const snapshotPath = path.join(projectRoot, ".synapse", "state", "pause-snapshot.json");
  if (fs.existsSync(snapshotPath)) {
    const snapshot = JSON.parse(fs.readFileSync(snapshotPath, "utf8"));
    pauseContext = [
      "## Paused Work Detected",
      "",
      `You paused ${snapshot.pipeline_active ? `RPEV pipeline at stage: ${snapshot.pipeline_stage}` : "a session"} on ${snapshot.paused_at}.`,
      "Run /synapse:continue to resume where you left off.",
    ].join("\n");
  }
} catch {
  // Silent fail — never block session start
}
if (pauseContext) contextParts.push(pauseContext);
```

### Pattern 4: JSON Merge/Strip via `bun -e`

**What:** Inline Bun script to surgically modify `.mcp.json` and `settings.json`.
**When to use:** Remove command needs the inverse of what install does.

Install merges (adds Synapse keys). Remove strips them. The same `synapseSignatures` array used in install can drive removal:

```javascript
// Source: install.sh lines 551-565 (synapseSignatures pattern)
const synapseSignatures = [
  'synapse-startup.js', 'tier-gate.js', 'precedent-gate.js',
  'tool-allowlist.js', 'audit-log.js', 'synapse-statusline.js',
  'conventional-commit.js', 'output-contract-gate.js',
];

// For remove: filter OUT Synapse hook groups (inverse of install logic)
for (const event of ['SessionStart', 'PreToolUse', 'PostToolUse']) {
  merged.hooks[event] = (existing.hooks[event] || [])
    .filter(group => !isSynapseHookGroup(group));
}
// Remove statusLine if it's a Synapse one
if (existing.statusLine?.command && synapseSignatures.some(s => existing.statusLine.command.includes(s))) {
  delete merged.statusLine;
}
// Remove synapse from mcpServers
delete merged.mcpServers.synapse;
```

**What:** `synapse_version` field added to `[project]` section of `project.toml`.
**When to use:** Upgrade reads current version, writes new version on completion.

Current `project.toml` template (from `packages/framework/config/project.toml`):
```toml
[project]

```

After install.sh modification:
```toml
[project]
project_id = "synapse-dev"
name = "Synapse Development"
skills = []
synapse_version = "v3.0.0-alpha.12"
```

`smol-toml` is used for reading. Writing can be done as string replacement in Bash:
```bash
# Add/update synapse_version field using sed or bun -e inline script
bun -e "
  const content = require('fs').readFileSync('.synapse/config/project.toml', 'utf8');
  const {parse, stringify} = require('smol-toml');
  // Note: smol-toml stringify may reorder; sed approach safer for single field
"
```

**Simpler approach:** Use `sed` to replace or append the `synapse_version` line, since `project.toml` has a stable format. This avoids toml-serialize round-trip issues.

### Anti-Patterns to Avoid

- **Hook-blocking pause detection:** Never `process.exit(1)` or write output that blocks session start based on pause detection. It must be advisory only.
- **Rewriting project.toml via smol-toml round-trip:** TOML serializers can reorder fields and strip comments. Use targeted `sed` or string append for single-field updates like `synapse_version`.
- **Remove leaving orphaned entries:** `.mcp.json` and `settings.json` must be surgically cleaned, not deleted wholesale. The user may have non-Synapse entries in both files.
- **Continue trying to spawn agents directly:** Continue's job is to reconstruct context and hand off to the RPEV pipeline — it does not spawn subagents itself, it re-invokes the pipeline via the orchestrator.
- **Snapshot in Synapse MCP document store:** CRITICAL — the startup hook runs before MCP servers are available (stated explicitly in `synapse-startup.js` lines 4-6). Pause snapshot MUST be a file on disk, not a Synapse document.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Version download/install | Custom tarball downloader | `install.sh` with `--version` flag | Script already handles GitHub API, tarball download, dedup, node_modules |
| JSON config merging | Custom JSON parser | `bun -e` inline script pattern | Established pattern in install.sh, handles edge cases like missing keys |
| TOML parsing in hooks | Custom parser | `smol-toml` (already a dep) | Already imported in `synapse-startup.js` and `synapse-statusline.js` |
| Changelog formatting | Custom git diff | `git log --oneline ${FROM}..${TO}` or GitHub compare API | Git already has the data; GitHub API returns structured diff |
| Hook deduplication on upgrade | Custom comparator | `synapseSignatures` pattern from install.sh | Already handles whitespace-robust matching |

**Key insight:** All heavy lifting (download, JSON merge, hook wiring) is already in `install.sh`. The upgrade command is a thin wrapper that adds version detection, changelog display, user confirmation, and post-upgrade git commit.

## Common Pitfalls

### Pitfall 1: Startup Hook Reads Pause Snapshot Before MCP Is Available
**What goes wrong:** Attempting to store pause state as a Synapse MCP document (`store_document`) and then querying it in the startup hook.
**Why it happens:** Synapse startup hook runs as a Claude Code `SessionStart` hook — MCP servers are not yet initialized at this point. Line 4-6 of `synapse-startup.js` explicitly documents this: "CRITICAL: This hook runs BEFORE MCP servers are available to the agent."
**How to avoid:** Store pause snapshot as a plain JSON file at `.synapse/state/pause-snapshot.json`. The startup hook detects it with `fs.existsSync()` — same pattern as `statusline.json`.
**Warning signs:** Any plan that involves `query_documents` in the startup hook for pause detection.

### Pitfall 2: TOML Round-Trip Corruption
**What goes wrong:** Using `smol-toml`'s `stringify()` to write back a modified `project.toml` reorders fields and strips inline comments.
**Why it happens:** TOML serializers normalize to their own format; `smol-toml` does not guarantee field order preservation.
**How to avoid:** For adding `synapse_version`, use targeted string append: check if field exists with `grep`, replace with `sed`, or append to the `[project]` section. Do not round-trip the entire file through parse/stringify.
**Warning signs:** `project.toml` losing its comment lines or field ordering after upgrade.

### Pitfall 3: Remove Clobbers Non-Synapse Config
**What goes wrong:** Delete `.mcp.json` or `.claude/settings.json` entirely, removing user's non-Synapse entries.
**Why it happens:** Simplest implementation of "remove Synapse" is to delete the files.
**How to avoid:** Use the `bun -e` inline script pattern to surgically remove only Synapse-owned entries, identified by `synapseSignatures`. The existing install.sh merge logic is the inverse of what remove needs.
**Warning signs:** Remove command using `rm .mcp.json` or `rm .claude/settings.json`.

### Pitfall 4: Self-Deletion Race in Remove Command
**What goes wrong:** The remove command deletes itself before completing all steps, causing an error if any step references the command file.
**Why it happens:** `/synapse:remove` is at `.claude/commands/synapse/remove.md`. The command deletes `.claude/commands/synapse/` as part of removing Synapse.
**How to avoid:** Self-deletion must be the absolute last step — after all other cleanup is verified complete. The command should be written so all deletion steps are completed first, then the final step removes the `.claude/commands/synapse/` directory. Since the agent has already loaded the command into context, deleting the file mid-execution does not affect the running session.
**Warning signs:** `rm -rf .claude/commands/synapse/` appearing before other cleanup steps.

### Pitfall 5: Upgrade Hot-Reload for Hooks Requires Session Restart
**What goes wrong:** Assuming that overwriting hook `.js` files on disk causes the currently-running Claude session to use the new hook code immediately.
**Why it happens:** Claude Code hooks are loaded at session start (or tool invocation). The running session's hooks are already in memory/process.
**How to avoid:** After upgrade, detect if any hook files changed (compare file hashes or timestamps before/after). If hooks changed, recommend a session restart: "Critical files updated — please restart Claude Code to apply the new hooks." For agent `.md` files and command `.md` files, changes ARE hot-reloadable since they are read from disk on each invocation.
**Warning signs:** Claiming "hot-reload complete" for hook JS files without a session restart recommendation.

### Pitfall 6: Continue Attempting to Re-Spawn Orchestrator Without Snapshot Validation
**What goes wrong:** `/synapse:continue` blindly dispatches the orchestrator with stale snapshot data — e.g., task IDs that have since been completed or deleted.
**Why it happens:** The snapshot was taken mid-pipeline; the world may have changed between pause and resume.
**How to avoid:** Continue should validate snapshot data against live state: verify task IDs still exist via `get_task_tree`, check RPEV stage documents still exist, confirm pool state doc is current. Only proceed after validation. If snapshot is stale, report what changed and ask user how to proceed.
**Warning signs:** Continue command that calls orchestrator directly with snapshot data without querying current state first.

### Pitfall 7: Graceful Drain Has No Mechanism to Actually Signal In-Flight Agents
**What goes wrong:** "Graceful drain" is announced but has no actual implementation — in-flight agents continue running and may commit work after the pause snapshot is taken.
**Why it happens:** Claude Code subagents run in Task tool calls; there is no inter-process signal mechanism.
**How to avoid:** Graceful drain is a best-effort advisory behavior. The pause command sets a `draining: true` flag in the pool-state document. The orchestrator, if it checks pool state before dispatching new agents, can respect this. Existing in-flight agents cannot be stopped mid-execution. The honest message to the user is: "Waiting for current in-flight agents to complete before saving state."
**Warning signs:** Documentation claiming agents can be "stopped" or "interrupted" mid-execution.

### Remove: Settings.json Surgical Strip
```bash
SETTINGS_FILE=".claude/settings.json"
EXISTING_CONTENT=$(cat "$SETTINGS_FILE")
bun -e "
  const existing = $EXISTING_CONTENT;
  const synapseSignatures = [
    'synapse-startup.js', 'tier-gate.js', 'precedent-gate.js',
    'tool-allowlist.js', 'audit-log.js', 'synapse-statusline.js',
    'conventional-commit.js', 'output-contract-gate.js',
  ];
  function isSynapseHook(hook) {
    return typeof hook.command === 'string' &&
      synapseSignatures.some(sig => hook.command.includes(sig));
  }
  function isSynapseHookGroup(group) {
    return group?.hooks?.some(isSynapseHook) ?? false;
  }
  const result = { ...existing };
  result.hooks = { ...(existing.hooks || {}) };
  for (const event of ['SessionStart', 'PreToolUse', 'PostToolUse']) {
    result.hooks[event] = (existing.hooks?.[event] || [])
      .filter(g => !isSynapseHookGroup(g));
    if (result.hooks[event].length === 0) delete result.hooks[event];
  }
  if (existing.statusLine?.command &&
      synapseSignatures.some(s => existing.statusLine.command.includes(s))) {
    delete result.statusLine;
  }
  if (Object.keys(result.hooks).length === 0) delete result.hooks;
  process.stdout.write(JSON.stringify(result, null, 2) + '\n');
" > "$SETTINGS_FILE.tmp" && mv "$SETTINGS_FILE.tmp" "$SETTINGS_FILE"
```

## Open Questions

1. **What is the exact format for hot-reload detection?**
   - What we know: Agent `.md` files are read from disk on each invocation (hot-reloadable). Hook `.js` files need a session restart to take effect.
   - What's unclear: Does `settings.json` hot-reload, or does Claude Code need restart? Does `synapse-startup.js` modification take effect in the same session?
   - Recommendation: Conservatively treat ALL hook `.js` files and `settings.json` as requiring session restart. Only agent `.md` and command `.md` files as hot-reloadable. When upgrade touches any hook, display restart recommendation.

2. **Graceful drain: how long to wait?**
   - What we know: There is no IPC mechanism to signal in-flight Task agents. They run to completion.
   - What's unclear: Should pause block until drain completes (potentially minutes), or just save a snapshot and recommend the user wait?
