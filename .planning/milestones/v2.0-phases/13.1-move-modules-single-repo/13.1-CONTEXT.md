<decisions>
## Implementation Decisions

### Repo structure
- Bun workspaces layout: `packages/server/` and `packages/framework/`
- Package names: `@synapse/server` and `@synapse/framework`
- Root package.json named `synapse`, private:true, with workspaces config
- `.planning/` and `.claude/` stay at repo root level (project-wide, not package-specific)

### Package management
- Shared dependencies (zod, smol-toml) hoisted to root package.json; each workspace lists them as peerDependencies
- Root aggregate scripts: `test` runs all workspace tests, `test:server`, `test:framework` for targeted runs, `lint` runs all linters (using `bun run --filter`)
- Shared `tsconfig.base.json` at root with common compiler options (target, module, strict, bun-types); each package extends it
- Single shared `biome.json` at root for consistent lint/format rules across all packages

### Git history
- Preserve synapse-framework history via `git subtree add` from local path (`/home/kanter/code/synapse-framework`)
- One atomic commit: create `packages/` structure, move server into `packages/server/`, subtree add framework into `packages/framework/`
- Delete standalone synapse-framework repo after confirming merge is clean

### Claude Code integration
- Convention files (agents/, skills/, hooks/, workflows/, commands/, config/) stay under `packages/framework/`
- Update `settings.template.json` to point to `packages/framework/` paths; user copies to `.claude/settings.json` at root
- Single root-level `CLAUDE.md` with monorepo-relative paths (e.g., `packages/framework/agents/`)

### Claude's Discretion
- Whether to enable cross-package workspace imports (`@synapse/server` from framework code) — decide based on current coupling (framework currently references server only via MCP protocol, not direct imports)
- Exact git subtree command flags and merge strategy
- How to handle the server's existing `postinstall` script (tree-sitter setup) in workspace context

</decisions>
