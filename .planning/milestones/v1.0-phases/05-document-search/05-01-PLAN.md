<task type="auto">
  <name>Task 2: Create shared search utility module via TDD</name>
  <files>src/tools/search-utils.ts, test/tools/search-utils.test.ts</files>
  <action>
  Create `src/tools/search-utils.ts` with shared utilities used by all search tools. Build via TDD: write tests first, then implement.

  **Exports to create:**

  1. **`SearchResultItem` interface** — the common result shape per CONTEXT.md locked decisions:
     ```typescript
     export interface SearchResultItem {
       doc_id: string;
       chunk_id: string;
       title: string;
       category: string;
       status: string;
       relevance_score: number; // 0.0-1.0 normalized
       snippet: string;         // ~100-200 token content snippet
       source: "document" | "code"; // table attribution
     }
     ```

  2. **`buildSearchPredicate(projectId, filters, opts?)`** — builds WHERE clause for doc_chunks queries:
     - Always includes `project_id = '{projectId}' AND status = 'active'`
     - If metadata filters (category, phase, tags, status, priority) are provided, pre-fetches matching doc_ids from documents table and adds `doc_id IN (...)` clause
     - Returns `{ predicate: string; docMap: Map<string, DocMeta> }` where DocMeta has title, category, status, phase, tags, priority
     - Tag validation: reject tags not matching `/^[a-zA-Z0-9_-]+$/` (same as query-documents.ts)
     - Cap pre-filter doc_ids at 200 (per RESEARCH.md pitfall 4); if more, skip pre-filter and return flag `postFilterRequired: true`
     - If `include_superseded` option is true, omit the `status = 'active'` chunk filter
     - Parameters: `projectId: string`, `filters: { category?, phase?, tags?, status?, priority? }`, `opts?: { includeSuperseded?: boolean; dbPath: string }`

  3. **`extractSnippet(content, query, maxTokens = 150)`** — extract best snippet from chunk content:
     - If content is already <= maxTokens per `countTokens()`, return content as-is
     - Otherwise, find first occurrence of any query word (case-insensitive) in content
     - Extract a character window centered on that position (~4 chars per token heuristic)
     - Trim to exact token budget using `countTokens` loop (trim 10 chars at a time from end)
     - Add "..." prefix if start > 0, "..." suffix if window doesn't reach end
     - Import `countTokens` from `gpt-tokenizer`

  4. **`normalizeVectorScore(distance: number)`** — cosine distance [0,2] to relevance [0,1]:
     - Formula: `Math.max(0, Math.min(1, 1.0 - (distance / 2)))`
     - Clamp to [0,1] for safety

  5. **`normalizeFtsScore(score: number)`** — BM25 score to [0,1] via sigmoid:
     - Formula: `score / (score + 1)`
     - Returns 0 for score <= 0

  6. **`fetchDocMetadata(dbPath, projectId, docIds)`** — batch fetch doc metadata:
     - Opens documents table, queries `doc_id IN (...)` for given IDs
     - Returns `Map<string, DocMeta>` where DocMeta = `{ title, category, status, phase, tags, priority }`
     - Used by all search tools to enrich chunk results with document-level metadata

  **Test cases (RED first, then GREEN):**
  - `normalizeVectorScore(0)` = 1.0 (identical)
  - `normalizeVectorScore(2)` = 0.0 (opposite)
  - `normalizeVectorScore(0.5)` = 0.75
  - `normalizeFtsScore(0)` = 0
  - `normalizeFtsScore(1)` = 0.5
  - `normalizeFtsScore(9)` = 0.9
  - `extractSnippet("short text", "short")` returns "short text" (under budget)
  - `extractSnippet(longText, "keyword", 50)` returns ~50 token window with "..." markers
  - `buildSearchPredicate("proj", {})` includes project_id and status=active
  - `buildSearchPredicate("proj", { category: "research" })` includes doc_id IN clause
  - Tag validation rejects special characters

  **What to AVOID:**
  - Do NOT put LanceDB search logic in this module — only utilities
  - Do NOT import from search tool files (this is a leaf dependency)
