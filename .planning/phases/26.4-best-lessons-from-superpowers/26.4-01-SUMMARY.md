# Phase 26.4 Plan 01: Output Contract Gate Summary

**One-liner:** PostToolUse hook enforcing agent output contracts via output-contracts.toml, using fail-closed pattern with {task_id} pattern matching.

## What Was Built

### output-contracts.toml

Single source of truth mapping all 13 pipeline agents to their required output documents. Uses a `required_docs` array with `doc_id_pattern` and `provides` fields per entry. The `{task_id}` placeholder is substituted at runtime.

Key design decisions:
- `task-designer` omitted — writes spec via `update_task(spec: ...)` field, no document required
- `validator` uses `required_on_fail_only = true` — only stores findings docs on FAIL; PASS goes straight to `update_task(status: done)` without a doc
- All 12 producer agents from `_synapse-protocol.md` provides vocabulary are covered (plus `product-researcher` as a separate entry from `researcher`)

### output-contract-gate.js

PostToolUse hook (157 lines) following the exact pattern from `tier-gate.js`:
- Gates only on `mcp__synapse__update_task` calls with `status === "done"`
- Passes all other tool calls silently
- Passes `update_task` calls with non-done status silently (spec updates, status=in_progress, etc.)
- Passes agents not listed in `output-contracts.toml` silently (orchestrator pattern)
- Loads config via `resolveConfig("output-contracts.toml")` — fail-closed if missing
- Checks each required doc pattern against `tool_input.output_doc_ids`
- Outputs deny with specific missing doc list when requirements unmet

### Unit Tests

13 tests covering all 6 specified behaviors:
1. Deny on null/empty `output_doc_ids` for executor
2. Allow when matching doc present
3. Fail-closed when config missing (isolated via `CLAUDE_PROJECT_DIR` env var)
4. Passthrough for non-`update_task` tools
5. Passthrough for non-done status values
6. Passthrough for agents not in config (synapse-orchestrator)
7. Custom config loading from `CLAUDE_PROJECT_DIR`
8. Malformed JSON → deny (fail-closed)
