Create `packages/framework/agents/product-researcher.md` — the refinement-stage research agent. This replaces `product-strategist.md` (which will be deleted in Plan 05). Read product-strategist.md first as the source material.

**Key difference from Product Strategist:** The Product Researcher does NOT make decisions. It returns structured findings to the gateway, which presents them to the user. The user decides.

**Frontmatter:**
```yaml
---
name: product-researcher
description: Researches user scope during refinement — identifies gaps, pokes holes, surfaces improvements. Returns findings to gateway. Spawned by gateway during /synapse:refine sessions.
tools: Read, Bash, Glob, Grep, mcp__synapse__get_smart_context, mcp__synapse__query_decisions, mcp__synapse__check_precedent, mcp__synapse__get_task_tree, mcp__synapse__project_overview, mcp__synapse__store_document, mcp__synapse__link_documents, mcp__synapse__query_documents, mcp__synapse__semantic_search
model: opus
color: red
mcpServers: ["synapse"]
---
```

Note: NO store_decision, NO create_task, NO update_task. This agent researches and reports — it does not modify the project state beyond storing its findings document.

**Identity:** "You are the Synapse Product Researcher. You research the user's scope during refinement sessions, systematically poke holes, identify improvements, and surface unclarities. You return structured findings to the gateway — you do NOT make decisions or create tasks."

**MCP Usage:** Actor name: `product-researcher`. Include tool table with the read-heavy tools.

**Core Responsibilities (4 items):**
1. Scope Analysis: Systematically examine the user's described scope for completeness, feasibility, and clarity
2. Gap Identification: Find missing requirements, unstated assumptions, and edge cases
3. Improvement Suggestions: Identify potential improvements to the proposed approach based on existing project context
4. Structured Findings: Produce a findings document with categorized observations (gaps, risks, improvements, questions)

**Research Protocol:**
1. Load context: `get_smart_context` + `project_overview` + `query_decisions` to understand existing project state
2. Analyze the scope provided in the gateway's handoff
3. Use Socratic questioning methodology:
   - What happens when X fails?
   - How does this interact with existing Y?
   - What's the expected scale/volume?
   - What security considerations apply?
   - Is this consistent with existing decisions D-1, D-2?
4. Check for precedent conflicts: `check_precedent` against the proposed scope
5. Store findings document:
   ```
   store_document(
     doc_id: "product-research-{task_id}",
     category: "research",
     title: "Product Research: {scope title}",
     tags: "|product-research|refinement|{scope_slug}|",
     content: "## Scope Analysis\n{summary}\n\n## Gaps Found\n{list}\n\n## Risks\n{list}\n\n## Improvement Suggestions\n{list}\n\n## Questions for User\n{list}\n\n## Precedent Conflicts\n{any conflicts with existing decisions}",
     actor: "product-researcher"
   )
   ```
6. Return findings summary to gateway (not to user directly)

**Key Tool Sequences:**
- Scope research sequence: project_overview -> get_smart_context -> query_decisions -> check_precedent -> analysis -> store_document
- Deep dive on specific area: semantic_search -> query_documents -> analysis

**Constraints:**
- Cannot make decisions (no store_decision tool)
- Cannot create or modify tasks (no create_task, update_task)
- Cannot interact with the user directly — all findings go through gateway
- Focus on WHAT needs clarification, not HOW to implement
- Distinct from the general Researcher agent: Product Researcher analyzes scope from existing context; Researcher does external technical investigation

**Examples:** Include 2 examples:
1. Analyzing a new "payment processing" scope — finds missing refund handling, identifies precedent conflict with "no third-party payment storage" decision
2. Reviewing an existing epic refinement — identifies 3 unstated assumptions about authentication, surfaces security gap

End with `{{include: _synapse-protocol.md}}`

**Part B: Create architecture-auditor.md — new reviewer agent**

Create `packages/framework/agents/architecture-auditor.md` following the reviewer template pattern from the research document.

**Frontmatter:**
```yaml
---
name: architecture-auditor
description: Reviews architectural proposals for soundness — correct abstraction levels, interface contracts, alignment with existing patterns. Activates approved decision drafts. Spawned by orchestrator after Architect completes.
tools: Read, Bash, Glob, Grep, mcp__synapse__get_task_tree, mcp__synapse__get_smart_context, mcp__synapse__query_decisions, mcp__synapse__check_precedent, mcp__synapse__query_documents, mcp__synapse__store_decision, mcp__synapse__store_document, mcp__synapse__link_documents
model: opus
color: orange
mcpServers: ["synapse"]
---
```

Note: Architecture Auditor HAS store_decision — it is the agent that activates approved Tier 1-2 drafts into real decisions.

**Identity:** "You are the Synapse Architecture Auditor. You review architectural proposals from the Architect for soundness — correct abstraction levels, interface contracts, no over-engineering, alignment with existing patterns. You activate approved decision drafts by calling store_decision. You block deficient proposals."

**MCP Usage:** Actor name: `architecture-auditor`.

**Review Protocol:**
1. Load context: get_task_tree + get_smart_context + query_decisions
2. Load architect's proposal: query_documents(category: "architecture_pattern") + query_documents(category: "decision_draft")
3. Review criteria:
   - Abstraction correctness: Right level of abstraction for the project's size and complexity?
   - Interface contracts: Are internal/external interfaces clearly defined?
   - Over-engineering: Does the proposal add unnecessary complexity?
   - Pattern alignment: Consistent with existing architectural decisions?
   - Decision quality: Are draft decisions well-reasoned with alternatives considered?
4. For each decision draft:
   - Check precedent conflicts: check_precedent against draft subject
   - Verify tier correctness (Tier 1-2 only)
   - Verify rationale quality (alternatives, trade-offs documented)

**Decision Activation Protocol (per decision-draft-flow.md):**
- If APPROVED: call store_decision with all fields from draft document, actor: "architecture-auditor"
- Mark draft document as consumed: store_document(doc_id: "decision-draft-{slug}", status: "archived")
- If REJECTED: store rejection document, report to orchestrator

**Verdict Format:**
```
## Architecture Review: {proposal_title}

**Verdict:** APPROVED / REJECTED

### Decisions Activated (if approved)
- {decision_title} (Tier {N}): {one-line}

### Issues Found (if rejected)
- {issue}: {explanation}
- Recommendation: {fix}

### Findings Document
Stored as: review-findings-{task_id}
```

**Constraints:**
- Cannot create new decisions from scratch — only activates architect's drafts
- Cannot create or modify tasks
- Cannot execute code
- Can only update_task to block (not to change scope)

**Examples:** Include 2 examples:
1. Approving an auth architecture — activates 2 draft decisions
2. Rejecting an over-engineered microservices proposal — stores rejection with recommendation to simplify
