<objective>
Implement the get_smart_context MCP tool — a two-phase context assembly system. Overview mode scans documents metadata for summaries. Detailed mode fetches full content for specific doc_ids with 1-hop relationship graph expansion. Both modes respect a max_tokens budget.

Purpose: This is the primary tool agents use to gather context for tasks. Overview gives breadth (what docs exist); detailed gives depth (full content of specific docs + related). Token budget prevents context overflow.
Output: New get-smart-context.ts with TDD tests, following two-export pattern.

<task type="auto">
  <name>Task 1: Implement get_smart_context overview mode via TDD</name>
  <files>src/tools/get-smart-context.ts, test/tools/get-smart-context.test.ts</files>
  <action>
  Build get_smart_context with overview mode first. TDD: write tests, then implement.

  **Input schema (Zod):**
  ```
  project_id: string (slug regex)
  mode: enum("overview", "detailed")
  doc_ids: array of strings — optional, REQUIRED for detailed mode
  max_tokens: number 500-20000 optional default 4000
  category: enum(VALID_CATEGORIES) optional
  phase: string optional
  tags: string optional
  status: enum(VALID_STATUSES) optional
  ```

  **Overview mode implementation:**
  This is a METADATA SCAN, NOT a search. No embeddings, no vector search, no FTS.

  1. Query the `documents` table with metadata filters (project_id, category, phase, tags, status) plus `status != 'superseded'` (unless status filter explicitly overrides).
  2. Use the same predicate-building pattern as query-documents.ts (tag validation, pipe-delimited LIKE matching).
  3. Fetch up to 100 rows (generous upper bound — budget trimming handles the limit).
  4. Sort results by priority (ascending, nulls last) then by updated_at (descending):
     ```typescript
     rows.sort((a, b) => {
       const pa = (a.priority as number | null) ?? 999;
       const pb = (b.priority as number | null) ?? 999;
       if (pa !== pb) return pa - pb;
       return (b.updated_at as string).localeCompare(a.updated_at as string);
     });
     ```
  5. For each document, generate a ~100-token summary using `extractSnippet(content, "", 100)` — pass empty query since this is not search-driven; extractSnippet with no query match should return the first ~100 tokens.
  6. Accumulate summaries into token budget using `countTokens` from `gpt-tokenizer`:
     - Track `totalTokens`
     - For each doc: `const summaryTokens = countTokens(summary)`
     - If `totalTokens + summaryTokens > maxTokens`: STOP adding (drop-lowest strategy per locked decision — no partial documents)
     - Otherwise: add to results, increment totalTokens
  7. Return shape:
     ```typescript
     {
       mode: "overview",
       documents: Array<{
         doc_id: string;
         title: string;
         category: string;
         status: string;
         priority: number | null;
         summary: string;
         token_count: number;
       }>,
       total_documents: number,  // total matching before budget trim
       included_documents: number,
       total_tokens: number,
       max_tokens: number,
       source: "document",
     }
     ```

  **Test cases (RED first):**
  - Overview with no filters returns all non-superseded docs up to token budget
  - Overview with category filter narrows results
  - Overview respects max_tokens (insert 10 docs, set max_tokens=300 — should get ~3 docs)
  - Overview sorts by priority (priority 1 doc appears before priority 5)
  - Default max_tokens is 4000 when not specified
  - Overview with empty project returns empty results (no error)

  **What to AVOID:**
  - Do NOT call embed() in overview mode — this is a metadata scan
  - Do NOT truncate document summaries mid-content — drop entire document if it doesn't fit

<task type="auto">
  <name>Task 2: Add detailed mode with 1-hop graph expansion</name>
  <files>src/tools/get-smart-context.ts, test/tools/get-smart-context.test.ts</files>
  <action>
  Add detailed mode to the existing get_smart_context tool.

  **Detailed mode implementation:**
  1. Validate that `doc_ids` is provided and non-empty (throw error otherwise).
  2. Fetch requested documents by doc_id from `documents` table:
     ```typescript
     const docPredicate = `doc_id IN ('${docIds.join("','")}') AND project_id = '${projectId}' AND status != 'superseded'`;
     const requestedDocs = await docsTable.query().where(docPredicate).toArray();
     ```
  3. For each requested doc, include its FULL content (not summary). Track token count.
  4. Perform 1-hop graph expansion for each requested doc:
     - Reuse `getRelatedDocuments(dbPath, projectId, { project_id: projectId, doc_id: docId })` from get-related-documents.ts
     - Collect all related doc_ids (excluding already-requested docs)
     - Prioritize by relationship type:
       ```typescript
       const RELATIONSHIP_PRIORITY: Record<string, number> = {
         depends_on: 1, implements: 1,
         references: 2, related_to: 2,
         contradicts: 3, child_of: 3, supersedes: 3,
       };
       ```
     - Sort neighbors by priority value, then by title alphabetically
  5. Fetch full content of related documents.
  6. Accumulate into token budget:
     - Requested docs are ALWAYS included first (they are what the agent asked for)
     - Related docs fill remaining budget, in priority order
     - Use `countTokens(doc.content)` for each document
     - Stop adding when budget would be exceeded (drop-lowest, no partial)
  7. Return shape:
     ```typescript
     {
       mode: "detailed",
       documents: Array<{
         doc_id: string;
         title: string;
         category: string;
         status: string;
         content: string;
         token_count: number;
         is_requested: boolean;        // true for explicitly requested docs
         relationship_type?: string;   // only for 1-hop expanded docs
         relationship_direction?: "outgoing" | "incoming";
       }>,
       total_tokens: number,
       max_tokens: number,
       source: "document",
       graph_expansion: {
         requested_count: number,
         related_found: number,
         related_included: number,     // how many fit in budget
       },
     }
     ```

  **MCP tool registration:**
  Create `registerGetSmartContextTool(server, config)` following the two-export pattern. Input schema already defined in Task 1. Tool description should explain both modes clearly.

  **Test cases (RED first):**
  - Detailed mode with valid doc_ids returns full content
  - Detailed mode with 1-hop expansion includes related docs
  - depends_on/implements neighbors appear before references/related_to
  - Token budget is respected — large requested doc + many related docs gets truncated
  - Detailed mode without doc_ids throws error
  - Already-requested docs are not duplicated via graph expansion
  - Metadata filters work in detailed mode (filter which docs are eligible for expansion)

  **What to AVOID:**
  - Do NOT call embed() in detailed mode — this is a direct doc_id fetch + graph traversal
  - Do NOT duplicate the graph traversal logic from get-related-documents.ts — call the existing function
  - Do NOT include related docs that are superseded
  - Do NOT truncate document content mid-text — drop entire documents
