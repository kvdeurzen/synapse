# Phase 26.4 Plan 04: Behavioral Enforcement — Fresh Agent, Inline Spec, Two-Stage Review Summary

One-liner: Fresh-agent mandate, controller-curated inline spec, code-quality-reviewer two-stage dispatch, and verify-first validator independence wired into orchestrator, validator, and executor.

## What Was Built

This plan implements four behavioral changes from Superpowers practices into the core agent prompts:

### Task 1: Orchestrator Updates (commit d7ecaf7)

Three additions to `synapse-orchestrator.md`:

**Fresh Agent Per Task** — New section after Core Responsibilities mandating fresh instances for every Task tool call. Explicitly forbids reusing executor agents across tasks, passing accumulated context between agents, and documents that the orchestrator manages continuity via stage documents and pool state.

**Controller-Curated Context Rule** — Added to executor dispatch step in Wave Execution Protocol. For depth=3 tasks, the orchestrator pastes the full task spec inline in a `<TASK_SPEC>` block after the SYNAPSE HANDOFF. The executor does not need to load its own spec via doc_id — it receives the spec directly. Doc_id references are reserved for supplementary context.

**Code-Quality-Reviewer Dispatch (Step 3b)** — After validator PASS, the orchestrator spawns the Code Quality Reviewer. Routing logic handles APPROVED (proceed), NEEDS_REVISION (route back to executor with retry limit from `max_revision_retries` in trust.toml), and REJECTED (treat as validation failure, HALT wave). Code Quality Reviewer added to RPEV Agent Roles list and Upstream Context Routing table.

### Task 2: Validator + Executor Updates (commit a1aa8b7)

**Validator Independence Rule** — New `## Validation Sequence (Independence Rule)` section added before the TDD Verification Protocol. The 5-step sequence requires: (1) load spec, (2) run tests FIRST, (3) read code independently, (4) form independent assessment, (5) THEN read executor output. A CRITICAL warning explicitly states not to read `output_doc_ids` before Step 5 — doing so anchors assessment on executor self-report.

**Executor Verification Evidence** — New `### Verification Evidence (Required)` subsection added to the Output Contract. The implementation summary must include a `## Verification` section with: exact commands run (literal commands), full output (complete stdout/stderr, not summarized), and pass/fail determination per command. Red flags list covers "should/probably/seems to" language, missing test output, described-but-not-run commands, and summarized output.
