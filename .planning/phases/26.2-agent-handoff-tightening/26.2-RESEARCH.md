**Gateway Protocol**
- Gateway protocol lives as a template file at `.synapse/config/gateway-protocol.md`
- `/synapse:init` creates the template file; `synapse-startup.js` injects its contents into additionalContext at session start

- Input/output contracts are advisory — defined as sections in agent .md files

## Summary

Phase 26.2 is a pure internal refactoring of the Synapse agentic pipeline. All changes are to: (1) the LanceDB task schema in the server package, (2) the MCP tool handlers for task CRUD, (3) the shared agent protocol document, (4) the 14 individual agent `.md` files, and (5) the gateway setup (startup hook + init command + gateway template).

The codebase is well-understood — all relevant files were inspected. No external library research is needed. The primary planning questions are how to sequence the 24+ file changes across plans to minimize regression risk and how to write contract sections that are consistent across 14 agents.

The existing test suite (638 server tests, 103 framework tests) provides clear regression coverage for the schema layer. The schema test at `packages/server/test/db/schema.test.ts` has a hardcoded field-count assertion (`tasks schema has 19 fields`) that will break and must be updated to 23 when 4 fields are added.

**Primary recommendation:** Split work into 3 plans — (1) server schema + tools + tests, (2) shared protocol + orchestrator wiring, (3) per-agent input/output contracts + gateway setup. This ordering ensures schema is stable before framework changes reference it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Array storage in LanceDB | Native array column | JSON string in Utf8 | LanceDB doesn't support native string arrays — this is the established pattern (confirmed by CONTEXT.md decision) |
| Schema migration for existing rows | Migration script | null defaults | LanceDB `.update()` on existing tables handles new nullable columns without migration |
| Contract enforcement in code | Hook or validator code | Advisory sections in agent prompts | Decided — reviewers enforce, not runtime code |
| Re-embedding on spec/context fields | trigger embed on spec change | Skip embed | Only title+description feed the embedding; spec is structured data, not semantic search target |

## Common Pitfalls

### Pitfall 1: Forgetting to update TaskTreeNode interface AND node construction

**What goes wrong:** Arrow schema and Zod schema are updated, but `get-task-tree.ts` has a separate TypeScript interface (`TaskTreeNode`) and explicit node construction block (lines 350-378) that must also be updated. The row has the new fields but they never appear in the output.

**How to avoid:** In get-task-tree.ts, update both the `TaskTreeNode` interface AND the node construction object at line 350 to include `context_doc_ids: row.context_doc_ids as string | null` etc.

### Pitfall 2: Schema field count test breaks silently

**What goes wrong:** `schema.test.ts` has `expect(TASKS_SCHEMA.fields.length).toBe(19)`. After adding 4 fields this becomes 23. The test will fail with a confusing assertion error rather than a clear indication of what changed.

**How to avoid:** Update the `tasks schema has 19 fields` test to `toBe(23)` as part of Plan 1. Also update the validTask fixture in the Zod test section to include the 4 new nullable fields.

**Warning sign:** Any schema test failure during Plan 1 testing indicates this was missed.

### Pitfall 3: VALID_AGENT_ROLES change breaks create_task/update_task validation for old role names

**What goes wrong:** The current enum includes `decomposer`, `plan_reviewer`, `product_strategist` — these were removed in Phase 26.1 but may still appear in task records. Removing them from the enum means existing tasks with those values will fail Zod validation when fetched.

**How to avoid:** The CONTEXT.md decision says "keep old 10 roles + add 4 new" — do NOT remove old roles. The planning doc's "new 14-role enum" list omits `decomposer`, `plan_reviewer`, `product_strategist` but the CONTEXT.md overrides. Reconcile: keep ALL existing valid roles and add the new ones. The final list should be 14 roles (old ones renamed, not removed).

**Actual current roles (from task-constants.ts):** executor, validator, architect, decomposer, plan_reviewer, integration_checker, debugger, codebase_analyst, product_strategist, researcher (10 roles)

**New roles to add (from CONTEXT.md):** architecture_auditor, planner, plan_auditor, task_designer, task_auditor, synapse_orchestrator, product_researcher (7 new)

**Roles to retire (from CONTEXT.md "expand: keep old 10 roles + add 4 new"):** CONTEXT.md says keep old + add new, but the planning doc lists exactly 14. The CONTEXT.md decision wins — verify which old roles are still referenced in agents.toml before removing any.

### Pitfall 4: _synapse-protocol.md grows beyond target size

**What goes wrong:** Currently 43 lines. Adding 5-step loading protocol (~30 lines) + tag vocab (~15 lines) + provides vocab (~15 lines) = ~60 additional lines, landing at ~103 lines. The planning doc says "stay under ~120 lines." Each agent file that drops its Task Start Protocol section should save ~10-15 lines in exchange.

**How to avoid:** Keep the shared protocol additions terse — use tables rather than prose. Reference rather than duplicate existing sections.

### Pitfall 5: update_task does not return new fields in UpdateTaskResult

**What goes wrong:** `UpdateTaskResult` only returns `task_id`, `status`, `is_blocked`, `is_cancelled`, `updated_at`, `changed_fields`. If orchestrator updates `spec` or `context_doc_ids`, it won't see the new values in the return value.

**How to avoid:** Either accept this limitation (orchestrators that write fields know what they wrote), or add new fields to `UpdateTaskResult`. Given that callers are agent prompts that already know the value they wrote, the simpler path is to leave `UpdateTaskResult` unchanged and just ensure `changed_fields` includes the new field names.

### Pitfall 6: Gateway injection order in synapse-startup.js

**What goes wrong:** The gateway protocol template is injected into additionalContext, but if it's injected before the RPEV matrix or tier context, the order may confuse agents that expect project context first.

**How to avoid:** Inject gateway-protocol.md content at the end of contextParts (after tier, RPEV, domain, skill contexts), so it acts as a behavioral overlay on top of established project context.

### 5-step context loading (for _synapse-protocol.md)

```markdown
## Mandatory Context Loading Sequence

Every agent MUST complete these steps before beginning agent-specific work:

1. **Parse SYNAPSE HANDOFF block** — extract `project_id`, `task_id`, `hierarchy_level`, `doc_ids`, `decision_ids`
2. **Load task** — `get_task_tree(project_id, root_task_id: task_id)` — read `spec`, `context_doc_ids`, `context_decision_ids` fields
3. **Load documents** — if task has `context_doc_ids`: `get_smart_context(mode: "detailed", doc_ids: [parsed from context_doc_ids])`. Else if handoff has doc_ids: use those. Else: `get_smart_context(mode: "overview", max_tokens: {level-appropriate})`
4. **Load decisions** — if task has `context_decision_ids`: `query_decisions` for those IDs
5. **Begin agent-specific work** — do NOT begin before completing steps 1-4
```
