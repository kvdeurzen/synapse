**CodeSearchResult interface:**

export interface CodeSearchResult {

**Zod input schema (SearchCodeInputSchema):**
- `project_id`: z.string().regex(/^[a-z0-9][a-z0-9_-]*$/)
- `query`: z.string().min(1)
- `mode`: z.enum(["semantic", "fulltext", "hybrid"]).default("hybrid") — per CONTEXT.md
- `language`: z.string().optional() — e.g. "typescript", "python", "rust"
- `symbol_type`: z.string().optional() — e.g. "function", "class", "method"
- `file_pattern`: z.string().optional() — glob syntax, e.g. "src/**/*.ts"
- `limit`: z.number().int().min(1).max(50).optional().default(10) — CONTEXT.md says default 10
- `min_score`: z.number().min(0).max(1).optional().default(0.0)

**globToSqlLike helper function:**
```typescript
function globToSqlLike(glob: string): string {
  return glob
    .replace(/%/g, "\\%")
    .replace(/_/g, "\\_")
    .replace(/\*\*/g, "%")
    .replace(/\*/g, "%")
    .replace(/\?/g, "_");
}
```

**Code-specific predicate builder (inline, NOT using buildSearchPredicate):**
DO NOT use `buildSearchPredicate` from search-utils.ts — that does cross-table joins to the documents table. Code search filters (language, symbol_type, file_pattern) are direct columns on code_chunks. Build the predicate inline:
```typescript
const parts: string[] = [`project_id = '${projectId}'`];
if (language) parts.push(`language = '${language}'`);
if (symbol_type) parts.push(`symbol_type = '${symbol_type}'`);
if (file_pattern) parts.push(`file_path LIKE '${globToSqlLike(file_pattern)}'`);
const predicate = parts.join(" AND ");
```
Use exactOptionalPropertyTypes-safe pattern: build filter conditionally with `if (val !== undefined)` before using val. Do NOT spread Zod-parsed objects.

**searchCode core function (exported for testability):**
Signature: `async function searchCode(dbPath, projectId, args, config) -> CodeSearchResult`

Three modes:
1. **hybrid (default):** Check getOllamaStatus(). If not "ok", fall back to FTS-only (same pattern as hybrid-search.ts). If ok: embed query via embed(), then run hybrid search with RRFReranker:
   ```typescript
   const reranker = await rerankers.RRFReranker.create(60); // MUST use .create(), NOT new
   const rows = await codeChunksTable.query()
     .nearestTo(queryVector).distanceType("cosine")
     .fullTextSearch(query).where(predicate)
     .rerank(reranker).limit(limit).toArray();
   ```
   Score extraction: try `_relevance_score`, then `_score`, then position-based `1/(i+60)`. Clamp to [0,1].

2. **semantic:** Require Ollama (throw OllamaUnreachableError if not "ok"). Embed query. Use `.nearestTo(queryVector).distanceType("cosine").where(predicate).limit(limit * 2)`. Score via normalizeVectorScore(row._distance).

3. **fulltext:** No Ollama needed. Use `.fullTextSearch(query).where(predicate).limit(limit * 2)`. Score via normalizeFtsScore(row._score).

Wrap FTS queries in try/catch — empty code_chunks table may throw on FTS (Pitfall 3 from RESEARCH.md). On FTS error, return empty results `{ results: [], total: 0, search_type }`.

**Result building (all modes):**
For each row after scoring and min_score filtering:
- `scope_chain`: Parse from stored dot-notation string. If null/empty, return empty array. Otherwise `(row.scope_chain as string).split(".")`. Do NOT add type labels — stored data has no type prefixes (Pitfall 1).
- `content`: Use `extractSnippet(row.content as string, query)` for trimmed snippet. Do NOT return full content (locked decision).
- All other fields: direct extraction from row (chunk_id, file_path, symbol_name, symbol_type, start_line, end_line, language).
- `source`: always "code".

**registerSearchCodeTool MCP registration:**
Follow the exact same pattern as registerHybridSearchTool. Tool name: `"search_code"`. Description should mention all three modes, code-specific filters, and that hybrid is default. inputSchema uses z.object with .describe() on each field.
