    Full rewrite of `packages/framework/skills/brainstorming/SKILL.md` following the decisions in `.planning/brainstorm output/brainstorm-skill-redesign.md`. This is a FULL REWRITE, not a patch.

    **Preserve from current skill:**
    - Frontmatter: `name: brainstorming`, `disable-model-invocation: true`, `user-invocable: false`
    - WHAT before HOW principle
    - No hybrid defaults — force clear choice
    - YAGNI discipline
    - Assumptions explicitly labeled
    - Open questions listed separately
    - Time-box: 3 rounds of questions without consensus = escalate
    - All quality criteria from current skill

    **New structure — 7-step sequential checklist:**

    1. **Load project context** — query Synapse DB (smart_context, recent decisions, related documents). NOT raw file exploration.
    2. **Scope gate** — map topic to hierarchy (Epic = decompose into Features first; Feature/WP = proceed). If Epic-sized, decompose before brainstorming each Feature.
    3. **Ask clarifying questions** — ONE question per message (this is the key behavioral enforcement rule). Prefer multiple choice. Understand purpose, constraints, success criteria.
    4. **Propose approaches** — enumerate at least 3 distinct approaches neutrally (description, benefits, drawbacks, "best when", risk assessment), THEN recommend with explicit selection criteria.
    5. **Present design** — scale to complexity: simple = one message; complex (3+ sections) = section-by-section with approval after each.
    6. **Store design document** — save to Synapse DB via store_document (category: plan or architecture_decision). Store decisions separately via store_decision. Link related documents.
    7. **Offer to continue** — ask user if they want to proceed to Plan phase or stop.

    **Hard gate:**
    ```
    <HARD-GATE>
    Do NOT take any implementation action until a design has been presented and the
    user has approved it. This applies to EVERY topic regardless of perceived
    simplicity. "Simple" topics are where unexamined assumptions cause the most
    wasted work. The design can be brief for truly simple topics, but it MUST exist
    and be approved.
    </HARD-GATE>
    ```

    **One-question-at-a-time behavioral enforcement:**
    - Only one question per message. No exceptions.
    - Break multi-part exploration into sequential questions across multiple messages.
    - Prefer multiple-choice questions — faster, more actionable.
    - Open-ended fine when possibility space is too large for predefined options.
    - Never combine a question with a long explanation.

    **Anti-rationalization table (inline in the skill):**
    Add a `## Anti-Rationalization` section with a markdown table. Source rationalizations from the Superpowers brainstorming skill analysis and the locked decisions in CONTEXT.md. At minimum include:
    - "This is too simple to need a design" — always false; brief design for simple topics, but skip = skipping where bad assumptions get caught
    - "The user seems impatient, I should skip the questions" — impatience is a signal to ask BETTER questions, not fewer
    - "I already know the right approach from context" — your first instinct may anchor on the wrong constraint; the process surfaces what you would miss
    - "Multiple questions would be more efficient" — one question at a time produces more actionable answers; batching produces vague responses
    - "The design approval step is just a formality" — it is the step where bad assumptions get caught; the process IS the value

    Use the WebFetch or WebSearch tools to pull relevant content from the Superpowers framework documentation to ensure these rationalizations are externally sourced, not self-generated. The Superpowers `using-superpowers` bootstrap skill and brainstorming skill are the primary sources.

    **Format:** Keep the 5-section SKILL.md format (Conventions, Quality Criteria, Vocabulary, Anti-patterns, Commands) but reorganize Conventions to be the 7-step checklist with the hard gate, and add the Anti-Rationalization section after Anti-patterns.
