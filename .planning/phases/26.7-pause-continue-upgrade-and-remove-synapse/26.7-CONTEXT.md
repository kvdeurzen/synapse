# Phase 26.7: Pause, Continue, Upgrade, and Remove Synapse - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Four new lifecycle management slash commands: `/synapse:pause`, `/synapse:continue`, `/synapse:upgrade`, `/synapse:remove`. Covers pausing work mid-session or at session boundary, resuming with full context, upgrading Synapse versions with changelog diff, and cleanly removing Synapse from a project. Does NOT cover new RPEV pipeline features or agent changes.

</domain>

<decisions>
## Implementation Decisions

### Pause behavior
- Support BOTH mid-session interrupt and session-boundary pausing
- Mid-session interrupt: present user with choice — graceful drain (let in-flight agents finish) or immediate stop with revert of partial work
- Save a full state snapshot: pipeline position, active decisions, in-progress task IDs, pool state
- `/synapse:pause` is invokable any time, not just during active RPEV pipeline — even outside RPEV, saves current session state (open refinements, what user was working on)

### Continue behavior
- `/synapse:continue` auto-detects what was paused from the snapshot and picks up automatically — no user menu
- Startup hook auto-detects pause snapshot on session start and prompts user: "You paused [phase/stage]. /synapse:continue to resume."

### Upgrade mechanism
- `/synapse:upgrade` as convenience slash command wrapping `install.sh` under the hood — two entry points (command + script), one mechanism
- Overwrite all framework files (agents, hooks, commands, skills) AND server source — no schema migration, framework files are Synapse-owned
- Show changelog diff between current and target version before applying — user confirms
- Auto-detect both current installed version (from `project.toml` field) and latest available version from GitHub — user just runs `/synapse:upgrade`
- Version tracked in `project.toml` as a `synapse_version` field (set by install.sh and upgrade)
- No backup of previous installation — framework files are reproducible from any version tag, git history is the backup
- Auto-commit all changed framework files after upgrade with conventional commit: `chore(synapse): upgrade to {version}`
- Hot-reload updated framework files in-session if possible; fall back to restart recommendation if critical files changed (hooks, settings.json)

### Remove behavior
- `/synapse:remove` deletes everything Synapse-owned: .claude/agents/, .claude/hooks/synapse-*, .claude/commands/synapse/, .claude/server/, Synapse entries from .mcp.json and settings.json, and .synapse/ config dir
- Prompt user specifically about LanceDB database deletion — let them choose to preserve or delete project data
- Remove the Synapse Gateway Protocol section from CLAUDE.md automatically
- Require confirmation before executing — show file list + data summary, explicit 'yes' required
- The remove command deletes itself as its last step

### Command interface
- Four new slash commands: `/synapse:pause`, `/synapse:continue`, `/synapse:upgrade`, `/synapse:remove`
- Total command set after this phase: init, map, refine, status, focus, pause, continue, upgrade, remove

### Claude's Discretion
- Exact snapshot format and storage location (file-based vs Synapse document)
- How hot-reload detection works (which file changes require restart)
- Graceful drain implementation details (how to signal in-flight agents)
- Revert mechanism for immediate stop (git revert vs file restore)

</decisions>

<specifics>
## Specific Ideas

- Upgrade should trigger git action because Synapse files are committed to the repo — auto-commit is natural
- Pause snapshot should be rich enough to resume without confusion — "full snapshot" preferred over minimal state
- Remove should handle the self-deletion paradox cleanly (slash command removes the file that defines it)

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `install.sh`: Already has tarball download, file copy, dedup logic for hooks/settings.json, --version flag support — upgrade command wraps this
- `.synapse/state/statusline.json`: Existing state file pattern — pause snapshot could use similar approach
- Stage documents (`rpev-stage-{task_id}`): Existing pipeline state persistence — pause reads these
- Pool state document (`pool-state-{project_id}`): Tracks active agent slots — pause needs to capture this
- `synapse-startup.js`: Startup hook that reads project.toml — can detect pause snapshot and inject resume prompt

### Established Patterns
- Copy-based install: Framework files are Synapse-owned copies, not symlinks — safe to overwrite on upgrade
- Conventional commits: `chore(synapse): upgrade to {version}` fits the established commit format
- JSON merge via `bun -e`: Used in install.sh for .mcp.json and settings.json — remove needs inverse operations
- project.toml as metadata hub: Adding `synapse_version` field follows established pattern

### Integration Points
- `synapse-startup.js`: Needs pause-detection logic for auto-resume prompt
- `.mcp.json`: Remove must strip Synapse server entries
- `.claude/settings.json`: Remove must strip Synapse hook entries
- `CLAUDE.md`: Remove must strip Gateway Protocol section
- `project.toml`: Upgrade writes `synapse_version` field; remove deletes the file

</code_context>
