6. **testing-strategy/SKILL.md** -- Language-agnostic testing pyramid strategy.
   Sources: Martin Fowler's Practical Test Pyramid, awesome-cursorrules Vitest/Cypress rules, caduh.com Testing Pyramid guide, AI Hero TDD Claude Code skill.
   Conventions (from condensed takeaways):
   - Pyramid ratio: 60-70% unit / 20-30% integration / 5-10% E2E -- enforce deliberately
   - Test behavior, not implementation: public interfaces only; private methods indicate design problems
   - Mocking discipline: mock at boundaries (external services, I/O, clock), not internal collaborators
   - No duplication across levels: if a unit test covers an edge case, E2E should not repeat it
   - TDD vertically: one test -> one implementation -> repeat; never bulk-write tests
   - Flake prevention: control time, randomness, and network deterministically; never sleep
   - Arrange-Act-Assert (or Given-When-Then) structure consistently across all levels
   - Push tests as far down the pyramid as possible (Fowler's universal rule)
   Quality Criteria (derive from sources):
   - Unit tests run in < 3 min total; integration < 7 min; E2E smoke < 3 min on PRs (caduh.com time budgets)
   - Coverage: ~80% line / ~60% branch overall; gate on changed-file coverage not repo-wide (caduh.com)
   - Each test is independent -- no shared mutable state between test functions (Fowler + awesome-cursorrules)
   - External services mocked in unit tests; real dependencies used in integration tests (Fowler two-level distinction)
   - Clear mocks between tests: reset/clear before each test (awesome-cursorrules)
   Anti-patterns (from sources):
   - Testing private methods directly (refactor to expose through public interface instead) (Fowler)
   - Bulk-writing tests before implementation ("tests imagined behavior, not observed behavior") (AI Hero TDD)
   - Hard-coded waits/sleeps in tests (use deterministic signals) (caduh.com + awesome-cursorrules)
   - Coverage theater: asserting code ran without asserting correctness (caduh.com)
   - Mocking the code under test (creates false positives) (caduh.com)
   - Snapshot tests for non-stable output (caduh.com)
   Commands section: "This is a strategy skill. For actual test commands, read the project's language-specific testing skill (e.g., pytest, cargo-test, vitest, go-test)."

7. **architecture-design/SKILL.md** -- Architectural conventions for clean boundaries and decision records.
   Sources: "The Architecture is the Prompt" hexagonal design (muthu.co), joelparkerhenderson/architecture-decision-record, Kirill Markin cursor IDE rules, DDD+Hexagonal for AI Agents (Khosravi).
   Conventions (from condensed takeaways):
   - Dependency direction is non-negotiable: infrastructure -> application -> domain; never reverse (Hexagonal + DDD+Hexagonal)
   - Interface before implementation: define ports/contracts first, then write adapters (Hexagonal)
   - ADR every significant decision: Title / Status / Context / Decision / Consequences; one decision per ADR; immutable once accepted (joelparkerhenderson)
   - Functional core, imperative shell: pure business logic at center; I/O and side effects at edges (Hexagonal)
   - Architecture as AI prompt: clean boundaries let agents work in isolated slices without full system context (muthu.co)
   - Single responsibility + composition: small, focused components composed rather than monolithic classes (Hexagonal + Markin)
   - Code is the primary documentation -- use clear naming, types, and docstrings; minimize external docs (Markin)
   - Y-statement format for decisions: "In the context of X, facing concern Y, we decided Z, to achieve Q, accepting R" (joelparkerhenderson)
   Quality Criteria (derive from sources):
   - No domain layer imports from infrastructure layer (dependency direction check)
   - Every port has a defined interface before any adapter implementation exists
   - ADR files use present-tense imperative, lowercase-with-dashes naming (joelparkerhenderson)
   - Each component handles one concern; extract when responsibility count exceeds 1
   - Repository interfaces defined in domain layer, implemented in infrastructure layer (DDD+Hexagonal)
   Anti-patterns (from sources):
   - Domain objects importing infrastructure libraries (violates dependency direction) (Hexagonal + DDD)
   - Making decisions without recording them (ADR discipline) (joelparkerhenderson)
   - Verbose documentation over clean code structure (Markin: "code is the primary documentation")
   - Monolithic classes combining business logic with I/O (violates functional core/imperative shell) (Hexagonal)
   - Duplicating documentation across locations (Markin: single source of truth)
   Commands section: "This is a cognitive skill -- no executable commands. ADR files go in `docs/decisions/` or `docs/adr/` using format: `NNN-title-with-dashes.md`."

8. **security/SKILL.md** -- Security conventions for application and LLM-assisted development.
   Sources: Cloud Security Alliance R.A.I.L.G.U.A.R.D. framework, Sean Goedecke "Principles for Coding Securely with LLMs", Van-LLM-Crew/cursor-secure-coding (OWASP/ASVS Level 1/2).
   Conventions (from condensed takeaways):
   - Parameterized queries always: never concatenate user input into SQL/queries (CSA + Van-LLM-Crew)
   - Secrets in vaults, never in code: environment variables minimum; proper secret management preferred (CSA + Van-LLM-Crew)
   - Least privilege everywhere: DB permissions, API scopes, agent tool access (CSA + Van-LLM-Crew)
   - LLM output = untrusted input: sanitize before execution or display; treat as user-submitted content (Goedecke)
   - No unsafe functions: `eval()`, `exec()`, raw SQL string construction are banned (CSA)
   - Dependency audit continuously: scan for CVEs; approve new dependencies explicitly (Van-LLM-Crew)
   - Human in the loop for destructive operations: shell commands, DB writes, user impersonation (Goedecke)
   - Validate and sanitize all user input at boundaries; escape output in HTML, JS, and SQL contexts (CSA)
   Quality Criteria (derive from sources):
   - Zero hardcoded secrets in source code or build artifacts (CSA + Van-LLM-Crew)
   - All database queries use parameterized statements or ORM -- no string concatenation (CSA + Van-LLM-Crew)
   - Security headers set: CSP, HSTS, X-Content-Type-Options (CSA)
   - No `eval()`, `exec()`, or dynamic code construction from user input in codebase (CSA)
   - Dependency vulnerability scan passes with no critical/high CVEs (Van-LLM-Crew)
   - LLM-generated outputs sanitized before DOM insertion or command execution (Goedecke)
   Anti-patterns (from sources):
   - String concatenation for SQL queries (CSA: "use parameterized queries for all database access")
   - Hardcoding API keys, passwords, or tokens in source files (CSA + Van-LLM-Crew)
   - Trusting LLM output as safe for execution without sanitization (Goedecke: "treat LLM output like user input")
   - Exposing stack traces or internal errors to users (CSA)
   - Logging sensitive data, secrets, or session tokens (Van-LLM-Crew)
   - Custom authentication implementations when established frameworks exist (CSA)
   - Granting LLM tools broader access than the end user would have (Goedecke: "scope LLM tools to current user permissions")
   Commands: `npm audit` / `cargo audit` / `pip-audit` (language-specific), `trivy fs .` (container/filesystem), `gitleaks detect` (secret scanning). List all tools neutrally; agent picks the relevant one for the project stack.

9. **brainstorming/SKILL.md** -- Structured technical problem-solving before implementation.
   Sources: ratacat/claude-skills brainstorming skill, claude-cortex brainstorming skill, TechnickAI brainstorming skill.
   Conventions (from condensed takeaways):
   - One question at a time: never overwhelm with multiple simultaneous questions (ratacat + TechnickAI)
   - Present options before recommending: enumerate at least 3 distinct approaches with explicit tradeoffs (claude-cortex + TechnickAI)
   - Structure each option: description / benefits / drawbacks / "best when" conditions / risk assessment (claude-cortex)
   - YAGNI discipline: simplest solution that solves the stated problem wins unless clear reason otherwise (ratacat + TechnickAI)
   - Document the decision: capture what, why, key decisions, and open questions in markdown before moving to implementation (ratacat + claude-cortex)
