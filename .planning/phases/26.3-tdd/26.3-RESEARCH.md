## User Constraints (from CONTEXT.md)

### Locked Decisions

**Pipeline Position and Agent Design:**
- Test-designer is a new standalone agent (no dedicated reviewer)
- Position: after task-designer, before task-auditor
- Model: Opus (test design requires deep requirement comprehension and edge-case thinking)
- Task-auditor remains Sonnet (triangulation is verification/pattern-matching work)
- Executor remains Sonnet (not downgraded; validate TDD first, optimize later)
- TDD is always-on — no per-project or per-task toggle

**Test-Designer Capabilities:**
- Has Read, Write, Edit, Glob, Grep, Bash, search_code tools — can inspect existing test files and write tests to disk
- Writes actual test files to disk (not documents for executor to materialize)
- Runs tests to verify RED (tests fail for the right reasons, not syntax/import errors)
- Stores a summary document via store_document (test file paths, @requirement mapping, RED verification results) — provides: test-contract
- Appends test file paths to task description
- Loads project-wide skills (same as other agents) — no special role_skills
- Gets test runner command from loaded skill content (e.g., vitest SKILL.md Commands section)
- Explicit "Convention Discovery" section in prompt: read existing test files, match naming, use existing helpers/fixtures

**Test Expectations Flow:**
- Planner writes test expectations as prose in task description (no new structured field)
- Test expectations are reframed as test-designer input, not validator guidance
- Test-designer transforms prose into executable tests; prose is an intermediate artifact

**@requirement Tracing:**
- Hard requirement: every test MUST have a @requirement comment linking to the specific plan requirement it verifies
- Task-auditor checks for coverage — requirements without tests and tests without requirements are flagged

**Anti-Patterns (embedded in prompts):**
- Test-designer gets 5 test-design anti-patterns; executor gets 4 implementation anti-patterns
- MUST NOT rules (hard): executor modifying/deleting test-designer tests, testing mock behavior
- Advisory rules (soft): everything else (over-specified tests, mock-heavy unit tests, etc.)
- Language-specific test conventions come from existing skills, not agent prompts (Phase 20 language-agnostic principle)

**Verification Protocol:**
- Both test-designer AND task-auditor independently run tests
- Test-designer runs to verify RED; task-auditor independently runs to confirm
- Task-auditor triangulates: planner requirements, task-designer spec, test-designer tests

**Executor Constraints:**
- MUST NOT modify or delete test-designer's tests — can only add supplementary unit tests
- Supplementary tests follow TDD discipline (write failing test, verify RED, implement, verify GREEN)
- Structured status reporting via output document: DONE, DONE_WITH_CONCERNS, BLOCKED, NEEDS_CONTEXT
- Self-review before handoff: completeness, quality, discipline, supplementary tests

**Escalation Protocol:**
- Executor reports BLOCKED to orchestrator (via output document with details)
- Orchestrator reads block document and decides routing:
  - Implementation gap → back to executor with guidance
  - Wrong test assumption → escalate to planner (planner owns the contract)
  - Ambiguous → orchestrator dispatches task-auditor to evaluate
- DONE_WITH_CONCERNS: validator reviews the concern alongside implementation

**Provides Vocabulary:**
- Add 'test-contract' as 12th slug to the provides vocabulary
- Test-designer tags output with |provides:test-contract|
- doc_id convention: test-designer-test-contract-{task_id}

**Config Changes:**
- agents.toml: register test-designer with tools, model, description
- trust.toml: test-designer tier_authority = [] (writes tests, not decisions)
- Orchestrator prompt: add test-designer dispatch step in per-task pipeline

## Summary

Phase 26.3 introduces TDD as a core methodology by adding a **test-designer** agent to the Synapse RPEV pipeline and updating six existing artifacts: task-auditor.md, executor.md, validator.md, planner.md, _synapse-protocol.md (provides vocabulary), and both agents.toml and trust.toml config files. The orchestrator also needs one dispatch step added.

The design is fully specified in `.planning/brainstorm output/tdd-as-agent-methodology.md`, which resolves all open design threads. The implementation is almost entirely agent prompt engineering — creating one new .md file and surgically modifying seven existing files. No new TypeScript, no schema changes, no new MCP tools required.

The central idea is that the **test-designer writes executable failing tests before execution begins**, breaking the tautology problem (executor writing tests for code they just wrote). The test-designer inherits codebase knowledge from the task-designer's spec, matches existing conventions by reading real test files, and stores a test-contract document that the task-auditor independently verifies before any execution starts.

**Primary recommendation:** Implement as sequential prompt-engineering tasks organized by agent. New test-designer agent first (provides the template), then task-auditor expansion, executor constraints, validator simplification, planner update, config changes, and skill updates last.

### Pattern 9: Validator Simplification

With test-designer tests pre-written and task-auditor having already verified RED, the validator's job simplifies:

1. **Confirm test-designer tests still pass** (they were RED before, should be GREEN now)
2. **Confirm test-designer tests were not modified** (diff or content check against test-contract doc)
3. **Confirm executor supplementary tests pass** (if any)
4. **Spec compliance check** — read implementation against spec requirements independently (does not rely on executor's self-report)

The validator currently runs tests via Bash as part of verification — this stays. The new responsibility is the immutability check: load the test-contract document (test-designer-test-contract-{task_id}) and compare listed test files against current filesystem state.

### Anti-Patterns to Avoid

- **Duplicating anti-patterns already in skills:** The test-designer prompt should NOT repeat vitest/pytest-specific patterns. Direct agents to read the skill instead. (Phase 20 language-agnostic principle)
- **Giving test-designer query_documents without search_code:** The test-designer needs search_code to find existing test patterns and helpers in the codebase.
- **Adding test-designer as a new pipeline stage without updating orchestrator RPEV agent roles table:** The orchestrator has a table listing all agents — test-designer must appear there.
- **Not adding Bash to test-designer tools:** Required for verify-RED step.
- **Not adding Write/Edit to test-designer tools:** Required to write test files to disk (locked decision).

## Common Pitfalls

### Pitfall 1: Test-Designer Overwrites Task Description Instead of Appending

**What goes wrong:** test-designer calls `update_task(description: "new content")` with full replacement, losing the planner's original WHAT content and prose test expectations.

**Why it happens:** update_task description param replaces, not appends. Agent might assume it should write the "complete updated description."

**How to avoid:** The test-designer prompt must explicitly state: "Append test file paths to the end of the existing task description using a '## Test Files' section. Do NOT overwrite the existing description content."

**Warning signs:** Task description no longer contains planner's original acceptance criteria after test-designer runs.

### Pitfall 2: Test-Designer Writes Implementation-Passing Tests, Not Requirement-Verifying Tests

**What goes wrong:** Tests are written to pass with a trivial implementation, not to verify the behavioral requirements from the plan.

**Why it happens:** Agent defaults to writing "easy" tests that clearly show red-to-green, rather than tests that encode the actual requirements.

**How to avoid:** The anti-patterns section must include: "MUST NOT: Write tests that pass with a trivial/stub implementation that doesn't fulfill the requirement. Every test must encode a specific behavioral requirement from the plan."

**Warning signs:** All tests pass after executor writes a function that returns hardcoded values.

### Pitfall 3: Task-Auditor Runs Tests Before Test Files Exist

**What goes wrong:** Task-auditor tries to run tests before test-designer completes, or test-designer's test files are not yet visible on disk.

**Why it happens:** Orchestrator dispatches test-designer and task-auditor out of order or before test-designer's Task call completes.

**How to avoid:** Orchestrator protocol must be explicit: test-designer Task call MUST complete before task-auditor is dispatched. No parallelization of these two steps.

**Warning signs:** Task-auditor reports "test files not found."

### Pitfall 4: Executor Reports BLOCKED Without Sufficient Diagnostic Detail

**What goes wrong:** Executor stores a BLOCKED status document with vague "test doesn't pass" content, giving the orchestrator nothing to route with.

**Why it happens:** Executor doesn't know what the orchestrator needs to make a routing decision.

**How to avoid:** The executor prompt must specify the exact required fields in a BLOCKED report: failing test names + assertion messages, what was attempted, why executor believes this is a test assumption problem vs implementation gap.

**Warning signs:** Orchestrator cannot determine whether to route back to executor, test-designer, or planner from the BLOCKED document.

### Pitfall 5: @requirement Comments Use Wrong Format

**What goes wrong:** Tests use inconsistent @requirement formats (`// @req AUTH-01` vs `# @requirement: AUTH-01` vs inline comments) making task-auditor's coverage check unreliable.

**Why it happens:** No format was specified explicitly and agents invent their own.

**How to avoid:** Specify the exact format in the test-designer prompt AND the task-auditor prompt: `@requirement {REQ-ID}: {one-line description}`. Language-specific comment syntax (// vs #) comes from the loaded skill.

**Warning signs:** Task-auditor cannot consistently find @requirement markers across test files.

### Pitfall 6: Provides Vocabulary Extended Incorrectly

**What goes wrong:** `_synapse-protocol.md` is updated with `test-contract` slug but the description row format doesn't match the existing table style, or the producer column is wrong.

**Why it happens:** Copyedit error when editing the table.

**How to avoid:** Read the existing table carefully and match format exactly. Producer should be `test-designer` only (not task-auditor, which verifies but does not provide).

### Pitfall 7: Orchestrator Escalation Routing Creates Infinite Loops

**What goes wrong:** Orchestrator routes BLOCKED back to executor repeatedly, or routes to planner but doesn't update the test-designer after planner revises requirements.

**Why it happens:** The escalation protocol specifies routing destinations but not what happens after the destination acts.

**How to avoid:** The orchestrator section must specify the full cycle: planner revises requirement → test-designer produces updated tests → task-auditor re-verifies → executor retries. Include max cycle count (same as existing: 3).

```
2. Read 2-3 representative test files — identify: test runner import, describe/it structure, fixture patterns, helper imports
3. Note: test file naming convention, test directory location, assertion library in use
Match ALL new test files to discovered conventions.
```

**Verify RED Protocol:**
```
### Verify RED (MANDATORY before storing test-contract)
1. Run tests: Bash("{test_command} {test_file_paths}") — get the test runner command from the loaded testing skill
2. Confirm: tests FAIL (non-zero exit code)
3. Inspect failures: each failure must say "X is not a function" or "Cannot find module" or assertion failure — NOT syntax error or import error
4. If tests have syntax/import errors: fix them and re-run until failures are behavioral
5. If tests pass immediately: the tests are wrong — they must fail before implementation
```

**@requirement format:**
```

// Example: @requirement AUTH-01: signToken returns signed JWT for access type
```
