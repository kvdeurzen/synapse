Create the MCP server with stdio transport and two starter tools.

**src/tools/ping.ts — Ping tool handler:**
- Export a function `registerPingTool(server: McpServer, config: SynapseConfig)` that registers the ping tool
- Input schema: `z.object({})` (no inputs)
- Handler logic:
  - Create tool logger via `createToolLogger('ping')`
  - Log invocation at info level
  - Build result: `{ success: true, data: { status: 'ok', version: '0.1.0', uptime: process.uptime(), dbPath: config.db, ollamaUrl: config.ollamaUrl, ollamaStatus: 'unknown' (placeholder -- real health check comes in Phase 3), toolCount: server.getRegisteredTools?.length ?? 2, embedModel: config.embedModel } }`
  - Note: The McpServer may not expose a getRegisteredTools method. Check the SDK API — if not available, hardcode the tool count or track it manually in server.ts. Do NOT break the build trying to call a non-existent method.
  - Log completion with duration
  - Return: `{ content: [{ type: 'text', text: JSON.stringify(result) }] }`
- Use the `ToolResult` envelope from types.ts for the response shape

**src/tools/echo.ts — Echo tool handler:**
- Export a function `registerEchoTool(server: McpServer)` that registers the echo tool
- Input schema: `z.object({ message: z.string().describe('Message to echo back') })`
- Handler logic:
  - Create tool logger via `createToolLogger('echo')`
  - Log invocation at info level
  - Build result: `{ success: true, data: { message: args.message } }`
  - Log completion with duration
  - Return: `{ content: [{ type: 'text', text: JSON.stringify(result) }] }`

**src/server.ts — MCP server setup:**
- Import `McpServer` from `@modelcontextprotocol/sdk/server/mcp.js`
- Import `StdioServerTransport` from `@modelcontextprotocol/sdk/server/stdio.js`
- Import tool registration functions from tools/ping.ts and tools/echo.ts
- Import logger, config types
- Export `createServer(config: SynapseConfig): McpServer` that:
  1. Creates `new McpServer({ name: 'synapse', version: '0.1.0' })`
  2. Registers ping tool (passing server and config)
  3. Registers echo tool (passing server)
  4. Returns the server instance
  - CRITICAL: All tools registered synchronously BEFORE transport connect. Per research: tools registered after connect may not appear in tools/list.
- Export `async startServer(server: McpServer): Promise<void>` that:
  1. Creates `new StdioServerTransport()`
  2. Calls `await server.connect(transport)`
  3. Logs "Synapse MCP server running on stdio" at info level via Pino (NOT console.log)

**src/index.ts — Entry point:**
- This is the main entry point: `bun run src/index.ts`
- Import `logger` and `setLogLevel` from './logger.js'
- Import `loadConfig` from './config.js'
- Import `createServer`, `startServer` from './server.js'
- Main flow:
  1. `const config = loadConfig()` — this may call process.exit(1) on validation failure
  2. `setLogLevel(config.logLevel)` — update logger level from config
  3. `logger.info({ db: config.db, ollamaUrl: config.ollamaUrl, embedModel: config.embedModel }, 'Synapse starting')`
  4. `const server = createServer(config)`
  5. `await startServer(server)`
  6. `logger.info('Synapse MCP server ready')`
- Wrap in async main() with `.catch()` that logs fatal error via `console.error` (acceptable for fatal crash) and exits
- NEVER use console.log anywhere in this file

**Import path note:** Use `.js` extensions in imports (e.g., `'./logger.js'`, `'./config.js'`). This is required for ESM module resolution in Bun with TypeScript. The `.js` extension maps to `.ts` files at runtime.

**test/smoke.test.ts — Stdout cleanliness integration test (CRITICAL for FOUND-07):**
This is the provable guarantee that stdout contains only valid JSON-RPC. Per research, stdout contamination silently kills the MCP connection with no useful error.

```typescript
import { describe, test, expect } from 'bun:test';
import { spawn } from 'bun';
```

Test 1: "server stdout is clean JSON-RPC on initialize":
- Spawn server: `spawn(['bun', 'src/index.ts', '--db', '/tmp/smoke-test-' + Date.now()], { stdout: 'pipe', stderr: 'pipe' })`
- Write an MCP initialize request to stdin:
  ```json
  {"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","clientInfo":{"name":"test","version":"1.0"},"capabilities":{}}}
  ```
  followed by a newline
- Read from stdout with a timeout (e.g., 5 seconds)
- Parse the response as JSON — if this throws, stdout is contaminated
- Verify: `parsed.jsonrpc === '2.0'` and `parsed.id === 1`
- Kill the process

Test 2: "server stderr contains log output":
- Same spawn as above
- After sending initialize, read stderr
- Verify stderr contains "Synapse" or log output (not empty)
- This confirms logs went to stderr, not stdout

Test 3: "tools/list returns ping and echo":
- Spawn server, send initialize request, read response
- Then send tools/list request:
  ```json
  {"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}
  ```
- Read response, parse JSON
- Verify response contains tools array with entries for "ping" and "echo"
- Kill process

**test/tools.test.ts — Tool handler unit tests:**

Test ping handler:

Test echo handler:

**Important test notes:**

- Handle the async read with proper timeouts to avoid hanging tests
- Clean up spawned processes in afterEach/finally blocks
- The smoke test is an integration test that proves the complete stack works end-to-end
