Implementation steps:
1. Open DB via connectDb(dbPath).
2. Query project_meta table for `last_index_at`:
   ```typescript
   const metaTable = await db.openTable("project_meta");
   const metaRows = await metaTable.query()
     .where(`project_id = '${projectId}'`).limit(1).toArray();
   const lastIndexAt = metaRows.length > 0 ? (metaRows[0].last_index_at as string | null) : null;
   ```
3. Query all code_chunks for this project:
   ```typescript
   const codeChunksTable = await db.openTable("code_chunks");
   const allRows = await codeChunksTable.query()
     .where(`project_id = '${projectId}'`)
     .select(["file_path", "language", "file_hash", "chunk_id"])
     .toArray();
   ```
4. Build per-language breakdown:
   ```typescript
   const langMap = new Map<string, { files: Set<string>; chunks: number }>();
   for (const row of allRows) {
     const lang = (row.language as string) ?? "unknown";
     if (!langMap.has(lang)) langMap.set(lang, { files: new Set(), chunks: 0 });
     const entry = langMap.get(lang)!;
     entry.files.add(row.file_path as string);
     entry.chunks++;
   }
   ```
5. Calculate total_files (distinct file_paths) and total_chunks:
   ```typescript
   const allFiles = new Set(allRows.map(r => r.file_path as string));
   const totalFiles = allFiles.size;
   const totalChunks = allRows.length;
   ```
6. Staleness check (only if projectRoot provided):
   ```typescript
   import { createHash } from "node:crypto";
   let staleFiles: number | null = null;
   if (projectRoot) {
     staleFiles = 0;
     const fileHashMap = new Map<string, string>(); // file_path -> file_hash
     for (const row of allRows) {
       if (!fileHashMap.has(row.file_path as string)) {
         fileHashMap.set(row.file_path as string, row.file_hash as string);
       }
     }
     for (const [filePath, storedHash] of fileHashMap) {
       try {
         const absPath = join(projectRoot, filePath);
         const content = await Bun.file(absPath).text();
         const currentHash = createHash("sha256").update(content).digest("hex");
         if (currentHash !== storedHash) staleFiles++;
       } catch {
         staleFiles++; // file deleted = stale
       }
     }
   }
   ```
   Import `join` from `node:path`.
   CRITICAL: When project_root is NOT provided, return `stale_files: null` (NOT 0 — that would be misleading per CONTEXT.md and Pitfall 6).

7. Build and return result (facts-only, no recommendations per CONTEXT.md).

Extend the existing get_smart_context tool to support unified document + code results. This addresses the SRCH-04 cross-table requirement and CONTEXT.md locked decisions for unified search blending.

**Changes to get-smart-context.ts:**

1. **Add `source_types` and `bias` parameters to Zod schema:**
   ```typescript
   source_types: z.enum(["documents", "code", "both"]).optional().default("both"),
   bias: z.number().min(0).max(1).optional().default(0.5),
   ```
   Per CONTEXT.md locked decisions:
   - `source_types`: optional parameter, defaults to "both". get_smart_context uses generic params only (query, limit, mode) — no code-specific filters like language/symbol_type/file_pattern.
   - `bias`: configurable bias for weighting documents vs code. 0.0 = all code, 1.0 = all documents, 0.5 = equal (default). Used to weight relevance scores before merging.

2. **New OverviewCodeItem interface (code-specific summary format):**
   ```typescript
   export interface OverviewCodeItem {
     chunk_id: string;
     file_path: string;
     symbol_name: string | null;
     symbol_type: string | null;
     language: string | null;
     summary: string;          // symbol signature + first comment via extractSnippet
     token_count: number;
     source: "code";
   }
   ```
   Per CONTEXT.md locked decision: "Code summaries in overview mode use code-specific format: symbol signature + docstring/first comment (not generic ~100-token text summary)."

3. **Extend OverviewResult type:**
   The current OverviewResult has `source: "document"` and `documents: OverviewDocument[]`. Extend to support unified results:
   ```typescript
   export interface UnifiedOverviewResult {
     mode: "overview";
   ```

4. **Extend runOverviewMode():**
   After existing document query (unchanged), add code_chunks query when source_types includes "code" or "both". Then merge candidates by relevance and fill budget from merged list.

   **Step 4a: Collect document candidates (existing logic, but defer budget filling):**
   The existing document query logic stays the same, but instead of directly filling the budget, collect all document candidates with their priority/relevance scores into a candidate list:
   ```typescript
   interface OverviewCandidate {
     type: "document" | "code";
     relevance_score: number;   // used for merged ranking
     token_count: number;
     data: OverviewDocument | OverviewCodeItem;
   }
   const candidates: OverviewCandidate[] = [];
   const bias = args.bias ?? 0.5;  // 0.0=all code, 1.0=all documents, 0.5=equal
   ```

   For each document candidate, compute a base relevance score from its priority field (normalize priority to 0-1 range, e.g. `priority != null ? priority / 10 : 0.5`), then apply bias weighting: `weightedScore = baseScore * (1 + bias) / 1`. Push to candidates array with `type: "document"`.

   **Step 4b: Collect code candidates (new):**
   ```typescript
   const sourceTypes = args.source_types ?? "both";
   let totalCodeCandidates = 0;

   if (sourceTypes === "code" || sourceTypes === "both") {
     try {
       const codeTable = await db.openTable("code_chunks");
       const codeRows = await codeTable.query()
         .where(`project_id = '${projectId}'`)
         .limit(100)
         .toArray();
       totalCodeCandidates = codeRows.length;

       for (const row of codeRows) {
         const content = row.content as string;
         const summary = extractSnippet(content, "", 100); // signature + first comment
         const summaryTokens = countTokens(summary);
         // Base relevance: use recency (newer = higher) or position-based score
         const baseScore = 0.5;  // uniform for overview (no query = no relevance ranking)
         // Apply bias weighting: lower bias favors code
         const weightedScore = baseScore * (1 + (1 - bias)) / 1;

         candidates.push({
           type: "code",
           relevance_score: weightedScore,
           token_count: summaryTokens,
           data: {
             chunk_id: row.chunk_id as string,
             file_path: row.file_path as string,
             symbol_name: (row.symbol_name as string | null) ?? null,
             symbol_type: (row.symbol_type as string | null) ?? null,
             language: (row.language as string | null) ?? null,
             summary,
             token_count: summaryTokens,
             source: "code",
           },
         });
       }
     } catch {
       // code_chunks table may not have data yet — graceful degradation
     }
   }
   ```

   When source_types is "documents", skip the code query entirely (existing behavior preserved).

   **Step 4c: Merge by relevance and fill budget (CONTEXT.md locked decision):**
   Per CONTEXT.md: "When token budget is tight, fill by pure relevance ranking regardless of source type." The bias parameter weights scores before merging, so higher-bias favors documents, lower-bias favors code.

   ```typescript
   // Sort merged candidates by weighted relevance score descending
   candidates.sort((a, b) => b.relevance_score - a.relevance_score);

   const includedDocuments: OverviewDocument[] = [];
   const includedCodeItems: OverviewCodeItem[] = [];
   let totalTokens = 0;
   let truncated = false;

   for (const candidate of candidates) {
     if (totalTokens + candidate.token_count > maxTokens) {
       truncated = true;
       continue;  // skip this candidate, try smaller ones
     }
     totalTokens += candidate.token_count;
     if (candidate.type === "document") {
       includedDocuments.push(candidate.data as OverviewDocument);
     } else {
       includedCodeItems.push(candidate.data as OverviewCodeItem);
     }
   }
   ```

   **Step 4d: Populate all CONTEXT.md-required metadata fields:**
   ```typescript
   const totalMatches = candidates.length;  // total candidates before budget filtering
   const docsReturned = includedDocuments.length;
   const codeReturned = includedCodeItems.length;
   ```

   Set `source` field to "both" if both types were queried, "document" if only documents, "code" if only code. Populate `total_matches`, `docs_returned`, `code_returned`, `truncated`, and `tokens_used` in the returned OverviewResult.

5. **Extend detailed mode for unified IDs (CONTEXT.md locked decision):**
   "Detailed mode accepts a unified ID list — tool resolves which table each ID belongs to internally."

   In runDetailedMode(), after fetching from documents table, check for any doc_ids not found in documents. Try those IDs against code_chunks table (as chunk_ids):
   ```typescript
   const foundDocIds = new Set(requestedRows.map(r => r.doc_id as string));
   const missingIds = docIds.filter(id => !foundDocIds.has(id));
   if (missingIds.length > 0 && (sourceTypes === "code" || sourceTypes === "both")) {
     // Try code_chunks table with chunk_id
     const codeTable = await db.openTable("code_chunks");
     for (const cid of missingIds) {
       const codeRows = await codeTable.query()
         .where(`chunk_id = '${cid}' AND project_id = '${projectId}'`)
         .limit(1).toArray();
       if (codeRows.length > 0) {
         const row = codeRows[0];
         const content = row.content as string;
         const tokenCount = countTokens(content);
         documents.push({
           doc_id: cid,
           title: (row.file_path as string) + " :: " + ((row.symbol_name as string) ?? "chunk"),
           category: "code",
           status: "active",
           content,
           token_count: tokenCount,
           is_requested: true,
         });
         totalTokens += tokenCount;
       }
     }
   }
   ```
   This is the "try documents first, then code_chunks" pattern from Pitfall 5.
