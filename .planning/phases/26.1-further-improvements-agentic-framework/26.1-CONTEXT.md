# Phase 26.1: Further Improvements Agentic Framework - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the Synapse agentic pipeline to establish clear role separation, critical reflection gates at every stage, and a gateway-led user interaction model. This phase restructures agent roles, introduces doer+reviewer pairs at each pipeline stage, clarifies the orchestrator as a pure dispatcher, and updates all related documentation.

**In scope:**
- Agent prompt rewrites (new agents, renamed agents, restructured responsibilities)
- Gateway protocol definition in CLAUDE.md
- Orchestrator prompt reduction to dispatcher + stage tracker
- Decision draft→review→activate flow implementation
- Config file updates (agents.toml, trust.toml)
- Documentation updates (user journey, agent descriptions, workflow docs)
- Slash command updates as needed to align with new pipeline

**Out of scope:**
- MCP server changes (no new tools, no schema changes)
- New slash commands for task tree/decision exploration (separate phase)
- Memory/context management improvements (separate phase)
- Brainstorming UX improvements (separate phase)

</domain>

<decisions>
## Implementation Decisions

### Gateway Architecture
- The main Claude session IS the gateway agent, established via CLAUDE.md protocol
- Gateway owns ALL user communication — no subagent interacts with the user directly
- Gateway runs refinement sessions (currently /synapse:refine), captures initial scope, presents findings
- Gateway spawns the Product Researcher as a research subagent during refinement
- Gateway delegates execution dispatch to the Orchestrator (subagent)
- Gateway handles failure routing — when orchestrator reports failures, gateway decides: retry, debug, or ask user

### Orchestrator Scope Reduction
- Orchestrator is a subagent, NOT the main session conductor
- Responsibilities: assign tasks to pool slots, persist stage transition documents, report completions/failures to gateway
- Does NOT make decisions, does NOT handle failure retries, does NOT interact with user
- State lives in Synapse MCP (task tree, stage documents, pool state) — orchestrator reads state, acts, reports
- If orchestrator crashes or runs out of context, a new one can be spawned — DB is the source of truth

### Agent Pipeline (doer + reviewer pairs)

**Stage 1 — Refinement (Gateway-led):**
- Agent: **Product Researcher** (renamed from Product Strategist)
- Role: Researches the user's scope, systematically pokes holes, identifies improvements and unclarities
- Does NOT make decisions — returns findings to gateway for user review
- Loop: Gateway ↔ Product Researcher ↔ User until product plan is solid
- Only then does scope move to Architecture stage

**Stage 2 — Architecture:**
- Agent: **Architect** (name kept, responsibilities narrowed)
- Role: Designs implementation architecture ONLY — files, abstraction layers, internal/external interfaces
- Does NOT decompose into executable work (that's the Planner's job)
- Drafts Tier 1-2 decisions as part of architectural proposal
- Involves user for Tier 0-1 decisions per involvement config
- Reviewer: **Architecture Auditor** (new agent)
- Role: Reviews architectural proposals for soundness — correct abstraction levels, interface contracts, no over-engineering, alignment with existing patterns
- Reviews and approves/rejects decision drafts
- Moves approved decisions from draft → active state

**Stage 3 — Planning (the big plan):**
- Agent: **Planner** (renamed from Decomposer)
- Role: Creates executable task tree from approved architecture (epics → features → components → tasks)
- Structures work, wires dependencies, attaches context refs
- Reviewer: **Plan Auditor** (new agent, replaces old Plan-Reviewer)
- Role: Goal-backward verification of the task tree — requirement coverage, scope sanity, locked-decision compliance, context alignment
- Modeled after GSD Plan-Checker's 8-dimension analysis
- Blocks execution if structure doesn't cover the goal

**Stage 4 — Task Design (detailed specs):**
- Agent: **Task Designer** (new agent, replaces task-level planning in old Decomposer)
- Role: Writes detailed task specifications with mock code, exact file paths, integration points, expected inputs/outputs
- Specs stored in Synapse MCP task descriptions (not separate files)
- Drafts Tier 2-3 decisions needed for implementation
- Reviewer: **Task Auditor** (renamed from Plan-Reviewer)
- Role: Verifies each task spec is execution-ready — mock code present, files identified, dependencies correct
- Reviews and activates Tier 2-3 decision drafts per involvement mode

**Stage 5 — Execution:**
- Agent: **Executor** (name kept)
- Role: Implements per the detailed task spec with mock code
- Reviewer: **Validator** (name kept)
- Role: Verifies implementation matches spec and decisions

### Decision Flow
- Decisions are drafted by the agent where they arise (Architect for Tier 1-2, Task Designer for Tier 2-3)
- Reviewers at each stage verify decision quality: rationale soundness, precedent conflicts, tier correctness
- Reviewers move approved decisions from draft → active
- Tier 0-1: User approval required — gateway presents to user
- Tier 2-3: Auto-approved by reviewer per involvement mode configuration
- Agents MUST NOT store decisions as active directly — always draft first, reviewer activates

### Supporting Agents (retained)
- **Orchestrator** — Pure dispatcher + stage tracker (see scope reduction above)
- **Debugger** — Root-cause analysis on execution/validation failures (unchanged)
- **Integration Checker** — Cross-task integration verification at feature/epic boundaries (unchanged)
- **Codebase Analyst** — Maintains code index (unchanged)
- **Researcher** — General research subagent for technical investigation (distinct from Product Researcher)

### Documentation Scope
- All agent prompt markdown files must be updated to reflect new roles, renamed agents, and responsibilities
- CLAUDE.md must define the gateway protocol
- agents.toml must be updated with new agent roster and tool allowlists
- trust.toml involvement matrix may need updates for new stage names
- User journey documentation must reflect the new pipeline
- Workflow documentation must describe the doer+reviewer pattern

### Claude's Discretion
- Exact prompt wording for each agent
- Tool allowlist details per agent
- Whether Architecture Auditor and Plan Auditor share a common "auditor" base prompt or are fully independent
- Whether Product Researcher and Researcher share capabilities or are fully distinct agents
- Ordering of documentation updates relative to agent changes

</decisions>

<specifics>
## Specific Ideas

- GSD Plan-Checker's 8-dimension analysis should inform the Plan Auditor's review protocol: requirement coverage, task completeness, dependency correctness, key links, scope sanity, verification derivation, context compliance
- GSD-style detailed plans with mock code in task specs — the Task Designer should produce specs detailed enough that the Executor spends tokens on implementation, not discovery
- The "too eager to decide" problem is solved by the draft→review→activate flow — no decision becomes active without reviewer approval
- The Architect should focus purely on architecture (like a real architect drawing blueprints) — it should never divide work into executable units

</specifics>
