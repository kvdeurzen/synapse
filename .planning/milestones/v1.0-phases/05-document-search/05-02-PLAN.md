  <action>
  Build semantic_search and fulltext_search as TDD features. Write failing tests, then implement.

  **semantic-search.ts — `semanticSearch(dbPath, projectId, args, config)` + `registerSemanticSearchTool(server, config)`:**

  Input schema (Zod):
  ```
  project_id: string (slug regex)
  query: string (min 1 char) — natural language search query
  category: enum(VALID_CATEGORIES) optional
  phase: string optional
  tags: string optional
  status: enum(VALID_STATUSES) optional
  priority: number 1-5 optional (minimum priority filter)
  limit: number 1-50 optional default 5
  min_score: number 0-1 optional default 0.0
  include_superseded: boolean optional default false
  ```

  Core logic:
  1. Check `getOllamaStatus()` — if not 'ok', throw `OllamaUnreachableError` (import from errors.ts). Semantic search REQUIRES embeddings.
  2. Call `buildSearchPredicate()` from search-utils to get the WHERE predicate and doc metadata map.
  3. Call `embed([query], projectId, config)` to get query vector.
  4. Run LanceDB vector search:
     ```typescript
     const rows = await docChunksTable
       .query()
       .nearestTo(queryVector)
       .distanceType("cosine")
       .where(predicate)
       .limit(limit * 2) // fetch extra for post-filter and min_score
       .toArray();
     ```
  5. For each row, call `normalizeVectorScore(row._distance)` to get relevance_score.
  6. Filter by `min_score` threshold.
  7. If `postFilterRequired` from buildSearchPredicate, filter rows against docMap.
  8. Enrich with doc metadata from docMap (or call `fetchDocMetadata` for any missing doc_ids).
  9. Call `extractSnippet(row.content, query)` for each result.
  10. Return `{ results: SearchResultItem[], total: number, search_type: 'semantic' }`.
  11. Do NOT select the vector column in results — it wastes tokens. Use `.select(["chunk_id", "doc_id", "content", "token_count", "header", "status"])` if LanceDB supports it; otherwise just omit vector from the returned shape.

  **fulltext-search.ts — `fulltextSearch(dbPath, projectId, args, config)` + `registerFulltextSearchTool(server, config)`:**

  Same input schema as semantic_search (minus the Ollama dependency).

  Core logic:
  1. No Ollama check needed — FTS works without embeddings.
  2. Call `buildSearchPredicate()` for WHERE predicate and doc metadata.
  3. Run LanceDB FTS:
     ```typescript
     const rows = await docChunksTable
       .query()
       .fullTextSearch(query)
       .where(predicate)
       .limit(limit * 2)
       .toArray();
     ```
  4. For each row, call `normalizeFtsScore(row._score)` to get relevance_score.
  5. Filter by `min_score`, enrich with metadata, extract snippets.
  6. Return `{ results: SearchResultItem[], total: number, search_type: 'fulltext' }`.

  **MCP registration for both:**
  Follow the two-export pattern from query-documents.ts. Use `createToolLogger()`. Wrap in try/catch returning `ToolResult`.

  **Test approach:**
  - Tests use temp LanceDB database with `initProject()` to create tables + FTS index
  - Insert doc_chunks directly (with embeddings for semantic, without for FTS) using `insertBatch`
  - For semantic_search tests that need actual embeddings: mock the embed function or skip vector tests and test only the Ollama-unreachable path (throw check)
  - For FTS tests: insert chunks with known content, verify search returns them ranked by relevance
  - Test metadata filtering: insert chunks for different categories, verify category filter narrows results
  - Test min_score filtering: verify low-scoring results are excluded
  - Test include_superseded flag

  **What to AVOID:**
  - Do NOT use `.search(query)` shorthand for vector — use `.query().nearestTo(vector).distanceType("cosine")`
  - Do NOT forget `.distanceType("cosine")` — default is L2 which has unbounded distance range
  - Do NOT return the vector column in results

  Build hybrid_search as a TDD feature.

  **hybrid-search.ts — `hybridSearch(dbPath, projectId, args, config)` + `registerHybridSearchTool(server, config)`:**

  Same input schema as semantic_search.

  Core logic:
  1. Check `getOllamaStatus()` — if not 'ok':
     - Log warning: "Ollama unreachable — hybrid_search falling back to FTS-only"
     - Call `fulltextSearch(dbPath, projectId, args, config)` from fulltext-search.ts as fallback
     - Add `fallback: true, fallback_reason: "Ollama unreachable"` to result
     - Return the FTS results with `search_type: 'hybrid_fts_fallback'`
  2. If Ollama is ok:
     - Call `buildSearchPredicate()` for WHERE predicate
     - Call `embed([query], projectId, config)` for query vector
     - Create reranker: `const reranker = await rerankers.RRFReranker.create(60)`
     - Run hybrid search:
       ```typescript
       import { rerankers } from "@lancedb/lancedb";

       const rows = await docChunksTable
         .query()
         .nearestTo(queryVector)
         .distanceType("cosine")
         .fullTextSearch(query)
         .where(predicate)
         .rerank(reranker)
         .limit(limit)
         .toArray();
       ```
     - The RRF reranker output score field: check `rows[0]` for `_relevance_score` or `_score` at runtime. Log the actual field names if uncertain. Use whichever field is present, defaulting to position-based scoring (1/(rank+60)) if no score field found.
     - Normalize the RRF score to [0,1] if not already in that range.
  3. Filter by `min_score`, enrich with metadata, extract snippets.
  4. Return `{ results: SearchResultItem[], total: number, search_type: 'hybrid' }`.

  **Test approach:**
  - Test the Ollama-unreachable fallback path: mock `getOllamaStatus` to return 'unreachable', verify `hybridSearch` returns FTS results with fallback flag
  - Test the full hybrid path if Ollama is available (can skip in CI with `test.skipIf`)
  - Test that RRFReranker is created with k=60
  - Test metadata filters and min_score work in hybrid mode

  **What to AVOID:**
  - Do NOT call `rerankHybrid()` directly — use the fluent `.rerank(reranker)` chain (Pitfall 5 from RESEARCH.md)
  - Do NOT implement manual RRF in JavaScript — use the native LanceDB reranker
  - Do NOT forget the fallback path — this is a locked decision from CONTEXT.md
