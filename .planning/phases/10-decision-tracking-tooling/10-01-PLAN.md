    - Lifecycle transitions: active->superseded (via supersedes param), active->revoked (via status param on new store or future update tool), superseded->revoked allowed
    - Revocation: store_decision called with status "revoked" on an existing decision_id is not a re-version -- this is handled by providing a "revoke" action. However, per CONTEXT.md decisions, the simpler pattern is: new store_decision with `supersedes: old_id` sets old to superseded. Revocation can be done by directly updating status. For Phase 10, store_decision only stores new decisions and optionally supersedes old ones.

    2. Create `src/tools/store-decision.ts` following the store-document.ts pattern exactly:
       - Import: McpServer, ulid, z, insertBatch, connectDb, DecisionRowSchema, createToolLogger, logActivity, embed, SynapseConfig, ToolResult, decision constants
       - `StoreDecisionInputSchema` Zod object:
         - project_id: string (slug regex)
         - subject: string.min(1) -- what is being decided
         - choice: string.min(1) -- what was chosen
         - context: string.min(1) -- problem statement, constraints, alternatives
         - rationale: string.min(1) -- justification for the choice
         - tier: z.number().int().min(0).max(3)
         - decision_type: z.enum(VALID_DECISION_TYPES)
         - tags: string.optional().default("") -- pipe-separated
         - phase: string.optional()
         - actor: string.optional().default("agent")
         - supersedes: string.optional() -- decision_id of the decision being superseded
         - status: z.enum(VALID_DECISION_STATUSES).optional().default("active")
       - `StoreDecisionResult` interface: { decision_id, tier, tier_name, status, created_at, has_precedent }
       - `storeDecision(dbPath, projectId, args, config)` core function:
         a. Validate input via StoreDecisionInputSchema.parse(args)
         b. Connect to DB
         c. Generate decision_id via ulid()
         d. If `supersedes` provided: query decisions table for that id, verify it exists and is active/superseded, update its status to "superseded" and set superseded_by to new decision_id
         e. Embed the rationale text: `await embed([args.rationale], projectId, config)` -- fail fast if Ollama down (same as store_document pattern)
         f. Quick precedent check: do a vector search on decisions table for similarity >= 0.85 with decision_type pre-filter, limit 1 -- set has_precedent accordingly. If table is empty or vector search fails, has_precedent = false
         g. Insert decision row with all fields, vector from step (e)
         h. Log activity: logActivity(db, projectId, "store_decision", decision_id, "decision", { tier, decision_type })
         i. Return StoreDecisionResult
       - `registerStoreDecisionTool(server, config)` following exact same MCP registration pattern as store-document.ts

    3. Wire into `src/server.ts`:
       - Add import: `import { registerStoreDecisionTool } from "./tools/store-decision.js";`
       - Add registration call after the last existing registerXTool call (after registerGetIndexStatusTool)
       - Increment toolCount

    IMPORTANT: The has_precedent flag in the return value is a convenience signal. It does a quick vector search after storing. If the decisions table is empty (first decision) or the search finds nothing above 0.85, return false. This is NOT the full check_precedent tool -- that comes in Plan 02.
