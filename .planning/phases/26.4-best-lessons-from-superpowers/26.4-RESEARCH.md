## Architecture Patterns

### Pattern 1: Fail-Closed PostToolUse Hook

The existing `tier-gate.js` (PreToolUse on `store_decision`) and `tool-allowlist.js` (PreToolUse on `mcp__synapse__*`) establish the hook pattern. The new `output-contract-gate.js` is a PostToolUse hook on `update_task` calls where status is set to "done".

**Hook structure (from verified existing pattern):**

```javascript
// Source: packages/framework/hooks/tier-gate.js (verified)
import fs from "node:fs";
import { parse } from "smol-toml";
import { resolveConfig } from "./lib/resolve-config.js";

// Input arrives via stdin as JSON
let input = "";
process.stdin.setEncoding("utf8");
process.stdin.on("data", (chunk) => (input += chunk));
process.stdin.on("end", () => {
  try {
    let data;
    try { data = JSON.parse(input); }
    catch { process.stdout.write(denyOutput("...")); process.exit(0); }

    const toolName = data.tool_name || "";
    // Gate only targeted tool — pass all others silently
    if (toolName !== "mcp__synapse__update_task") { process.exit(0); }

    const toolInput = data.tool_input || {};
    // ... check logic ...

    // Load config — fail-closed on missing or unparseable
    const contractsPath = resolveConfig("output-contracts.toml");
    if (!contractsPath) {
      process.stdout.write(denyOutput("DENIED: output-contracts.toml not found."));
      process.exit(0);
    }
    const contracts = parse(fs.readFileSync(contractsPath, "utf8"));

    // For PostToolUse, output is JSON to stdout:
    process.stdout.write(JSON.stringify({
      hookSpecificOutput: {
        hookEventName: "PostToolUse",
        permissionDecision: "deny",
        permissionDecisionReason: "reason"
      }
    }));
  } catch (_e) {
    process.stdout.write(denyOutput("Unexpected error — fail-closed."));
    process.exit(0);
  }
});
```

**PostToolUse vs PreToolUse:** PreToolUse intercepts before tool execution (deny prevents the call). PostToolUse intercepts after. For output contracts, PostToolUse is correct because we need to inspect what documents were produced (checking `output_doc_ids` field on `update_task` calls).

### Pattern 5: Validator Independence Resequencing

Current validator sequence (verified from validator.md):
1. Load task spec (get_task_tree)
2. Fetch context (get_smart_context)
3. Query decisions
4. TDD Verification Protocol (Steps A-D)
5. Read executor output doc (Step D)

New sequence (from CONTEXT.md):
1. Load spec
2. Run tests / verify independently (Steps A-C)
3. Read code directly
4. Form independent assessment
5. THEN read executor output doc for discrepancies (Step D)

The key behavioral change: validator forms its own verdict before consulting what the executor claimed. The validator must NOT read the executor's implementation summary before running its own verification. This closes the "Do Not Trust the Report" gap.

### Anti-Patterns to Avoid

- **Self-generated rationalization tables:** If the implementer writes rationalization tables from scratch, they will encode rationalizations the model already knows to avoid. The tables must come from external adversarial testing (Superpowers catalogs, published agentic frameworks). This is a LOCKED DECISION.
- **Hook that blocks on READ errors:** If output-contracts.toml is missing, the hook must deny as fail-closed (same as tier-gate.js). A hook that silently allows when config is missing provides no enforcement.
- **Validator reading executor report before forming own verdict:** The independence rule is the entire behavioral value of Practice #3. If the validator reads the executor's self-report first, it will anchor on it.
- **Centralizing anti-rationalization in _synapse-protocol.md:** The decision is per-agent inline tables. Centralized tables are not visible to the agent when it's working the specific constraint. Inline tables in the agent's own file are immediately visible.
- **PostToolUse hook on update_task only checking status field:** The hook must also check whether `output_doc_ids` is populated (verifying documents were actually stored). An agent can set status="done" without having stored output documents.

## Common Pitfalls

### Pitfall 1: PostToolUse Hook for update_task Fires on Every Status Change

**What goes wrong:** `update_task` is called with `status: "done"` by orchestrator, executor, and validator. The hook must only gate on calls where completion output is expected, not on all status updates.

**Why it happens:** The hook sees all `update_task` calls. Orchestrator calls `update_task` on parent tasks after children complete. Validator calls `update_task` on tasks it passes. Each agent has different output expectations.

**How to avoid:** Check the `actor` field in the tool input. Only enforce on specific actors where output documents are required (executor, validator, architect, planner, etc.). The `output-contracts.toml` config maps actor names to their required docs.

**Warning signs:** If the hook fires on orchestrator's parent-task updates, it will block legitimate pipeline progression.

### Pitfall 2: Reviewer Agents Don't Currently Have Output Documents on Pass

**What goes wrong:** Looking at `validator.md` (verified), validator only stores a document on FAIL. On PASS, it calls `update_task(status: "done")` directly. This makes output contract enforcement for validators asymmetric.

**Why it happens:** Current design stores findings only when they are negative. This is intentional for token efficiency.

**How to avoid:** The output contract hook for reviewer agents should gate differently — on FAIL it checks for `validator-validation-findings-{task_id}`, on PASS it only verifies spec and output_doc_ids are populated on the task. Alternatively, the hook only enforces for doer agents (executor, architect, planner, task-designer) where output documents are always required. Reviewer enforcement can be lighter. The exact approach is Claude's Discretion.

### Pitfall 3: Escalation Status Conflicts with Existing executor.md Status

**What goes wrong:** Phase 26.3-tdd already added DONE/DONE_WITH_CONCERNS/BLOCKED/NEEDS_CONTEXT to the executor. Phase 26.4 generalizes these to all doer agents. There is no conflict on the executor itself — just needs a consistency check that the wording matches.

**Why it happens:** Different phases adding overlapping behavioral specs.

**How to avoid:** When adding escalation statuses to non-executor doer agents, verify the executor's existing status section wording first and replicate the exact same format. The status values are identical for all doer agents.

### Pitfall 4: Anti-Rationalization Research Bottleneck

**What goes wrong:** The CONTEXT.md makes clear that rationalization tables MUST be sourced from external frameworks. If the planner creates tasks that say "add anti-rationalization table to agent X" without providing source material, the executor will be forced to self-generate (which defeats the purpose) or stall.

**Why it happens:** The research phase has the context; the executor doesn't.

**How to avoid:** The RESEARCH.md should pre-catalog the best external sources for each agent type. Tasks for adding rationalization tables should include the source material or explicit references to where to pull from. The planner must include source citations in task specs.

### Pitfall 5: Brainstorming SKILL.md Rewrite Loses Existing Quality Criteria

**What goes wrong:** A "full rewrite" instruction can accidentally drop valuable existing content from `brainstorming/SKILL.md`. The current skill has good content (one question at a time, YAGNI, present options neutrally).

**Why it happens:** "Full rewrite" is interpreted as "start from scratch."

**How to avoid:** The `brainstorm-skill-redesign.md` document explicitly lists what is preserved. The task spec must reference both the brainstorm redesign doc AND the existing SKILL.md and require the implementer to carry forward all `## Preserved From Current Skill` items.

### Pitfall 6: output-contracts.toml Becoming Stale After code-quality-reviewer Is Added

**What goes wrong:** If output-contracts.toml is written first (correctly) and code-quality-reviewer is added later, the new agent's contract needs to be added to the TOML too.

**Why it happens:** Sequential implementation across tasks.

**How to avoid:** Plan the output-contracts.toml to be written last (after all agents are defined) OR plan the code-quality-reviewer task to also update output-contracts.toml as part of its deliverable.

## Open Questions

1. **PostToolUse hook: what exact tool call to intercept?**
   - What we know: output-contract-gate fires when an agent "completes" — this maps to `update_task` with status: "done", or possibly to `store_document` calls where the produces slug should appear.
   - What's unclear: intercepting `update_task` is sufficient for blocking false completions, but does not verify the document was stored BEFORE the status update. Intercepting `store_document` instead would verify document production but wouldn't block the status update directly.
   - Recommendation: Intercept `update_task(status: "done")`. Check that `output_doc_ids` field is populated (executor already calls `update_task(output_doc_ids: [...])` before `update_task(status: "done")`). This is the enforcement point. Claude's Discretion for exact logic.

2. **code-quality-reviewer: how does it produce NEEDS_REVISION routing?**
   - What we know: NEEDS_REVISION auto-routes back to executor with reviewer feedback. The orchestrator must read the code-quality-reviewer's output document to detect NEEDS_REVISION status.
   - What's unclear: Whether code-quality-reviewer spawning is in the existing `synapse-orchestrator.md` pool dispatch flow or requires a new dispatch step.
   - Recommendation: Add code-quality-reviewer dispatch in the Wave Execution Protocol section of synapse-orchestrator.md, after the validator PASS step. The pool manager step 1b (check executor status before validator routing) should gain a parallel step: "after validator PASS, dispatch code-quality-reviewer before proceeding to next wave."

3. **Pressure testing protocol document location**
   - What we know: CONTEXT.md says ".planning/ or packages/framework/docs/". The .planning/ directory is for planning artifacts; packages/framework/docs/ would ship with the framework.
   - Recommendation: `packages/framework/docs/skill-pressure-testing.md` since it's a methodology users of the framework benefit from. Create the `docs/` directory if it doesn't exist.
