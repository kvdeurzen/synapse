## Summary

This phase is a pure structural migration: two independent git repos (`synapse` at `/home/kanter/code/synapse` and `synapse-framework` at `/home/kanter/code/synapse-framework`) become a single Bun workspace monorepo at the existing `synapse` repo root. No new features, no logic changes.

The migration has three distinct concerns: (1) the git history merge via `git subtree`, (2) the Bun workspace scaffold (root package.json, shared tsconfig/biome, aggregate scripts), and (3) path updates for all cross-repo references that currently use filesystem-relative assumptions. Both repos have healthy test suites (612 server tests, 65 framework tests) that serve as the post-migration green light.

The biggest non-obvious risk is the tree-sitter `postinstall` script, which uses `__dirname`-relative paths to locate `node_modules/tree-sitter/` — in a workspace, this package may be hoisted to the root `node_modules/`, causing the script to silently skip the binary copy. This must be verified after `bun install` in the workspace context.

**Primary recommendation:** Execute in this order: git scaffold (git subtree add + server move), then workspace config files, then path updates, then `bun install` + test both packages, then clean up the standalone framework repo.

### Pattern 1: Root package.json with Bun Workspaces
**What:** Root declares workspaces, hoist shared deps, provide aggregate scripts.
**When to use:** Always for Bun monorepos.

```json
// Source: https://github.com/oven-sh/bun/blob/main/docs/pm/workspaces.mdx
{
  "name": "synapse",
  "private": true,
  "workspaces": ["packages/*"],
  "dependencies": {
    "zod": "^4.0.0",
    "smol-toml": "latest"
  },
  "scripts": {
    "test": "bun run --filter '*' test",
    "test:server": "bun run --filter '@synapse/server' test",
    "test:framework": "bun run --filter '@synapse/framework' test",
    "lint": "bun run --filter '*' lint"
  },
  "devDependencies": {
    "@biomejs/biome": "2.4.4",
    "@types/node": "latest",
    "bun-types": "latest",
    "typescript": "latest"
  }
}
```

### Pattern 2: Workspace Package package.json (peerDependencies)
**What:** Each workspace lists shared deps as `peerDependencies`, omits them from `dependencies`.

```json
// packages/server/package.json
{
  "name": "@synapse/server",
  "version": "0.1.0",
  "type": "module",
  "peerDependencies": {
    "zod": "^4.0.0",
    "smol-toml": "latest"
  },
  "dependencies": {
    // server-specific only: @lancedb/lancedb, tree-sitter, etc.
  }
}
```

### Pattern 3: Shared tsconfig.base.json
**What:** Root tsconfig holds common options; each package extends it and overrides package-specific fields only.

```json
// tsconfig.base.json at root
{
  "compilerOptions": {
    "target": "ESNext",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "types": ["bun-types"]
  }
}
```

```json
// packages/server/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src",
    "declaration": true,
    "sourceMap": true
  },
  "include": ["src/**/*.ts"],
  "exclude": ["node_modules", "dist", "test"]
}
```

Note: The server currently has `"noUncheckedIndexedAccess": true` and `"exactOptionalPropertyTypes": true` while the framework does not. These should go in the base config; both repos already build cleanly at those strictness levels.

### Pattern 4: git subtree add from local path
**What:** Merge a local repo's history into a subdirectory of another repo.

```bash
# Step 1: Add the local framework repo as a temporary remote
git -C /home/kanter/code/synapse remote add framework /home/kanter/code/synapse-framework

# Step 2: Fetch the framework history
git -C /home/kanter/code/synapse fetch framework

# Step 3: Move server files into packages/server/ (git mv to preserve history)
git -C /home/kanter/code/synapse mv src scripts test biome.json bun.lock package.json tsconfig.json packages/server/
# ... (all server files)

# Step 4: Subtree-add framework into packages/framework/ (WITHOUT --squash to preserve full history)
git -C /home/kanter/code/synapse subtree add --prefix=packages/framework framework/main

# Step 5: Remove the temporary remote
git -C /home/kanter/code/synapse remote remove framework
```

Note: The CONTEXT.md says "one atomic commit" — the `subtree add` creates its own merge commit. The server files move and root scaffold must be committed before or staged carefully. In practice this becomes 2-3 commits (scaffold + server move, then subtree add merge commit). "Atomic" in intent means no broken intermediate state, not necessarily literally one git commit.

### Pattern 5: bun run --filter for aggregate scripts
**What:** Run a named script across all (or specific) workspaces.

```bash
# Source: https://github.com/oven-sh/bun/blob/main/docs/pm/filter.mdx
bun run --filter '*' test          # all workspaces
bun run --filter '@synapse/server' test    # server only
bun run --filter '@synapse/framework' test # framework only
```

### Anti-Patterns to Avoid
- **Duplicating biome.json in each package:** Wastes config, allows drift. Use single root config with `"include"` patterns if needed, or just rely on root config being found by traversal.
- **Duplicating devDependencies in each workspace:** biome, typescript, bun-types belong at root only.
- **Preserving old separate `bun.lock` files:** After `bun install` at root, a single monorepo lockfile is generated. The old per-package locks are replaced.
- **Using `--squash` on git subtree add:** This would discard the framework's 16 commits. Since CONTEXT.md requires history preservation, omit `--squash`.
- **Cross-package TypeScript imports before verifying coupling:** The framework references the server only via MCP stdio protocol — not via direct TypeScript imports. Do NOT add `"@synapse/server": "workspace:*"` to framework dependencies unless explicitly needed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Workspace script running | Shell loop with `cd packages/X && bun test` | `bun run --filter '*' test` | Built-in to Bun; handles dependency ordering, parallel execution |
| Cross-repo history merge | Copy-paste files and `git add` | `git subtree add` | Preserves all 16 framework commits in the merged repo |
| Shared TypeScript config | Copy-paste tsconfig in each package | `"extends": "../../tsconfig.base.json"` | Single source of truth for compiler options |

**Key insight:** Bun workspaces + `--filter` handles the aggregate script problem natively. No custom shell scripts needed.

## Common Pitfalls

### Pitfall 1: tree-sitter postinstall path resolution in workspace context
**What goes wrong:** The `setup-tree-sitter.js` postinstall script uses `__dirname` to compute `projectRoot = join(__dirname, "..")` which resolves to `packages/server/`. It then looks for `packages/server/node_modules/tree-sitter/`. But in a Bun workspace, `tree-sitter` is a server-specific dependency (not shared) and should NOT be hoisted — however Bun may still hoist it to the root `node_modules/` under certain conditions.

If tree-sitter is hoisted, `packages/server/node_modules/tree-sitter/build/Release/...` won't exist, the script exits early with "No compiled binary found — skipping", and the server fails at runtime when importing `tree-sitter`.

**Why it happens:** Bun workspaces hoist packages to the root `node_modules/` when the version is compatible across all workspaces. Since only `@synapse/server` depends on `tree-sitter`, it should stay in `packages/server/node_modules/` — but this must be verified empirically after `bun install`.

**How to avoid:** After `bun install` in the monorepo, run:
```bash
ls packages/server/node_modules/tree-sitter/build/Release/ 2>/dev/null || echo "HOISTED — check root node_modules"
ls node_modules/tree-sitter/build/Release/ 2>/dev/null || echo "not in root"
```
If hoisted, update `setup-tree-sitter.js` to walk up to find `tree-sitter` (e.g., check both `packages/server/node_modules` and `../../node_modules`), or add `bundleDependencies: ["tree-sitter"]` to the server package.json to prevent hoisting.

**Warning signs:** Server tests fail with "Cannot find module tree-sitter" or "no native binding found" after `bun install`.

### Pitfall 2: Integration test helper hard-coded path to server
**What goes wrong:** `packages/framework/test/helpers/synapse-client.ts` currently has:
```typescript
const serverPath = join(import.meta.dir, "..", "..", "..", "project_mcp", "src", "index.ts");
```
This path was written when framework was at `/home/kanter/code/synapse-framework/` and server was at `/home/kanter/code/project_mcp/`. After migration, from `packages/framework/test/helpers/`, the server is at `../../server/src/index.ts` (relative) — a completely different path.

**How to avoid:** Update `synapse-client.ts` and `test/integration/startup.test.ts` to:
```typescript
const serverPath = join(import.meta.dir, "..", "..", "..", "..", "server", "src", "index.ts");
// packages/framework/test/helpers/../../../.. = monorepo root
// Wait: packages/framework/test/helpers = depth 4 from root
// ../../../.. goes to root, then server/src/index.ts
// Actually: ../../../server/src/index.ts (3 levels up = packages/, then server/)
```
Verify: `packages/framework/test/helpers/` → `../` = `packages/framework/test/` → `../` = `packages/framework/` → `../` = `packages/` → `server/src/index.ts`.
Correct path: `join(import.meta.dir, "..", "..", "..", "server", "src", "index.ts")`

**Warning signs:** Integration tests skip silently (they use `describe.skipIf(!serverExists)`) — check that `serverExists` is actually true.

### Pitfall 3: Hook paths in settings.template.json
**What goes wrong:** Current `settings.template.json` has:
```json
"command": "node hooks/synapse-startup.js"
```
This is relative to wherever Claude Code runs (typically the project root). After migration, the hooks are at `packages/framework/hooks/`. A user who copies `settings.template.json` to root-level `.claude/settings.json` and uses relative paths will get "command not found" for the hooks.

**How to avoid:** Update `settings.template.json` hook commands to use `packages/framework/` prefix:
```json
"command": "node packages/framework/hooks/synapse-startup.js"
```
This path is correct when Claude runs from the monorepo root (which is the expected usage per CONTEXT.md).

### Pitfall 4: biome.json version mismatch between the two repos
**What goes wrong:** `synapse` uses `biome.json` schema version `2.4.4` with `"vcs"`, `"assist"`, and tab-based formatting. `synapse-framework` uses schema version `1.9.4` with no `vcs` block and tab indentation. When consolidating, these have different feature sets.

**How to avoid:** Use the framework's biome version `2.4.4` (the newer one, already installed in `synapse`) as the base for the consolidated root `biome.json`. Verify both packages' source files still pass `bunx biome check` after applying the root config. The main difference is `indentStyle: "space"` (synapse) vs default tab (framework) — pick one consistently (space is more explicit, align with the synapse setting).

### Pitfall 5: tsconfig option gap between repos
**What goes wrong:** `synapse/tsconfig.json` has `noUncheckedIndexedAccess: true` and `exactOptionalPropertyTypes: true` which the framework tsconfig lacks. If both extend a shared base that includes these options, framework code that previously compiled without them may now fail type checking.

**How to avoid:** Run `bun run --filter '@synapse/framework' tsc --noEmit` after updating the framework tsconfig to extend the base. In practice, the framework currently has `"noEmit": true` and `"allowImportingTsExtensions": true` which the server lacks — these should stay in the framework-specific tsconfig, not the base. The strictness options (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`) should be in the base only if the framework code passes type checks with them enabled. Test before committing.

### Pitfall 6: bun.lock conflict
**What goes wrong:** Each repo has its own `bun.lock`. After moving server files to `packages/server/`, the root no longer has the server's lock data. Running `bun install` at root generates a new unified `bun.lock` — this is correct behavior, not a problem — but the old `packages/server/bun.lock` file (if it exists) will confuse Bun.

**How to avoid:** Do NOT copy the old `bun.lock` files into `packages/server/` or `packages/framework/`. Remove any workspace-level `bun.lock` files before running `bun install` at root. The root `bun.lock` is authoritative.
