**Part B: Create 11 new skills** -- each in its own directory under `packages/framework/skills/`.

All new skills use the 5-section format. All have YAML frontmatter with `name`, `description`, `disable-model-invocation: true`, `user-invocable: false`. Target 60-100 lines each.

**New language skills (5):**

1. **rust/SKILL.md** -- Adapt from Microsoft Pragmatic Rust Guidelines (agent-optimized). Key conventions: ownership model, `Result<T, E>` error handling, `?` operator, traits over inheritance, `clippy` for linting, `rustfmt` for formatting, no `unwrap()` in production code. Commands: `cargo build`, `cargo test`, `cargo clippy -- -D warnings`, `cargo fmt`, `cargo check`, `cargo doc --open`.

2. **cargo-test/SKILL.md** -- From Rust Book Chapter 11. Key conventions: `#[cfg(test)]` module in same file, `tests/` for integration tests, `assert!`/`assert_eq!`/`assert_ne!` macros, `Result<(), Box<dyn Error>>` return type for tests using `?`, `#[should_panic]` for expected failures. Commands: `cargo test`, `cargo test -- --nocapture`, `cargo test <test_name>`, `cargo test --test <integration_file>`.

3. **go/SKILL.md** -- From Effective Go + Google Style Guide. Key conventions: `gofmt`/`goimports` always, error wrapping with `fmt.Errorf("context: %w", err)`, interfaces for behavior not data, `context.Context` as first parameter for cancellable operations, goroutine lifecycle management. Commands: `go build ./...`, `go test ./...`, `go vet ./...`, `golangci-lint run`, `goimports -w .`, `go mod tidy`.

4. **go-test/SKILL.md** -- From Go testing conventions. Key conventions: table-driven tests with `[]struct{name, input, want}`, `t.Run()` for subtests, `t.Parallel()` for independent tests, `testdata/` directory for fixtures, no `init()` in test files. Commands: `go test ./...`, `go test -run TestName`, `go test -v`, `go test -race`, `go test -cover`.

5. **pytest/SKILL.md** -- From fastmcp.me python-testing-patterns + PEP conventions. Use the full example from 20-RESEARCH.md Code Examples section as the baseline. Key conventions: `test_<behavior>_<condition>` naming, AAA structure, `@pytest.fixture`, `@pytest.mark.parametrize`, `conftest.py`, `unittest.mock.patch`. Commands: `pytest`, `pytest -x`, `pytest tests/unit/test_auth.py`, `pytest -m integration`, `pytest --cov=src`.

**New generic skills (6):**

6. **testing-strategy/SKILL.md** -- Language-agnostic testing pyramid concepts. Conventions: unit tests for pure logic (fast, isolated), integration tests for boundaries (DB, API, filesystem), e2e tests for critical user paths only, mock external deps in unit tests, test behavior not implementation. Commands section: reference to project's language-specific testing skill for actual commands.

7. **architecture-design/SKILL.md** -- ADR format, separation of concerns, dependency direction (depend inward), interface-first design, defer decisions until last responsible moment. Conventions on when to use shared vs. local modules. Commands: none (cognitive skill -- note this in Commands section).

8. **security/SKILL.md** -- Input validation at boundaries, parameterized queries (never string concatenation for SQL), secret management (env vars, never hardcoded), least-privilege principle, dependency auditing. Commands: `npm audit`, `cargo audit`, `pip-audit`, `trivy fs .` (list all tools, agent picks the relevant one).

9. **brainstorming/SKILL.md** -- Structured technical problem-solving. Conventions: enumerate 3+ options before committing, explicit tradeoff analysis (pros/cons/risks), DECIDED/OPEN/EMERGING state tracking, time-box exploration, avoid premature optimization. Commands: none (cognitive skill).

10. **defining-requirements/SKILL.md** -- Requirements quality. Conventions: testable (can write a pass/fail check), unambiguous (one interpretation), prioritized (must/should/could/won't), acceptance criteria in Given-When-Then format, avoid gold-plating (build what's needed). Commands: none (cognitive skill).

11. **documentation/SKILL.md** -- When to document (decisions, APIs, non-obvious behavior, setup steps), what NOT to document (obvious code, self-documenting names). README structure (what, why, how, examples). Inline comment conventions (explain WHY, not WHAT). Commands: doc generators listed neutrally (typedoc, rustdoc, pdoc, godoc).
