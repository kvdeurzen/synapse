# Phase 26.6: Auto Re-index - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Automatically keep the code index fresh during RPEV execution so agents always search current code. Also fix the Bun API incompatibility bug that prevents indexing from working at all under the `npx tsx` runtime. Additionally, integrate indexing into `/synapse:init` so project setup is a single command.

</domain>

<decisions>
## Implementation Decisions

### Trigger Timing
- Re-index after each wave of executor tasks completes, before validators are dispatched
- Validators and downstream agents always search fresh code
- Keep the existing feature-completion re-index (synapse-orchestrator.md line 172) as belt-and-suspenders
- RPEV pipeline only — no hooks on manual git commits; users run `/synapse:map` manually when needed
- `/synapse:init` runs full `index_codebase` (with embeddings) at the end of initialization, so the user doesn't need a separate `/synapse:map` step
- `/synapse:map` remains as a standalone command for manual re-indexing after individual changes

### Wiring Mechanism
- Orchestrator calls `index_codebase` directly between wave completion and validator dispatch — no codebase-analyst agent spawn
- Explicit step added to synapse-orchestrator.md wave execution section: "After wave executors complete, call index_codebase before dispatching validators"
- Audit log captures re-index calls automatically (MCP layer) — no extra logging to stage documents
- `/synapse:init` (init.md) updated to call `index_codebase` after `init_project` succeeds

### Index Scope
- Always full project scan (no include_patterns) — incremental hashing already skips unchanged files
- Default gitignore-based exclusions — no extra exclude_patterns needed
- Same approach for both wave-level re-index and /synapse:init index

### Failure Handling
- Wave-level re-index failure: warn and continue — validators run with stale index rather than blocking the pipeline
- `/synapse:init` re-index failure: warn but complete init — project is usable without index; user told to run `/synapse:map` when Ollama is available
- No consecutive failure tracking — each wave independently tries to re-index
- Re-index is non-blocking: pipeline never halts for indexing failures

### Bun API Bugfix
- Server source code uses `Bun.Glob` and `Bun.file().text()` but runs under `npx tsx` (Node.js runtime)
- Causes `ReferenceError: Bun is not defined` when calling `index_codebase` or `get_index_status`
- Fix: replace `Bun.Glob` with `glob` + `minimatch` packages in scanner.ts
- Fix: replace `Bun.file().text()` with `readFile()` from `node:fs/promises` in index-codebase.ts and get-index-status.ts
- Add `glob` and `minimatch` as server package dependencies
- Detailed fix log: `.planning/issues/260313_bun.md`

### Claude's Discretion
- Exact wording of the orchestrator prompt addition for wave-level re-indexing
- How to surface the re-index warning message when it fails (log format, verbosity)
- Whether to add a brief status message ("Re-indexing...") before the index_codebase call in the orchestrator

</decisions>
