    **Step 1: Create output-contracts.toml**

    Create `packages/framework/config/output-contracts.toml` mapping each pipeline agent to its required output documents. This file replaces the provides vocabulary in _synapse-protocol.md as the single source of truth (per locked decision). Use the provides vocabulary slugs from _synapse-protocol.md as the canonical reference.

    Schema structure (each agent entry):
    ```toml
    [agents.{agent-name}]
    required_docs = [
      { doc_id_pattern = "{agent}-{type}-{task_id}", provides = "{slug}" }
    ]
    ```

    Map all 12 producer agents from the provides vocabulary table. Key entries:
    - executor: `executor-implementation-{task_id}` provides `implementation`
    - architect: `architect-architecture-{task_id}` provides `architecture`
    - planner: `planner-plan-{task_id}` provides `plan`
    - task-designer: writes to task.spec field (no doc required — skip or mark as field-based)
    - test-designer: `test-designer-test-contract-{task_id}` provides `test-contract`
    - validator: only stores doc on FAIL; on PASS mark `required_on_fail_only = true` or similar to handle asymmetry (see Research Pitfall 2)
    - All auditor agents: `{agent}-audit-findings-{task_id}` provides `audit-findings`
    - integration-checker: `integration-checker-integration-report-{task_id}` provides `integration-report`
    - researcher/product-researcher: `{agent}-research-findings-{task_id}` provides `research-findings`
    - debugger: `debugger-debug-diagnosis-{task_id}` provides `debug-diagnosis`
    - codebase-analyst: `codebase-analyst-code-analysis-{task_id}` provides `code-analysis`

    Include a comment header explaining the file's purpose and that it is the single source of truth for output contracts.

    **Step 2: Create output-contract-gate.js hook**

    Create `packages/framework/hooks/output-contract-gate.js` as a PostToolUse hook following the exact pattern from tier-gate.js:
    - Import fs, smol-toml parse, resolveConfig from ./lib/resolve-config.js
    - Read stdin JSON, check tool_name
    - Only gate on `mcp__synapse__update_task` calls — pass all others with process.exit(0)
    - For update_task: check if `tool_input.status === "done"` — pass all other status values silently
    - Extract `actor` from tool_input — if actor is not in output-contracts.toml agents section, allow silently (orchestrator updates parent tasks, no output expected)
    - Load output-contracts.toml via resolveConfig — fail-closed (deny) if missing or unparseable
    - Check the agent's required_docs: for each required doc, verify that `tool_input.output_doc_ids` contains a matching doc_id (pattern matching with `{task_id}` replaced by the actual task_id from tool_input)
    - Handle the validator asymmetry: if agent has `required_on_fail_only` flag, check tool_input.status logic
    - If any required docs are missing: output deny JSON with `hookEventName: "PostToolUse"`
    - If all requirements met: process.exit(0) silently

    The denyOutput function uses `hookEventName: "PostToolUse"` (not "PreToolUse").

    **Step 3: Write unit tests**

    Create `packages/framework/test/unit/output-contract-gate.test.ts` testing all 6 behaviors listed above. Use the established test pattern from existing framework tests — spawn the hook as a child process with stdin piped, assert on stdout/exit code. Use Bun test runner (describe/it/expect).

    For tests that need output-contracts.toml, create a temporary config file in a temp directory and set CLAUDE_PROJECT_DIR to that temp directory's parent so resolveConfig finds it.

    **Important:** The hook file header comment should explain it is a PostToolUse hook that enforces agent output contracts by checking update_task(status: "done") calls against output-contracts.toml.
