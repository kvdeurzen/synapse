# Phase 26.2: Agent Handoff Tightening - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Promote agent handoff data from free-text task descriptions to first-class structured fields, add input/output contracts to every agent in the 14-agent pipeline, unify context loading in the shared protocol, and ensure the gateway protocol reaches user projects with proper trigger recognition and intake flow.

**In scope:**
- Task schema changes (4 new handoff fields, 10→14 agent role enum)
- Shared protocol update (_synapse-protocol.md) — unified context loading
- Per-agent input/output contracts with fixed provides vocabulary
- Orchestrator fixed routing table for upstream doc_id injection
- Gateway protocol template (gateway-protocol.md) installed by init, injected by startup hook
- `/synapse:init` updated to create gateway-protocol.md template
- `project.toml` schema updated with gateway_mode field
- Tag vocabulary standardization for store_document calls

**Out of scope:**
- New MCP tools (all changes use existing create_task/update_task/store_document)
- Runtime hook enforcement of contracts (reviewers are the enforcement layer)
- Backward compatibility migration for CONTEXT_REFS/SPEC in old task descriptions (clean break)

</domain>

<decisions>
## Implementation Decisions

### Gateway Protocol
- Gateway protocol lives as a template file at `.synapse/config/gateway-protocol.md`
- `/synapse:init` creates the template file; `synapse-startup.js` injects its contents into additionalContext at session start
- **Always-on by default:** Information requests handled normally; implementation goals/features automatically route through Synapse workflow
- **Configurable:** `gateway_mode = "always-on" | "explicit"` in project.toml. Explicit mode requires manual `/synapse:refine` invocation
- **Intake sequence:** Assess complexity (trivial = just code, non-trivial = pipeline) → map codebase if stale → refine (same process as `/synapse:refine`) → scope-lock → dispatch to orchestrator
- **Gateway protocol references /synapse:refine, does NOT duplicate it** — the template is a behavioral overlay that defines when/how to trigger the existing refine flow
- Post-refinement: spawn orchestrator → handle completions/failures → present results to user

### Task Schema Changes
- 4 new Utf8 nullable fields on tasks table: `context_doc_ids`, `context_decision_ids`, `spec`, `output_doc_ids`
- Arrays stored as JSON strings in Utf8 columns (LanceDB limitation)
- Agent role enum expanded: keep old 10 roles + add 4 new (architecture_auditor, planner, plan_auditor, task_designer, task_auditor, synapse_orchestrator, product_researcher) — no migration needed, old roles stay valid
- New fields default to null on existing rows — no data migration
- **Clean break:** No backward-compatible fallback parsing of CONTEXT_REFS/SPEC blocks in descriptions. Structured fields only going forward

### Contract Enforcement
- Input/output contracts are **advisory** — defined as sections in agent .md files
- **Reviewers are the enforcement layer:** Each doer+reviewer pair has the reviewer checking the doer's output contract
  - Architecture Auditor checks architect produced architecture doc + decision drafts
  - Plan Auditor checks planner produced task tree with context fields populated
  - Task Auditor checks task-designer wrote spec field on tasks
  - Validator checks executor produced implementation + output_doc_ids
- Contract violation = **block + report** to orchestrator. Orchestrator can retry the doer or escalate to gateway
- **Both human-readable and machine-queryable:** Contracts documented in agent prompts + structured tags on documents for reviewer queries
- **Fixed provides vocabulary** (11 slugs): architecture, decision-draft, plan, task-spec, implementation, validation-findings, debug-diagnosis, research-findings, audit-findings, integration-report, code-analysis

### Document Standards
- **No DOCUMENT HEADER in content** — redundant with existing store_document fields (tags, category, phase)
- **Full tag set** on every store_document call:
  - Agent: `|architect|`
  - Type: `|architecture|`
  - Provides: `|provides:architecture|`
  - Task: `|task-abc123|`
  - Stage: `|stage:PLANNING|`
- **doc_id convention kept:** `{agent}-{type}-{task_id}` (e.g., `architect-architecture-task_abc123`)

### Orchestrator Wiring
- **Fixed routing table** in orchestrator prompt defines which upstream doc_ids to include per agent dispatch
- **Context fields set at task creation** by planner/task-designer — orchestrator does NOT modify context fields before dispatch
- **Validator discovers missing outputs** — orchestrator does NOT pre-verify output_doc_ids before dispatching validator. Validator reads task's output_doc_ids; if empty, that's a contract violation → block + report
- Orchestrator stays lean: dispatches, tracks stages, reports. Does not micromanage data flow

### Shared Protocol (_synapse-protocol.md)
- Unified 5-step context loading sequence replaces per-agent "Task Start Protocol" sections:
  1. Parse SYNAPSE HANDOFF block → extract IDs
  2. get_task_tree → load task spec, context_doc_ids, context_decision_ids
  3. get_smart_context with doc_ids from context_doc_ids field
  4. query_decisions for context_decision_ids
  5. Begin agent-specific work
- Standard tag vocabulary section added
- Standard provides vocabulary section added (11 fixed slugs)

### Claude's Discretion
- JSON string storage approach for array fields in LanceDB
- Exact gateway-protocol.md template wording
- Whether to split schema changes and framework changes into separate plans or one combined plan
- Exact prompt wording for input/output contract sections in each agent .md

</decisions>

## Specific Ideas

- The planning doc at `.planning/tighten-agent-handoff-contracts.md` contains detailed implementation guidance for Parts 1-4 — use as primary reference
- Gateway protocol should feel seamless: user describes what they want to build, Synapse kicks in automatically for non-trivial work
