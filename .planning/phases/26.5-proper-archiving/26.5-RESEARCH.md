## User Constraints (from CONTEXT.md)

### Locked Decisions

**Document Controller Agent:**
- Sonnet model, positioned after Integration Checker and before PR creation in the pipeline
- Read-only reviewer role (no code edits, no decision authority, tier_authority = [])
- Exception: Write tool access for CHANGELOG.md only — anti-rationalization entry prevents writing other files
- Runs per-feature (whole PR scope), not per-task
- Status values: APPROVED / NEEDS_REVISION / REJECTED
- Output: `document-controller-record-review-{task_id}` with category `change_record`, provides `record-review`
- Follows integration-checker.md as structural template

**Staleness Detection:**
- Hybrid approach: file-level git diff as first pass (cheap), then semantic_search for flagged items to confirm staleness
- Uses git diff + semantic_search only — no code index (search_code) dependency
- Scope: project docs only (README.md, docs/**/*.md, root *.md files) — no inline JSDoc/comments
- ADR check: NOT included — decisions live in Synapse DB via store_decision, not as separate ADR documents
- Threshold: any semantic mismatch flags as stale — no distinction between interface vs internal changes
- Flat list reporting — all stale docs reported equally, no severity levels
- Missing documentation triggers NEEDS_REVISION — if a project has no docs at all, docs should be created

**NEEDS_REVISION Routing:**
- When DC issues NEEDS_REVISION → orchestrator creates a new executor task specifically for doc fixes
- Doc fix task goes through shortened pipeline: executor → validator only (skip code-quality-reviewer)
- After doc fixes, DC runs full re-check (not targeted) to verify
- Retry limit: same as code review (max_revision_retries = 2 from trust.toml). After 2 cycles, escalate to gateway
- REJECTED → HALT and escalate to gateway (something fundamentally wrong)

**Requirement Traceability:**
- Orphaned work (tasks without requirement links) triggers NEEDS_REVISION
- Configurable: add `require_traceability = true/false` to trust.toml — default true
- When no requirement documents exist AND require_traceability = false → skip traceability checks and APPROVE
- When require_traceability = true but no requirements exist → flag as NEEDS_REVISION (requirements should be documented)

- DC generates/updates a full project CHANGELOG.md (not just PR-scoped summary)
- Parses `git log --oneline --no-merges main...HEAD` with conventional commit types grouped into sections
- git-cliff config template (cliff.toml) installed as convenience for users — DC does NOT depend on git-cliff

## Summary

Phase 26.5 adds a Document Controller (DC) agent as the final quality gate in the RPEV pipeline before PR creation. It closes three gaps that have been documented since the brainstorm: documentation going stale after code changes, absence of requirement traceability, and inconsistent version management. All three are addressed by a single cohesive DC agent plus supporting infrastructure.

The implementation is primarily additive: a new agent file, a new hook, new config entries, and orchestrator pipeline wiring. The brainstorm implementation plan is thorough and directly actionable — it specifies 6 concrete steps with exact file changes. The planner's primary job is to sequence those 6 steps into atomic tasks while respecting the existing test patterns and the constraint that the executor commit format changes before the hook is registered (to avoid circular breakage).

**Primary recommendation:** Execute the 6 steps from the brainstorm plan as distinct tasks. The most important sequencing constraint is: update executor.md commit format rule BEFORE registering the conventional-commit.js hook in install.sh/settings.json. This prevents the hook from blocking executors who haven't learned the new format yet.

### Pattern 1: Read-Only Reviewer Agent (structural)

**What:** Agent that performs analysis only — no code edits, no task creation, no store_decision. Reports APPROVED / NEEDS_REVISION / REJECTED via a stored document.
**When to use:** Post-integration quality gates (DC is 3rd such agent after Integration Checker and Code Quality Reviewer)
**Example:**
```markdown

       mcp__synapse__query_decisions, mcp__synapse__check_precedent, mcp__synapse__update_task,

       mcp__synapse__semantic_search
```

Note: Write tool for CHANGELOG.md only — enforced by anti-rationalization. NOT in frontmatter tools; the agent uses Bash to write CHANGELOG.md via `git` output redirection.

Actually per CONTEXT: the Write tool IS needed for CHANGELOG.md. Add Write to frontmatter but document the constraint explicitly in the anti-rationalization table.

### Pattern 2: PostToolUse Hook (JS)

**What:** Bun-executed JS file that reads stdin JSON, inspects tool call, returns allow/deny JSON.
**When to use:** Enforcing pre-conditions on specific MCP tool calls.
**Example (from output-contract-gate.js — established pattern):**
```javascript
#!/usr/bin/env node
import fs from "node:fs";
import { parse } from "smol-toml";
import { resolveConfig } from "./lib/resolve-config.js";

let input = "";
process.stdin.setEncoding("utf8");
process.stdin.on("data", (chunk) => (input += chunk));
process.stdin.on("end", () => {
  const data = JSON.parse(input);
  const toolName = data.tool_name || "";
  // only fire on Bash tool calls
  if (toolName !== "Bash") { process.exit(0); }
  // ... validate commit format
  // deny:
  process.stdout.write(JSON.stringify({
    hookSpecificOutput: {
      hookEventName: "PostToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "reason"
    }
  }));
  process.exit(0);
});
```

### Pattern 3: Changelog Generation via git log

**What:** DC parses `git log --oneline --no-merges main...HEAD` to build grouped changelog sections.
**When to use:** After all feature tasks are committed, before PR creation.
**Example:**
```bash
git log --oneline --no-merges main...HEAD
# Output: abc1234 feat(auth): add JWT signing
#         def5678 fix(session): handle expired tokens
```
Parsing: split on space, extract type from conventional commit prefix, group by type into sections (Features, Bug Fixes, Refactoring, etc.), write to CHANGELOG.md.

### Pattern 4: Requirement Traceability via link_documents

**What:** query_documents(category: "requirement") → for each task in scope, check link_documents for "implements" relationship type pointing to a requirement.
**When to use:** DC requirement traceability check.
**Note:** `requirement` category is already defined in `doc-constants.ts` but unused — DC is the first consumer.

### Anti-Patterns to Avoid

- **Gating on conventional-commit hook before updating executor format:** The hook will deny every executor commit if executor.md still says `[task:id]` suffix. Update executor.md first in the task sequence.
- **Adding Write to document-controller frontmatter without constraint:** Write access must be limited to CHANGELOG.md. Anti-rationalization entries enforce this; it's not enforced by tooling.
- **Registering output-contract-gate.js before adding document-controller entry to output-contracts.toml:** The gate is fail-closed. If DC is not in output-contracts.toml but gets dispatched and calls update_task(done), the hook will see an unlisted actor and allow it — which is fine for the gate. But adding DC to output-contracts.toml must happen before the gate is active, not after.
- **Adding document-controller to synapseSignatures array without adding it to SYNAPSE_HOOKS_JSON:** The dedup logic in install.sh uses synapseSignatures; if a hook command file is listed there but not in SYNAPSE_HOOKS_JSON it gets stripped, not added.

## Common Pitfalls

### Pitfall 1: Executor Commit Format Conflict

**What goes wrong:** The hook enforces `type(scope): description` with NO `[task:id]` suffix. Executor.md currently instructs `feat({task_title_slug}): {one-line summary} [task:{task_id}]` — the `[task:id]` suffix would fail the new hook validation.
**Why it happens:** The conventional-commit hook is being added in this phase, but executor.md's commit format predates it.
**How to avoid:** Task sequence MUST update executor.md's Git Commit Protocol section BEFORE registering the hook. The executor update should happen in an earlier task than the install.sh hook registration.
**Warning signs:** If the hook is registered before executor is updated, executors will get DENIED on every git commit and report BLOCKED status.

### Pitfall 2: Documentation SKILL.md already exists but is minimal

**What goes wrong:** The `documentation` skill directory already has a SKILL.md at 70 lines — it lacks: changelog format conventions, documentation freshness standards, AI-specific anti-patterns (Qodo/IBM), and the 5 quality criteria (Accuracy, Completeness, Audience awareness, Consistency, Freshness) from the CONTEXT decisions.
**Why it happens:** The skill was created in a prior phase with a general code-comment focus, not a full documentation standards focus.
**How to avoid:** The task for documentation SKILL.md is a REWRITE of the existing file, not a new file. Use Write tool to replace it entirely. The existing content has good material (why over what, single source of truth) but needs the additional sections from CONTEXT.
**Warning signs:** If the task just appends sections, the result will exceed 100 lines and violate the format standard.

### Pitfall 3: install.sh hook registration ordering

**What goes wrong:** The install.sh SYNAPSE_HOOKS_JSON is a single JSON string. Both new hooks need distinct entries: conventional-commit.js as PostToolUse + matcher "Bash", output-contract-gate.js as PostToolUse + matcher "mcp__synapse__update_task". If the PostToolUse block structure is misread, entries may be appended incorrectly.
**Why it happens:** The existing PostToolUse block has only audit-log.js with no matcher (fires on everything). The new hooks require matcher-scoped entries — different structure.
**How to avoid:** Study the existing PreToolUse structure in install.sh (lines 466-488) which already has matcher-scoped hook groups. PostToolUse currently has only the unmatched audit-log.js entry. New entries add to PostToolUse as separate hook groups with matchers.
**Warning signs:** If conventional-commit.js runs on ALL tool calls, it will slow every tool use. If output-contract-gate.js has no matcher it fires on every tool call too.

### Pitfall 4: Document Controller tool list includes search_code

**What goes wrong:** The brainstorm plan Step 1 lists `search_code` in the DC allowed_tools, but CONTEXT decision says "Uses git diff + semantic_search only — no code index (search_code) dependency."
**Why it happens:** Brainstorm was written before the refinement session locked the no-search_code decision.
**How to avoid:** Do NOT include `search_code` in the DC's agents.toml allowed_tools. The tool list in the brainstorm plan Step 1 is partially superseded by CONTEXT decisions. Follow CONTEXT.
**Warning signs:** If search_code appears in DC's tools list, it may work but introduces a code-index dependency that fails when index is stale.

### Pitfall 5: output-contract-gate.js dev mirror in .claude/settings.json

**What goes wrong:** The brainstorm plan says to also update `.claude/settings.json` (the dev mirror). However, `.claude/settings.json` is already verified to NOT have output-contract-gate.js registered. The dev mirror needs updating too — but it must follow the same matcher structure as install.sh.
**Why it happens:** Dev settings.json is maintained as a manual mirror; it's easy to forget the PostToolUse entry pattern must be identical.
**How to avoid:** After updating install.sh SYNAPSE_HOOKS_JSON, verify the exact same JSON structure is reflected in `.claude/settings.json`. The PostToolUse array needs two new matcher-scoped entries appended.
**Warning signs:** If dev mirror is out of sync, the hooks work in installed projects but not in the monorepo dev environment (where testing happens).

### Pitfall 6: DC in RPEV agent roles list in orchestrator

**What goes wrong:** synapse-orchestrator.md's RPEV Agent Roles section (line 97+) does not list Document Controller. The pipeline step 6→7 change must also update this roles list for completeness.
**Why it happens:** The agent roles list is maintained separately from the pipeline protocol steps.
**How to avoid:** When updating orchestrator Wave Execution Protocol (step 6→7 change), also add DC to the RPEV Agent Roles list and the Upstream Context Routing table.
**Warning signs:** Missing from roles list is cosmetic but could confuse future maintainers and would fail a BEH-01 audit.

### Conventional Commit Hook (PostToolUse, fail-open pattern)
```javascript
// Source: modeled on output-contract-gate.js (packages/framework/hooks/output-contract-gate.js)
#!/usr/bin/env node
import { resolveConfig } from "./lib/resolve-config.js";

let input = "";
process.stdin.setEncoding("utf8");
process.stdin.on("data", (chunk) => (input += chunk));
process.stdin.on("end", () => {
  try {
    let data;
    try {
      data = JSON.parse(input);
    } catch {
      // fail-open on parse error (not fail-closed like output-contract-gate)
      process.exit(0);
    }

    const toolName = data.tool_name || "";
    if (toolName !== "Bash") { process.exit(0); }  // only fire on Bash calls

    const toolInput = data.tool_input || {};
    const command = (toolInput.command || toolInput.cmd || "");
    if (!command.includes("git commit")) { process.exit(0); }  // non-commit calls pass

    // Validate conventional commit format
    const firstLine = (command.split("\\n")[0] || command).replace(/.*-m\s+["']/, "").replace(/["'].*/, "");
    const CONVENTIONAL_RE = /^(feat|fix|refactor|test|docs|chore|style|perf|ci|build)\([a-z0-9-]+\): .{1,}/;
    if (!CONVENTIONAL_RE.test(firstLine)) {
      process.stdout.write(JSON.stringify({
        hookSpecificOutput: {
          hookEventName: "PostToolUse",
          permissionDecision: "deny",
          permissionDecisionReason: `Conventional commit format required: type(scope): description\nAccepted types: feat|fix|refactor|test|docs|chore|style|perf|ci|build\nScope is REQUIRED (lowercase, hyphens allowed)\nFirst line max 72 chars. Got: "${firstLine}"`
        }
      }));
    }
    process.exit(0);
  } catch {
    process.exit(0);  // fail-open on any unexpected error
  }
});
```

## Open Questions

1. **Write tool access in document-controller frontmatter vs. constraint enforcement**
   - What we know: CONTEXT says DC has Write tool access for CHANGELOG.md only; the anti-rationalization table enforces this behaviorally. Write tool cannot be scoped to a single filename at the hook level.
   - What's unclear: Should `Write` appear in the frontmatter tools list, or should DC use Bash to write CHANGELOG.md (avoiding Write entirely)?
   - Recommendation: Include `Write` in frontmatter; add an anti-rationalization entry with "Using Write to modify any file other than CHANGELOG.md violates read-only reviewer role." Using Bash redirection as alternative is fragile (no partial-write protection). The behavioral constraint is sufficient.

2. **Commit message length: 72 vs 100 chars**
   - What we know: CONTEXT marks this as Claude's discretion.
   - Recommendation: Use **72 chars** — the Git community standard (matches 80-char terminal with 8-char indent). 100 chars is an alternative but 72 is more broadly compatible with `git log --oneline` display.

3. **Hook execution runtime: bun vs node**
   - What we know: MEMORY.md documents a tree-sitter + Bun runtime incompatibility (BLOCKER #39). However, ALL existing hooks already run via `bun` (not npx/node). The conventional-commit.js hook does not use tree-sitter, so Bun is safe.
   - Recommendation: Use `bun $CLAUDE_PROJECT_DIR/.claude/hooks/conventional-commit.js` consistent with all other hooks.
