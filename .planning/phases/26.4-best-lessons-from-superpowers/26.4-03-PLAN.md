    Add a new section `### Review-Reception Protocol` to the shared protocol footer. This protocol applies to ALL agents when they receive feedback from an auditor, validator, or code-quality-reviewer.

    Content (per locked decision in CONTEXT.md):
    When any agent receives review feedback:
    1. **Verify** — Check the feedback against the actual codebase. Read the referenced files/lines. Confirm the issue exists.
    2. **Evaluate** — Determine if the suggested change is actually needed. Check for false positives (reviewer misread code, suggestion conflicts with spec, suggestion violates existing decision).
    3. **Respond** — If feedback is correct: implement the fix. If feedback is wrong: push back with technical reasoning citing specific code, specs, or decisions. Do NOT agree reflexively.

    Include anti-sycophancy rules (sourced from Superpowers receiving-code-review skill):
    - Do NOT say "You're absolutely right!" or "Great catch!" before analyzing the feedback
    - Do NOT implement all suggestions without evaluating each one independently
    - Do NOT skip the YAGNI check (grep for actual usage before "implementing properly")
    - If you agree with 4 of 5 suggestions, push back on the 5th with reasoning — do not batch-accept
