## Recommended Stack (NEW for v3.0)

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Claude Code `.claude/` directory | N/A (file convention) | Project-scoped settings, hooks, agents, skills, commands | The native integration layer for Claude Code projects. `.claude/settings.json` holds hook wiring and permissions. `.mcp.json` holds MCP server config. This is where Synapse's framework files must live for Claude Code to load them automatically. |
| `.mcp.json` (project root) | N/A (file convention) | MCP server configuration, version-controlled | Separate from `.claude/settings.json` per current Claude Code conventions. Holds `mcpServers` config with `bun run packages/server/src/index.ts --db <path>`. This is the correct location for team-shared MCP config — NOT inside `settings.json`. |
| Bun shell scripts (`*.sh`) | Bun ≥1.1 | Install script, first-run experience, Ollama health check | Bun is the project runtime; the install script does not need Node.js. `Bun.$\`command\`` for subprocess calls or plain bash for portability. Use plain bash (`#!/usr/bin/env bash`) for the install script so it runs before Bun is installed. |

### Supporting Libraries

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| `smol-toml` | `1.6.0` (already installed) | TOML parse/stringify in hooks and config scripts | Already a workspace dependency. Hooks and the install script use it to read/write `synapse.toml` and `agents.toml`. No version change needed. |
| `zod` | `^4.0.0` (already installed) | Config validation in install script | Already a workspace peer dep. Reuse existing `loadSynapseConfig()` from `packages/framework/src/config.ts` in any new setup tooling. |

### No New npm Packages Needed

| Statement | Rationale |
|-----------|-----------|
| No new testing framework | `bun:test` handles E2E scenario tests. The existing 708-test suite uses it. Add E2E tests as additional `bun:test` files under `packages/framework/test/e2e/`. |
| No new config format | TOML config already in place (`synapse.toml`, `agents.toml`, `trust.toml`). The install script generates these from templates. |
| No new process manager | Synapse MCP server runs as a stdio subprocess of Claude Code — no daemon, no PID file, no systemd unit needed. |
| No Playwright / browser testing | E2E workflow validation is agent-level (does the orchestrator complete a PEV loop on a real project?), not UI-level. `bun:test` + scenario fixtures is the right tool. |
| No new HTTP client | Ollama health check uses `fetch()` (globally available in Bun). No extra package needed. |

## Claude Code Integration: Key Files and Conventions

### Directory Layout After v3.0

```
<user-project>/
├── .mcp.json                          # MCP server config (version-controlled)
├── .claude/
│   ├── settings.json                  # Hook wiring, permissions (version-controlled)
│   ├── settings.local.json            # Personal overrides (gitignored)
│   ├── agents/                        # Custom subagent definitions (*.md with YAML frontmatter)
│   │   ├── synapse-orchestrator.md    # Main orchestrator agent
│   │   ├── architect.md
│   │   └── ...                        # All 10 agents migrated here from packages/framework/agents/
│   ├── skills/                        # Skill directories (replaces packages/framework/skills/)
│   │   ├── typescript/SKILL.md
│   │   ├── bun/SKILL.md
│   │   └── project/SKILL.md          # Project-specific skill (generated by /synapse:init)
│   └── commands/                      # Slash commands (legacy path, still supported)
│       └── synapse/
│           ├── init.md                # /synapse:init
│           ├── map.md                 # /synapse:map
│           ├── plan.md                # /synapse:plan
│           └── status.md             # /synapse:status (already exists)
```

### .mcp.json Format (HIGH confidence — official docs)

```json
{
  "mcpServers": {
    "synapse": {
      "command": "bun",
      "args": ["run", "/absolute/path/to/synapse/packages/server/src/index.ts", "--db", "/absolute/path/to/synapse.db"],
      "env": {
        "OLLAMA_URL": "http://localhost:11434",
        "EMBED_MODEL": "nomic-embed-text"
      }
    }
  }
}
```

The install script generates this file in the user's project directory. Absolute paths are required because Claude Code spawns the server as a subprocess without guaranteeing working directory.

### settings.json Hook Wiring (HIGH confidence — official docs)

Hook event names (current, from official docs):

| Event | Use in Synapse |
|-------|---------------|
| `SessionStart` | Inject startup context, load tier identity, surface open work streams |
| `PreToolUse` | Tier gate enforcement for `store_decision`, tool allowlist enforcement |
| `PostToolUse` | Audit log for all tool calls |
| `SubagentStart` | Optional: log agent spawn events |
| `SubagentStop` | Optional: log agent completion |

The existing `settings.template.json` uses correct event names. One correction needed: the template uses `node` to invoke hooks — this should use the absolute path to the hook file or use `bun` consistently.

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node packages/framework/hooks/synapse-startup.js"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "mcp__synapse__store_decision",
        "hooks": [
          { "type": "command", "command": "node packages/framework/hooks/tier-gate.js" },
          { "type": "command", "command": "node packages/framework/hooks/precedent-gate.js" }
        ]
      },
      {
        "matcher": "mcp__synapse__.*",
        "hooks": [
          { "type": "command", "command": "node packages/framework/hooks/tool-allowlist.js" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "hooks": [
          { "type": "command", "command": "node packages/framework/hooks/audit-log.js" }
        ]
      }
    ]
  }
}
```

Note: The install script copies this template to `.claude/settings.json` in the user's project, replacing placeholder paths with absolute paths.

### Agent Frontmatter (HIGH confidence — official sub-agents docs)

Current agents are markdown system-prompt files with no YAML frontmatter in `packages/framework/agents/`. The v3.0 migration adds frontmatter to enable Claude Code's native subagent delegation:

```yaml
---
name: architect
description: Designs system architecture for TypeScript/Bun projects. Use when making structural or dependency decisions that affect multiple components.
tools: Read, Bash, Glob, Grep, mcp__synapse__store_decision, mcp__synapse__query_decisions, mcp__synapse__check_precedent, mcp__synapse__create_task, mcp__synapse__update_task, mcp__synapse__get_task_tree, mcp__synapse__get_smart_context, mcp__synapse__store_document, mcp__synapse__link_documents
model: opus
skills:
  - typescript
---
```

Supported frontmatter fields for agents:
- `name` (required): lowercase letters and hyphens
- `description` (required): when Claude should delegate to this agent
- `tools`: explicit allowlist (matches `allowed_tools` in current `agents.toml`)
- `model`: `sonnet`, `opus`, or `haiku`
- `skills`: list of skill names to preload (full skill content injected at startup)
- `permissionMode`: `default`, `acceptEdits`, `dontAsk`, `bypassPermissions`, `plan`
- `maxTurns`: cap on agentic turns
- `hooks`: lifecycle hooks scoped to this agent

The `skills` field in agent frontmatter enables **dynamic skill injection** — the correct mechanism for v3.0. Skills listed here get their full SKILL.md content injected into the agent's context at spawn time, not loaded lazily. This replaces the current `agents.toml` skill registry approach for Claude Code native agents.

### Skill Frontmatter (HIGH confidence — official skills docs)

Current skills already have valid frontmatter (`name`, `description`, `disable-model-invocation`, `user-invocable`). The location needs to move from `packages/framework/skills/` to `.claude/skills/` in the user's project for Claude Code to discover them.

Additional frontmatter fields available (not currently used):
- `allowed-tools`: tools the skill grants access to without per-use approval
- `model`: model override while skill is active
- `context: fork`: run skill in an isolated subagent context
- `argument-hint`: autocomplete hint for `/skill-name <hint>` invocations

For project-specific skills (generated by `/synapse:init`), the `user-invocable: false` + no `disable-model-invocation` pattern is correct: Claude auto-loads domain context without polluting the slash menu.

### Slash Commands: `.claude/commands/` vs `.claude/skills/` (MEDIUM confidence)

Official docs state commands and skills are now unified. A file at `.claude/commands/synapse/init.md` creates `/synapse:init`. A skill at `.claude/skills/synapse-init/SKILL.md` also works. The legacy `commands/` path continues to work and is the simpler approach for the 3 new commands (`/synapse:init`, `/synapse:map`, `/synapse:plan`).

Use `commands/` for the new user-facing commands because:
1. They are explicitly user-invoked (not auto-invoked by Claude)
2. The subdirectory `synapse/` creates the `synapse:` namespace automatically
3. No need for a separate skills directory per command

## Install Script: What It Must Do

The install script (`install.sh`) is a plain bash script run before Bun may be installed. It:

1. **Checks prerequisites**: `bun --version`, `ollama --version`, `claude --version`
2. **Checks Ollama health**: `curl -sf http://localhost:11434/api/tags` — Ollama's `/api/tags` endpoint returns 200 when ready (HIGH confidence — official Ollama issue tracker)
3. **Checks nomic-embed-text model**: `ollama list | grep nomic-embed-text`
4. **Runs `bun install`** in the Synapse repo root
5. **Generates `.mcp.json`** in the target project directory with absolute paths
6. **Copies `.claude/settings.json`** (from `packages/framework/settings.template.json`) into the target project's `.claude/` directory
7. **Copies agent files** from `packages/framework/agents/` into `.claude/agents/`
8. **Copies skill directories** from `packages/framework/skills/` into `.claude/skills/`
9. **Copies command files** from `packages/framework/commands/` into `.claude/commands/`
10. **Generates `config/synapse.toml`** in the target project from template, replacing `SYNAPSE_DB_PATH` with an absolute path
11. **Reports success** with next steps

No new packages are needed for the install script. It uses `bash` built-ins + `curl` (standard) + `bun` (post-check).

## Dynamic Skill Injection: Mechanism

The v2.0 `skills.ts` loaded skill content programmatically for an orchestrator process. In v3.0, skills are injected differently depending on context:

| Context | Mechanism | Notes |
|---------|-----------|-------|
| Agent spawned by Claude Code | `skills:` field in agent frontmatter | Full SKILL.md content injected at agent startup. No runtime code needed. |
| User-invoked skill | `/skill-name` slash command | Loads SKILL.md on invocation |
| Auto-invoked skill | Claude reads description, decides relevance | `user-invocable: false` hides from menu but keeps auto-invocable |

The `packages/framework/src/skills.ts` module and its token-budget logic remain useful for:
- The install-time validation (verifying skills don't exceed token budget before copying)
- Any future programmatic skill assembly (e.g., composing a project skill from domain components)

No changes to `skills.ts` are needed for v3.0. The install script copies skill files; Claude Code handles injection.

## E2E Workflow Validation: Approach

E2E testing for agent orchestration workflows does NOT use Playwright, Selenium, or browser automation. The correct approach:

1. **Scenario fixture tests** (`packages/framework/test/e2e/`): Define a small real project (e.g., a TypeScript utility library with 5 files), run the full PEV loop via the `/synapse:plan` command in a controlled environment, assert the resulting task tree structure and decision records in LanceDB.

2. **Hook integration tests**: Invoke hooks directly with synthetic JSON payloads (the existing hook test pattern already does this). No new test infrastructure.

3. **MCP server smoke tests**: Verify the 24 tools respond correctly with a test database. Already covered by the 708-test suite.

4. **Manual E2E validation**: Run Synapse on a real project (e.g., the Synapse repo itself) and walk the user journey from `/synapse:init` through `/synapse:plan` and agent execution. Document the session transcript as acceptance evidence.

Framework: `bun:test` (already in use). No new test libraries needed.

## Alternatives Considered

| Recommended | Alternative | Why Not |
|-------------|-------------|---------|
| `.mcp.json` for MCP server config | `mcpServers` in `.claude/settings.json` | `.mcp.json` is the current Claude Code convention for project-scoped MCP (separate from hook/permission settings). Mixing concerns in `settings.json` is the legacy pattern. |
| Agent `skills:` frontmatter for injection | Programmatic skills.ts loading | Skills.ts requires a running orchestrator process. Frontmatter injection is zero-code, native to Claude Code, and survives context compaction correctly. |
| Plain bash install script | Bun-based install script | Install script must run before Bun is confirmed present. Bash is universal on macOS/Linux/WSL. |
| `bun:test` for E2E scenarios | Playwright / vitest | E2E for agents is about LanceDB state and agent behavior, not UI. bun:test already runs in CI and covers the data layer. |
| `.claude/commands/synapse/` for slash commands | `.claude/skills/synapse-init/` | Commands directory gives the `synapse:` namespace automatically via directory structure. Skills directory would require `name: synapse:init` in frontmatter manually, and adds a `/SKILL.md` indirection. |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| `smol-toml` `stringify()` for config generation | Round-trips lose comments; generated config becomes unmaintainable | Generate config files from string templates in the install script; preserve human-readable TOML with comments |
| `@anthropic-ai/claude-agent-sdk` (standalone) | v2.0 decision established Claude Code native framework; Agent SDK would create a second orchestration runtime | Claude Code `.claude/` framework files |
| Global `~/.claude/settings.json` for Synapse hooks | Pollutes user's global Claude Code config; breaks other projects | Project-scoped `.claude/settings.json` only |
| `npm` / `npx` in install scripts | Project uses Bun; mixing package managers causes hoisting conflicts | `bun` / `bunx` throughout |
| Hardcoded relative paths in hook scripts | Hooks run from Claude Code's working directory which may differ | Use `process.cwd()` resolution with fallback paths (existing hook pattern already does this correctly) |
| `node` in settings.json hook commands | `node` may not be on PATH in all environments; hooks in the current template use it for compatibility but should be verified | Absolute path to `bun` or ensure `bun` is on PATH; the install script should patch hook command paths |
