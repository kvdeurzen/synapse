## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Gateway Architecture
- The main Claude session IS the gateway agent, established via CLAUDE.md protocol
- Gateway owns ALL user communication — no subagent interacts with the user directly
- Gateway runs refinement sessions (currently /synapse:refine), captures initial scope, presents findings
- Gateway spawns the Product Researcher as a research subagent during refinement
- Gateway delegates execution dispatch to the Orchestrator (subagent)
- Gateway handles failure routing — when orchestrator reports failures, gateway decides: retry, debug, or ask user

#### Orchestrator Scope Reduction
- Orchestrator is a subagent, NOT the main session conductor
- Responsibilities: assign tasks to pool slots, persist stage transition documents, report completions/failures to gateway
- Does NOT make decisions, does NOT handle failure retries, does NOT interact with user
- State lives in Synapse MCP (task tree, stage documents, pool state) — orchestrator reads state, acts, reports
- If orchestrator crashes or runs out of context, a new one can be spawned — DB is the source of truth

#### Agent Pipeline (doer + reviewer pairs)

Stage 1 — Refinement (Gateway-led):
- Agent: **Product Researcher** (renamed from Product Strategist)
- Role: Researches the user's scope, systematically pokes holes, identifies improvements and unclarities
- Does NOT make decisions — returns findings to gateway for user review
- Loop: Gateway ↔ Product Researcher ↔ User until product plan is solid
- Only then does scope move to Architecture stage

Stage 2 — Architecture:
- Agent: **Architect** (name kept, responsibilities narrowed)
- Role: Designs implementation architecture ONLY — files, abstraction layers, internal/external interfaces
- Does NOT decompose into executable work (that's the Planner's job)
- Drafts Tier 1-2 decisions as part of architectural proposal
- Involves user for Tier 0-1 decisions per involvement config
- Reviewer: **Architecture Auditor** (new agent)
- Role: Reviews architectural proposals for soundness — correct abstraction levels, interface contracts, no over-engineering, alignment with existing patterns
- Reviews and approves/rejects decision drafts
- Moves approved decisions from draft → active state

Stage 3 — Planning (the big plan):
- Agent: **Planner** (renamed from Decomposer)
- Role: Creates executable task tree from approved architecture (epics → features → components → tasks)
- Structures work, wires dependencies, attaches context refs
- Reviewer: **Plan Auditor** (new agent, replaces old Plan-Reviewer)
- Role: Goal-backward verification of the task tree — requirement coverage, scope sanity, locked-decision compliance, context alignment
- Modeled after GSD Plan-Checker's 8-dimension analysis
- Blocks execution if structure doesn't cover the goal

Stage 4 — Task Design (detailed specs):
- Agent: **Task Designer** (new agent, replaces task-level planning in old Decomposer)
- Role: Writes detailed task specifications with mock code, exact file paths, integration points, expected inputs/outputs
- Specs stored in Synapse MCP task descriptions (not separate files)
- Drafts Tier 2-3 decisions needed for implementation
- Reviewer: **Task Auditor** (renamed from Plan-Reviewer)
- Role: Verifies each task spec is execution-ready — mock code present, files identified, dependencies correct
- Reviews and activates Tier 2-3 decision drafts per involvement mode

Stage 5 — Execution:
- Agent: **Executor** (name kept)
- Role: Implements per the detailed task spec with mock code
- Reviewer: **Validator** (name kept)
- Role: Verifies implementation matches spec and decisions

#### Decision Flow
- Decisions are drafted by the agent where they arise (Architect for Tier 1-2, Task Designer for Tier 2-3)

## Summary

Phase 26.1 is a prompt-engineering and documentation phase. It does not change the MCP server, add new slash commands, or alter the data model. All work consists of rewriting, renaming, and creating markdown files for agent prompts and configuration — plus updating CLAUDE.md to establish the gateway protocol.

The central challenge is restructuring responsibilities across a pipeline that currently blurs stage boundaries. The existing orchestrator does too much (makes decisions, handles failures, interacts with users, manages the pool). The new design separates concerns cleanly: a main session gateway handles everything visible to the user; the orchestrator becomes a pure dispatcher; each pipeline stage has a dedicated doer agent paired with a reviewer. The draft→review→activate decision flow enforces that no decision is ever "eagerly" stored — the reviewer at each stage acts as an activation gate.

The existing codebase gives solid starting material. Every new agent derives from an existing one (Product Researcher from Product Strategist, Planner from Decomposer, Plan Auditor from Plan-Reviewer, Task Designer from Architect, Task Auditor from Plan-Reviewer). The Architect and Executor keep their names but have significantly narrowed scopes. The `_synapse-protocol.md` shared header stays unchanged and included in all agents. `agents.toml` needs entries for 4 new agent slugs and removal/rename of 3 old ones.

**Primary recommendation:** Implement in three groups: (1) CLAUDE.md gateway protocol + new agent skeletons to establish the structure, (2) pair-by-pair stage rewrites (Stage 1 through 5), (3) config + documentation updates. This sequencing ensures the architectural intent is visible before the details are filled in.

## Architecture Patterns

### Pattern 1: Gateway as Main Session

The main Claude Code session is established as a gateway via CLAUDE.md. This pattern is not new to Claude Code — CLAUDE.md instructions already govern the main session. The new work is making the gateway protocol EXPLICIT.

**CLAUDE.md gateway protocol (new section):**
```markdown
## Synapse Gateway Protocol

You are the Synapse Gateway — the sole point of contact with the user for all Synapse work.

**Your responsibilities:**
- Capture user scope and run refinement sessions (spawn Product Researcher as subagent)
- Present architectural proposals and decision drafts requiring Tier 0-1 user approval
- Delegate execution dispatch to the Orchestrator subagent
- Receive completion/failure reports from the Orchestrator and decide next action
- NEVER allow a subagent to interact directly with the user

**Subagent spawning:**
- Spawn Product Researcher for scope research during refinement
- Spawn Orchestrator to dispatch execution pipeline
- All other agents are spawned BY the Orchestrator, not by you directly

**Failure routing:**
When the Orchestrator reports a failure:
- Retry: spawn a new Orchestrator with the failure context in the handoff
- Debug: spawn Debugger directly to diagnose before retrying
- Ask user: surface the failure and options, wait for user decision
```

**When to use:** Every Synapse project. This is the universal session model.

### Pattern 2: Doer + Reviewer Pair Structure

Each pair follows a consistent structure:

**Doer prompt sections:**
1. Frontmatter (name, description, tools, model, mcpServers)
2. MCP Usage (actor name, examples, tool table)
3. Core Responsibilities (what this agent produces)
4. Work Protocol (numbered steps)
5. Decision Draft Protocol (how to store draft decisions — NOT active)
6. Key Tool Sequences (concrete call patterns)
7. Constraints (what agent CANNOT do)
8. `{{include: _synapse-protocol.md}}`

**Reviewer prompt sections:**
1. Frontmatter (read-only tool set)
2. MCP Usage (actor name)
3. Review Protocol (load context → check criteria → issue verdict)
4. Decision Activation Protocol (when reviewer approves, it calls store_decision with status:active)
5. Verdict Format (APPROVE / REJECT with structured findings)
6. Constraints (cannot make new decisions, cannot execute)
7. `{{include: _synapse-protocol.md}}`

### Pattern 3: Draft → Review → Activate Decision Flow

The existing `store_decision` MCP tool accepts a `status` field. The new convention:

```
Doer agent:
  store_decision(status: "draft", tier: N, title: "...", rationale: "...", actor: "doer-slug")

Reviewer agent (after checking rationale quality, precedent conflicts, tier correctness):
  // If approved:
  update_decision(decision_id: "...", status: "active", actor: "reviewer-slug")
  // If rejected:
  store_document(doc_id: "rejection-{decision_id}", content: "Rejection reason...", actor: "reviewer-slug")
```

**Anti-pattern to avoid:** Any agent calling `store_decision(status: "active")` directly. All active decisions must pass through the reviewer gate.

**Note on MCP capability:** The existing `store_decision` tool must support `status: "draft"` and there must be an `update_decision` tool (or `store_decision` upsert with the same decision_id and updated status). Verify this is supported before writing it into agent prompts — if not, the Plan Auditor and Architecture Auditor need to use a workaround (e.g., store a draft as a document, then re-call store_decision as active after review). This is a LOW confidence detail that requires checking the MCP server's decision API during implementation.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Shared MCP protocol instructions | Copy into each agent | `{{include: _synapse-protocol.md}}` | Already used in all 11 agents; adding to new agents is one line |

| Stage gate logic | Re-implement in each agent | Keep in orchestrator (the only dispatcher) | Distributed gate logic creates inconsistency |

**Key insight:** The tooling layer (hooks, toml configs) already enforces most constraints at runtime. Agent prompts should state behavioral intent; the system enforces it. Don't duplicate enforcement logic in prompts.

## Common Pitfalls

### Pitfall 1: Forgetting mcpServers in New Agent Frontmatter

**What goes wrong:** Agent is spawned as a subagent and silently loses all MCP tools. Tool calls fail or are silently skipped.
**Why it happens:** Claude Code only injects MCP connections listed in agent frontmatter. This was a silent bug for the first 18 phases.
**How to avoid:** Every new agent file MUST have `mcpServers: ["synapse"]` in frontmatter. This is non-negotiable.
**Warning signs:** Agent produces output without any MCP tool calls in a task that should call several.

### Pitfall 2: Orchestrator Still Making Decisions

**What goes wrong:** Old orchestrator habits bleed into the rewrite. The orchestrator stores decisions when it "synthesizes" results, or resolves involvement modes itself instead of receiving them from the gateway.
**Why it happens:** The current orchestrator has `[0,1,2,3]` tier authority in trust.toml. Simply removing that authority in toml without updating the prompt leaves the agent confused.
**How to avoid:** Remove tier authority from orchestrator in trust.toml AND explicitly state in orchestrator.md that it has no decision authority. The orchestrator's only write operations are task status updates and pool-state documents.
**Warning signs:** Orchestrator prompt still contains "store_decision" in its tool list.

### Pitfall 3: Draft Decision Status Not Supported

**What goes wrong:** Agent prompts instruct agents to `store_decision(status: "draft")` but the MCP tool may not support a draft status. Agents fail silently or decisions land as active immediately.
**Why it happens:** The design assumes `status: "draft"` exists on the decision schema, but this was not verified against the actual MCP server implementation.
**How to avoid:** Before writing draft→activate patterns into agent prompts, check `packages/server/src/tools/` for the decision tool's accepted status values. If draft status is not supported, implement a workaround (store draft as a document, call store_decision only when reviewer approves).
**Warning signs:** All decisions end up as "active" regardless of review status.

### Pitfall 4: Plan Auditor Trying to Re-Decompose

**What goes wrong:** Plan Auditor (goal-backward verifier) starts proposing new tasks or restructuring the task tree instead of just blocking and reporting.
**Why it happens:** The old Plan-Reviewer had "block and report" authority but the new Plan Auditor inherits from a context where it saw the full 8-dimension analysis and may try to fix problems itself.
**How to avoid:** Explicitly state in Plan Auditor prompt: "You block and report. You NEVER create tasks, modify task descriptions, or suggest restructuring directly. Your output is a structured rejection report that the Planner uses to revise."
**Warning signs:** Plan Auditor output contains `create_task` calls.

### Pitfall 5: Gateway Protocol Not Honored in Sessions Where Orchestrator Was Already Active

**What goes wrong:** User runs `/synapse:status` or continues a session where the orchestrator is already mid-pipeline. The gateway protocol in CLAUDE.md is not retroactively applied to an in-progress session — the main session may still route queries directly to the orchestrator.
**Why it happens:** CLAUDE.md instructions govern new sessions. Existing sessions have context from before the protocol update.
**How to avoid:** Document in CLAUDE.md that the gateway protocol applies to all sessions, including resumed ones. Gateway should check for in-flight orchestrator work and surface it via the pool-state document on session start.
**Warning signs:** User discovers they can still bypass the gateway by typing directly to orchestrator behaviors.

### Pitfall 6: agents.toml Not Updated for Deleted Slugs

**What goes wrong:** Old agent slugs remain in agents.toml (`product-strategist`, `decomposer`, `plan-reviewer`). The tool-allowlist hook enforces against the old slug names. New agents (`product-researcher`, `planner`, `plan-auditor`, `task-designer`, `task-auditor`, `architecture-auditor`) have no toml entries and therefore fall through to no-restriction behavior.
**Why it happens:** Agent files and config are separate — easy to update one and forget the other.
**How to avoid:** Treat the agents.toml update as a required parallel task with every agent file write. Never create a new agent file without its toml entry.
**Warning signs:** `tool-allowlist.js` logs show unknown agent slug warnings.

### Pitfall 7: Tier Authority Mismatch After Orchestrator Scope Reduction

**What goes wrong:** trust.toml still grants the orchestrator `[0,1,2,3]` tier authority. The tier-gate hook will allow orchestrator to store any decision tier. The prompt says "no decisions" but the hook doesn't enforce it.
**Why it happens:** trust.toml and agent prompts are not automatically synchronized.
**How to avoid:** Update `[tier_authority]` in trust.toml: orchestrator should be `[]` (no tier authority) in the new design. New agents (product-researcher, planner, task-designer, architecture-auditor, plan-auditor, task-auditor) need appropriate tier entries:
- `product-researcher = []` (returns findings to gateway; gateway/user decides)
- `architect = [1, 2]` (unchanged)
- `architecture-auditor = []` (reviews drafts, activates — does not create new decisions)
- `planner = [2]` (functional design decisions during decomposition)
- `plan-auditor = []` (blocks/approves; no new decisions)
- `task-designer = [2, 3]` (detailed spec decisions)
- `task-auditor = []` (blocks/approves; no new decisions)

### GSD Plan-Checker 8-Dimension Analysis (to inform Plan Auditor, HIGH confidence)

From `gsd-plan-checker.md` — these are the dimensions Plan Auditor should implement:

1. **Requirement Coverage** — every requirement has covering task(s)
2. **Task Completeness** — every task has Files + Action + Verify + Done
3. **Dependency Correctness** — dependencies are valid, acyclic, wave-assigned correctly
4. **Key Links Planned** — artifacts wired together, not created in isolation
5. **Scope Sanity** — no scope creep, no deferred items included
6. **Verification Derivability** — verifications can be derived from acceptance criteria
7. **Context Compliance** — plans honor locked decisions, exclude deferred items
8. **Context Budget** — execution can complete within context window limits

The Plan Auditor adapts these to Synapse's task tree model: requirements map to epic/feature task descriptions, "tasks" are Synapse leaf tasks, dependencies are the `dependencies` field in task tree.

   - What we know: The decision schema has lifecycle states (active, superseded, revoked) from brainstorm docs and STATE.md
   - What's unclear: Whether "draft" is a valid initial status the MCP server accepts, or whether only "active" is accepted on creation
   - Recommendation: Check `packages/server/src/tools/` for `store_decision` implementation before writing agent prompts. If draft is unsupported, use the document-based workaround pattern documented above.

2. **Does `update_decision` exist as an MCP tool?**
   - What we know: `store_document` supports upsert via doc_id reuse. `store_decision` may support the same pattern.
   - What's unclear: Whether there is a separate `update_decision` tool or whether reviewers must re-call `store_decision` with the decision_id to activate a draft.
   - Recommendation: Check `packages/server/src/tools/` for the decision mutation API.

3. **Architecture Auditor vs Plan Auditor — shared base or independent?**
   - This is Claude's discretion per CONTEXT.md.
   - What we know: Both are reviewer agents with similar structure (read-only tools, load context, check criteria, issue verdict). Both activate decision drafts.
   - Recommendation: Start with fully independent agents. If 70%+ content is identical after writing both, consider extracting `_synapse-auditor-protocol.md` as a shared base (similar to `_synapse-protocol.md`). Don't pre-optimize before writing.

4. **Product Researcher vs Researcher — capabilities overlap?**
   - This is Claude's discretion per CONTEXT.md.
   - What we know: `researcher.md` uses WebSearch/WebFetch/Context7 for external technical research. Product Researcher should poke holes in user scope — this is more analytical than web-search intensive.
   - Recommendation: Make them distinct. Product Researcher's core capability is structured Socratic questioning and gap identification from existing context (query_decisions, get_smart_context). General Researcher does external technical investigation. They're complementary, not overlapping.
