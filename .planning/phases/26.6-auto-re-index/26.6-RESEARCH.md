**Primary recommendation:** Three plans — one per stream — executed in order (bugfix first, orchestrator second, init third). Existing scanner tests run under `bun test` and will catch regressions in the Bun-API replacement.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Trigger Timing**
- Re-index after each wave of executor tasks completes, before validators are dispatched
- Validators and downstream agents always search fresh code
- Keep the existing feature-completion re-index (synapse-orchestrator.md line 172) as belt-and-suspenders
- RPEV pipeline only — no hooks on manual git commits; users run `/synapse:map` manually when needed
- `/synapse:init` runs full `index_codebase` (with embeddings) at the end of initialization, so the user doesn't need a separate `/synapse:map` step
- `/synapse:map` remains as a standalone command for manual re-indexing after individual changes

**Wiring Mechanism**
- Orchestrator calls `index_codebase` directly between wave completion and validator dispatch — no codebase-analyst agent spawn
- Explicit step added to synapse-orchestrator.md wave execution section: "After wave executors complete, call index_codebase before dispatching validators"
- Audit log captures re-index calls automatically (MCP layer) — no extra logging to stage documents
- `/synapse:init` (init.md) updated to call `index_codebase` after `init_project` succeeds

**Index Scope**
- Always full project scan (no include_patterns) — incremental hashing already skips unchanged files
- Default gitignore-based exclusions — no extra exclude_patterns needed
- Same approach for both wave-level re-index and /synapse:init index

**Failure Handling**
- Wave-level re-index failure: warn and continue — validators run with stale index rather than blocking the pipeline
- `/synapse:init` re-index failure: warn but complete init — project is usable without index; user told to run `/synapse:map` when Ollama is available
- No consecutive failure tracking — each wave independently tries to re-index
- Re-index is non-blocking: pipeline never halts for indexing failures

- Add `glob` and `minimatch` as server package dependencies
- Detailed fix log: `.planning/issues/260313_bun.md`

### Pattern 1: Bun.Glob Replacement in scanner.ts

**What:** Replace `Bun.Glob` async iterator with `glob()` from the `glob` npm package and `minimatch()` for pattern matching.

**When to use:** Whenever the server runs under Node.js (`npx tsx`) instead of Bun.

**Current broken code (scanner.ts lines 104-114):**
```typescript
let includeGlobs: InstanceType<typeof Bun.Glob>[] | null = null;
if (opts?.include_patterns && opts.include_patterns.length > 0) {
  includeGlobs = opts.include_patterns.map((p) => new Bun.Glob(p));
}

const files: FileEntry[] = [];
const glob = new Bun.Glob("**/*.{ts,tsx,py,rs}");

for await (const relRaw of glob.scan({ cwd: projectRoot, absolute: false })) {
```

**Fixed code (from prototype in `.planning/issues/260313_bun.md`):**
```typescript
import { glob } from "glob";
import { minimatch } from "minimatch";

// ... (in scanFiles function body) ...

const includePatterns: string[] | null =
  opts?.include_patterns && opts.include_patterns.length > 0 ? opts.include_patterns : null;

const files: FileEntry[] = [];

const matches = await glob("**/*.{ts,tsx,py,rs}", { cwd: projectRoot, nodir: true });

for (const relRaw of matches) {
  // ... rest of loop body unchanged ...

  // include_patterns check (replaces Bun.Glob match):
  if (includePatterns !== null) {
    const matchesInclude = includePatterns.some((p) => minimatch(relPath, p));
    if (!matchesInclude) continue;
  }
```

Note: The `for await` becomes a plain `for` loop since `glob()` returns a `Promise<string[]>` (resolved with `await`), not an async iterable.

### Anti-Patterns to Avoid

- **Applying the fix only to `.claude/server/`:** The prototype fix was applied to the deployed copy but NOT to `packages/server/src/`. The planner must target `packages/server/src/` — install.sh will pick up the fix on next install.
- **Making re-index blocking in the orchestrator:** The pipeline never halts for indexing failures. Warn and continue.
- **Checking Ollama in init before step 7:** The current anti-pattern rule says "Do NOT check for Ollama during init." The new step 7b is added AFTER init_project (step 7) and is opt-in based on Ollama availability — it does not block the init flow.
- **Forgetting to add `mcp__synapse__index_codebase` to init.md allowed-tools:** Without this, the tool call will fail silently in the command.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| File glob matching | Custom recursive directory walker | `glob` npm package | Handles symlinks, nodir, cross-platform correctly |
| Pattern matching | Custom string match | `minimatch` | Handles `**` glob patterns correctly (including edge cases like `**/` prefix matching) |
| File reading | `Bun.file().text()` | `node:fs/promises readFile` | Works under both Node.js and Bun runtimes |

**Key insight:** The Bun API surface is not available under `npx tsx` (Node.js). Never use `Bun.*` APIs in the MCP server code — the server is always deployed under Node.js per the MEMORY.md constraint.

## Common Pitfalls

### Pitfall 1: Fix Applied to Wrong Directory
**What goes wrong:** Editor targets `.claude/server/src/` (the deployed copy already fixed by prototype) instead of `packages/server/src/` (the source of truth that install.sh distributes).
**Why it happens:** The prototype fix log references `.claude/server/src/` paths.
**How to avoid:** Always edit `packages/server/src/`. The `.claude/server/` copy is generated by install.sh from `packages/server/`.
**Warning signs:** If the fix edits files under `.claude/server/` but `packages/server/` still has `Bun.Glob` — it will be lost on next install.

### Pitfall 2: for-await vs for Loop
**What goes wrong:** `glob()` from the npm package returns `Promise<string[]>`, not an async iterable. Using `for await` on the resolved array will still work (arrays are sync iterables) but is semantically wrong and slightly confusing.
**Why it happens:** The old `Bun.Glob.scan()` used `for await`. Copy-paste migration keeps `for await`.
**How to avoid:** Use plain `for (const relRaw of matches)` after `const matches = await glob(...)`.

### Pitfall 3: glob Package API Version Mismatch
**What goes wrong:** `glob` v9+ changed the API (removed callbacks, made it ESM-native). The import and call signature in the prototype (`import { glob } from "glob"` + `await glob(pattern, { cwd, nodir })`) is the current v10+ ESM API.
**Why it happens:** Older tutorials show `glob.sync()` or callback style.
**How to avoid:** Use exactly the import/call shown in the prototype fix. `glob` v11 is current as of early 2026.

### Pitfall 4: Orchestrator Re-index Position
**What goes wrong:** Re-index step placed after validators instead of before — validators see stale code.
**Why it happens:** Misreading the CONTEXT decision.
**How to avoid:** Step must be "after wave executors complete, BEFORE validator dispatch" — between step 2 and step 3 in the Wave Execution Protocol.

### Pitfall 5: init.md Re-index Blocks on Missing Ollama
**What goes wrong:** init.md calls `index_codebase` unconditionally — fails loudly if Ollama is unavailable, making init appear broken.
**Why it happens:** Not checking Ollama first.
**How to avoid:** Always check Ollama + nomic-embed-text before calling `index_codebase` in init. Treat failure as a warning, not an error. The existing anti-pattern "Do NOT check for Ollama during init" refers to the core setup flow — the indexing step is explicitly opt-in and happens after all setup is done.

### Pitfall 6: scanner.ts Tests Break if glob Behavior Differs
**What goes wrong:** The existing scanner.test.ts tests call `scanFiles()` with temp directories. If `glob` package handles `nodir` differently than `Bun.Glob`, test expectations may drift.
**Why it happens:** Subtle behavioral differences between glob implementations.
**How to avoid:** Run `bun test packages/server/test/services/code-indexer/scanner.test.ts` immediately after applying the fix. All 15 existing tests must pass. The prototype fix was validated in a real project (rpi-camera-py) so behavioral equivalence is confirmed.
