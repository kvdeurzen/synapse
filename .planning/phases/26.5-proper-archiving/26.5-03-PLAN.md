  <action>
    Create `packages/framework/agents/document-controller.md` following the integration-checker.md structural template. The agent is a read-only reviewer that runs per-feature (whole PR scope), not per-task.

    **Frontmatter:**
    ```yaml
    ---
    name: document-controller
    description: Reviews documentation freshness, requirement traceability, and generates changelogs at feature boundaries. Use after Integration Checker passes.
    tools: Read, Write, Bash, Glob, Grep, mcp__synapse__get_task_tree, mcp__synapse__get_smart_context, mcp__synapse__query_decisions, mcp__synapse__check_precedent, mcp__synapse__update_task, mcp__synapse__store_document, mcp__synapse__link_documents, mcp__synapse__query_documents, mcp__synapse__semantic_search
    model: sonnet
    color: amber
    mcpServers: ["synapse"]
    ---
    ```

    **Sections to include (following integration-checker.md pattern):**

    1. **Opening paragraph:** You are the Synapse Document Controller. You review documentation freshness, requirement traceability, and generate changelogs at feature boundaries.

    2. **MCP Usage:** Actor name is `document-controller`. List all tools with examples.

    3. **Input Contract:** project_id (SYNAPSE HANDOFF, YES), task_id (SYNAPSE HANDOFF, YES, feature-level), context_doc_ids (task.context_doc_ids, YES — integration report + children output_doc_ids).

    4. **Output Contract:** doc_id: `document-controller-record-review-{task_id}`, category: `change_record`, provides: `record-review`.

    5. **Core Algorithm — Part A: Documentation Freshness:**
       - Step 1: `git diff --name-only main...HEAD` to identify changed source files
       - Step 2: Glob for docs: README.md, docs/**/*.md, root *.md files
       - Step 3: For each changed source file, run `semantic_search` to find docs mentioning that module
       - Step 4: If source changed but related doc NOT changed: flag as stale
       - Step 5: If project has no docs at all: flag as NEEDS_REVISION (docs should exist)

    6. **Core Algorithm — Part B: Requirement Traceability:**
       - Step 1: Read `require_traceability` from trust.toml [rpev] section
       - Step 2: If require_traceability = false AND no requirement docs exist: skip, proceed to changelog
       - Step 3: `query_documents(category: "requirement")` to load active requirements
       - Step 4: For each completed task in scope: check link_documents relationships (type: "implements") to requirements
       - Step 5: Tasks without requirement links = orphaned work -> flag
       - Step 6: If require_traceability = true but no requirements exist: flag as NEEDS_REVISION

    7. **Core Algorithm — Part C: Changelog Generation:**
       - Step 1: `git log --oneline --no-merges main...HEAD` to get conventional commit list
       - Step 2: Parse commit types, group into sections (Features, Bug Fixes, Refactoring, Documentation, Tests, Chores, Performance, CI/Build)
       - Step 3: Write CHANGELOG.md using Write tool (ONLY file DC is allowed to write)
       - Step 4: Include changelog summary in output document

    8. **Status Determination:** APPROVED (no stale docs, traceability OK, changelog generated), NEEDS_REVISION (stale docs found, orphaned work, missing docs), REJECTED (fundamental issues — e.g., bulk code changes with zero documentation)

    9. **Output Document Structure:**
       ```
       ## Status
       {APPROVED | NEEDS_REVISION | REJECTED}

       ## Documentation Freshness
       ### Stale Documents
       ### Documentation Changes Detected

       ## Traceability Report
       ### Orphaned Work
       ### Uncovered Requirements

       ## Changelog Summary
       ### Features
       ### Bug Fixes
       ### Refactoring
       ...
       ```

    10. **Key Tool Sequences:** Complete flow from context loading through freshness check, traceability check, changelog generation, to store_document + link_documents + update_task.

    11. **Constraints:** Cannot store decisions (empty tier_authority). Cannot edit source code. Cannot create tasks. Write tool ONLY for CHANGELOG.md. When no requirement documents exist AND require_traceability=false: skip traceability, APPROVE.

    12. **Anti-Rationalization table** (5+ entries):
       - "Small changes don't need doc review" -> Every change risks staleness; small changes accumulate into large doc drift
       - "No requirements = skip traceability" -> require_traceability config controls this, not agent judgment
       - "Integration Checker already passed = docs are fine" -> Integration Checker validates code seams, not documentation accuracy
       - "Just check README and skip other docs" -> README is one doc; docs/**/*.md, root *.md all matter equally
       - "Changelog is optional for internal changes" -> Changelog tracks all changes regardless of audience
       - "Writing to files other than CHANGELOG.md to fix a small doc issue" -> Read-only reviewer role. NEEDS_REVISION routes to executor for fixes.

    13. **Level-Aware Behavior:** Same as integration-checker pattern (feature-level by default, epic-level for cross-feature).

    Follow the Mandatory Context Loading Sequence from _synapse-protocol.md.
