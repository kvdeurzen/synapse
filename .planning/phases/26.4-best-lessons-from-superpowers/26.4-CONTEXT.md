## Implementation Decisions

### Scope & Priority
- 9 of 10 practices in scope (P1: #1, #3, #10; P2: #2, #4, #7; P3: #5, #6, #8)
- Visual brainstorming companion (#9) excluded — P4, high effort, deferred
- P3 items (#5, #6, #8) get full implementation, not lightweight prompt additions
- Phase 26.3 keeps executor-specific TDD changes; this phase generalizes escalation statuses and behavioral enforcement to ALL agents
- Skill pressure testing: protocol document with 2-3 example scenarios; actual pressure testing happens organically in future usage

### Anti-Rationalization Engineering (#1)
- Rationalization tables are inline per-agent (not centralized in _synapse-protocol.md)
- Every agent (all 14+) gets at least 3-5 rationalizations for its critical constraints
- Format: markdown table with columns | Rationalization | Why It's Wrong | What To Do Instead |
- **Critical: rationalizations sourced from external frameworks and best practices only** — NOT hypothetically generated. Research phase must systematically source from Superpowers' documented catalogs and other published agentic frameworks. Reason: self-generated rules are rules the model already knows and can rationalize around; externally battle-tested rules are more effective
- Initial tables are hypothesis-based from external sources; refined through actual pressure testing in future usage sessions

### Verification-Before-Completion Gate (#3)
- Defense in depth: hardened executor prompt AND runtime PostToolUse hook
- Executor output contract requires: exact verification commands run, full output, pass/fail determination
- Validator independence: validator runs tests/verification FIRST, forms independent assessment, THEN reads executor's implementation summary document. Explicitly prohibited from reading executor's report before own verification
- Validator context loading sequence changes: load spec → run tests → read code → form assessment → then read executor output doc for discrepancies

### Output Contract Enforcement via Hooks (#10)
- New `output-contracts.toml` config maps agent → expected doc_id patterns → provides slug → required tags
- This config is the **single source of truth** for both prompt guidance and runtime enforcement (replaces the provides vocabulary currently in _synapse-protocol.md)
- Hook covers ALL pipeline agents (executor, validator, architect, planner, task-designer, all auditors, code-quality-reviewer)
- Enforcement mode: fail-closed — if required documents are missing, update_task(status: done) is blocked
- Matches existing tier-gate and tool-allowlist enforcement model

### Skill Pressure Testing Protocol (#2)
- Create a methodology document (not a SKILL.md) describing how to test skills and agent prompts
- Include 2-3 worked example scenarios showing the RED-GREEN-REFACTOR cycle for process skills
- Actual pressure testing happens in future usage, not in this phase
- Document stored in `.planning/` or `packages/framework/docs/`

### Two-Stage Review Pipeline (#4)
- New agent: **code-quality-reviewer** (15th agent, Sonnet model)
- Runs AFTER validator passes spec compliance — mandatory sequential ordering
- If validator FAILS, code-quality-reviewer is skipped (no point reviewing quality of wrong code)
- No reviewer for the code-quality-reviewer (it IS a reviewer — no infinite regress)
- Scope: craftsmanship (naming, structure, DRY/SOLID, readability, consistency) + security (injection, auth, input validation, OWASP) + performance (algorithmic complexity, allocations, N+1, caching)
- Does NOT re-check spec compliance or tests (validator and task-auditor own those)

### Structured Escalation Statuses (#7)
- Role-specific status sets:
  - **Doer agents** (executor, architect, planner, task-designer, researcher, product-researcher, debugger, codebase-analyst): DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
  - **Reviewer agents** (validator, code-quality-reviewer, architecture-auditor, plan-auditor, task-auditor, integration-checker): APPROVED / REJECTED / NEEDS_REVISION
- Statuses stored in agent output documents (consistent with "documents ARE the handoffs" principle), not as task fields
- NEEDS_REVISION auto-routes back to the doer with reviewer feedback attached
- Auto-route with configurable retry limit: default 2 retries (3 total attempts), configurable in trust.toml
- After retry limit exhausted: escalate to orchestrator/gateway

### Controller-Curated Context (#5)
- Orchestrator handoff for work-package-level tasks includes full task spec text inline, not just doc_id references
- doc_id references reserved for supplementary context the agent might need to look up
- Prevents context loading failures and improves task comprehension

### Fresh Agent Per Task (#6)
- Make the "fresh agent per task" rule explicit in the orchestrator's pool management protocol
- When an agent completes a task, it reports results and terminates
- Next task gets a new agent instance, even for the same role
- Prevents accumulated context from biasing later tasks

### Anti-Sycophancy in Review Reception (#8)
- Add a review-reception protocol to _synapse-protocol.md shared footer
- When any agent receives feedback from an auditor or validator: (1) verify feedback against codebase, (2) check if suggested change is actually needed, (3) push back with technical reasoning if feedback is wrong
- Prevents "correction cascade" where review feedback introduces new problems

### Brainstorming SKILL.md Redesign
- Full rewrite of `packages/framework/skills/brainstorming/SKILL.md`
- Adopts brainstorm-skill-redesign.md decisions as-is:
  1. Load project context (query Synapse DB)
  2. Scope gate (hierarchy mapping: Epic → decompose, Feature/WP → proceed)
  3. Ask clarifying questions (one per message, prefer multiple choice)
  4. Propose approaches (3+ options neutrally, then recommend with criteria)
  5. Present design (scale to complexity: simple one-shot, complex section-by-section)
  6. Store design document (Synapse DB: plan or architecture_decision)
  7. Offer to continue (proceed to Plan or stop)
- Hard gate: no implementation before design approval, regardless of perceived simplicity
- One-question-at-a-time is the key behavioral enforcement rule
- Includes anti-rationalization table (consistent with phase-wide practice)

### Skill-Wide Anti-Rationalization
- ALL existing skills get anti-rationalization sections added (not just brainstorming)
- Skills: brainstorming, vitest, python, bun, sql, tailwind, react, typescript, testing-strategy, architecture-design
- Each skill gets rationalization tables specific to its critical rules
- Sourced from external frameworks and best practices (same principle as agent tables)
